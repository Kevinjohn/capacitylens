Architecture

How the pieces fit and which way dependencies point. The whole system is a set of concentric layers around a pure core, with exactly one seam to persistence and one port to auth. Get the layering and the seams right and everything else has an obvious home.

The layers, in one diagram

"shared/ is a pure domain core imported by both app and server so they can't drift; the store is the orchestrator; the scheduler builds a view-model; persistence is server-backed by default behind one adapter, with the localStorage demo build behind that same seam; auth wraps the whole app through one provider and is inert when off." That sentence is the architecture. Here it is as a picture:

The two builds: server (default) and demo (opt-in)

One env flag decides which side of the persistence seam a build lives on, and it cascades through four behaviours. This is the single most important fact about the current architecture:

Server build (the default — empty env)Demo build (VITE_CAPACITYLENS_DEMO=1)
AdapterServerSyncAdapter against ${API_BASE}/api; empty VITE_CAPACITYLENS_API = same originLocalStorageAdapter (key capacitylens/v3)
First runEmpty — lands on the create-your-company picker (the server seeds only under CAPACITYLENS_SEED_DEMO=1, which npm run dev sets)Self-seeds the demo dataset (seedIfEmpty: isDemoMode() ? seed() : undefined)
AuthAuthProvider asks /api/auth/me; the server's authMode decidesZero-fetch pass-through; the cosmetic fake sign-in may gate instead
Tenant switchThe switch orchestrator re-hydrates the picked account's slice (page 05)Inert — all accounts are in the one local blob
Load-bearing rule

The decision is isDemoMode() — import.meta.env.VITE_CAPACITYLENS_DEMO === '1' — and nothing else. An empty VITE_CAPACITYLENS_API is still server mode (same-origin), and a server build with no reachable backend shows a "can't reach the server" screen; it does not silently fall back to localStorage. A silent fallback would look like data loss the moment the server came back. Keep the two builds' naming straight: the persistence demo (localStorage) is a different axis from the auth demo (the cosmetic fake sign-in screen).

The runtime gate chain

Above the router, the app is a fixed sequence of gates; each renders a whole-screen state and the order is deliberate (failure states first, identity next, tenancy last):

  1. AuthProvider (in main.tsx, outside the router) — one boot-time GET /api/auth/me. authMode 'off' passes through; a 401 swaps in the lazy LoginScreen. It re-checks when a later write hits persistError, so an expired session walls the app off instead of failing writes silently.
  2. connectionError → ConnectionError (server unreachable; retry).
  3. loadError → StorageRecovery (corrupt local bytes; reset/import/export). Two different failures, two different screens — never one generic error page.
  4. Fake sign-in (demo polish, auth-off only) → the cosmetic "Choose an account" screen. Never mounts when real auth is on.
  5. Tenant gate → AccountPicker (pick or create a company; "New company" hides at the single-company cap).
  6. One-time intro → IntroPage (device-global capacitylens/introSeen).
  7. PermissionProvider wraps the app body — resolves the caller's role for the active account and drives the Viewer read-only affordances (page 12).

The dependency rule

Dependencies point inward. The core depends on nothing; everything depends on the core. Concretely, within shared/ the sub-layers stack by how much of the dataset they touch:

types/   →  pure types + type-derived constants/clamps (the base; depends on nothing)
   ↑
lib/     →  pure per-record / generic utilities (id, strings, dates, colour,
            integrity, sanitize, scheduling-days) — may import types/
   ↑
data/    →  whole-dataset lifecycle (migrate, seed, transfer, fixtures,
            internalClient) — imports types/ + lib/
   ↑
domain/  →  whole-AppData mutations + tenancy + the pure access matrix (can) +
            the lifecycle state machine — imports all of the above
Load-bearing rule

The core never imports up the stack and never imports an environment API. If a function in shared/ needs the DOM, window, process, React, or the store, it is in the wrong package — move it to src/lib (browser) or server/src (Node). The reward is that the entire core is trivially unit-testable in plain Node and runs identically in the browser and on the server.

Where does new code go?

A rule about the data (validation, integrity, a derived value the server also needs, a permission) → shared/. A browser-only utility (theme, fuzzy search, palette generation, drag math, capacity display) → src/lib. State transition / id minting / undo → the store. Session verification / control tables → the server behind their ports (page 12). Pixels & ARIA → a component. When in doubt, push the logic down toward the core and keep components thin.

The data model

The single source of truth is a normalised, in-memory tree: AppData, a bag of arrays, one per entity table — nine tables: accounts plus eight scoped. Two base interfaces establish the universal fields.

shared/src/types/entities.ts (the bases)
/** Fields every persisted entity carries — cheap now, impossible to backfill later. */
export interface Entity {
  id: ID            // crypto.randomUUID()
  createdAt: ISOTimestamp
  updatedAt: ISOTimestamp
}

/** Every domain entity belongs to exactly one account. Accounts themselves don't. */
export interface ScopedEntity extends Entity {
  accountId: ID
}
Why id/timestamps on everything, from day one

Stable ids enable diff-based server sync (page 05) and id-remapping import (page 04). Timestamps enable last-writer-wins. Both are "cheap now, impossible to backfill later" — retrofitting a created-at onto existing rows is a migration; having it from the first commit is free. Carry both bases into a sibling unchanged.

CapacityLens's concrete tables (your sibling will rename these to its own domain, but the shape — a tenant at the top, everything else scoped under it, with required vs. optional foreign keys — carries over):

EntityScoped?Key relationships
AccountNo (it is the tenant)Holds the team-wide config: calendar (timezone, weekStartsOn — frozen after creation), schedulingMode, language, and the feature toggles disciplinesEnabled / placeholdersEnabled / externalEnabled.
DisciplineYesGroups resources; has a sortOrder.
ResourceYeskind: 'person' | 'placeholder' | 'external' — a person, a project-bound placeholder, or an outsourced third party (no capacity, zero-load allocations only). Optional disciplineId. Carries lifecycle fields.

The required-vs-optional FK distinction is not decoration — it drives the cascade and the import repair (a dangling required FK drops the row; a dangling optional FK just unbinds it). Resource, Client and Project additionally carry the lifecycle fields archivedAt/deletedAt driving the Active→Archived→Soft-deleted→Purged machine (page 04). Note what is not here: memberships and invites are control-plane tables on the server, structurally excluded from AppData (principle 8, page 12).

Two derived constructs make the "loop over every scoped table" pattern safe and DRY:

shared/src/types/entities.ts (the scoped-table seam)
export type ScopedEntityKey = 'disciplines' | 'resources' | 'clients' | 'projects'
  | 'phases' | 'activities' | 'allocations' | 'timeOff'

// The single runtime source of "all scoped tables". Cascade, scope-to-account,
// and import all derive their loops from this — add an entity here and they extend.
export const SCOPED_KEYS: ScopedEntityKey[] = ['disciplines','resources','clients',
  'projects','phases','activities','allocations','timeOff']

// A uniform ScopedEntity[] view of AppData's scoped tables — isolates into ONE named
// seam the single cast TS can't infer through a heterogeneous-union index.
export function scopedTables(data: AppData): Record<ScopedEntityKey, ScopedEntity[]> {
  return data as Record<ScopedEntityKey, ScopedEntity[]>
}
The "one cast, one place" pattern

TypeScript can't narrow a heterogeneous-union index, so iterating all scoped tables as a common supertype needs one cast. Rather than scatter as never casts through every loop, scopedTables() isolates that single cast behind a named function. When you must cheat the type system, cheat it once, in a named seam, with a comment.

The persistence seam

The app never reads or writes storage directly. It talks to one interface; an implementation is chosen at module load by isDemoMode() (see the two builds).

src/data/PersistenceAdapter.ts
export interface PersistenceAdapter {
  loadAll(accountId?: string): Promise<AppData>   // accountId (server only): load ONLY that
                                                    // account's slice + re-seed the diff snapshot
  saveAll(data: AppData, opts?: { unload?: boolean }): Promise<void>
  hasExisting?(): Promise<boolean>   // distinguishes first-run from "user cleared everything"
}

export type LoadErrorKind = 'corrupt' | 'unavailable'   // picks the recovery screen

Three methods (one optional) is the entire contract, plus a typed LoadError whose kind routes to the right recovery screen. ServerSyncAdapter implements it against the REST API by diffing snapshots; LocalStorageAdapter against the versioned localStorage key. The selection is one ternary in storageAdapter.ts. Full detail — including the debounced write, flush-on-unload, per-account hydration, and the diff algorithm — is on page 05.

Multi-tenancy mechanics

The tenant boundary is defined in exactly one place so a read filter, a write guard, and a cascade can never disagree about what "belongs to account X" means:

shared/src/domain/tenancy.ts
export const belongsToAccount = (entity: ScopedEntity, accountId: ID): boolean =>
  entity.accountId === accountId

export const byAccount = (accountId: ID) => (entity: ScopedEntity): boolean =>
  belongsToAccount(entity, accountId)            // positive predicate for .filter()

export const notInAccount = (accountId: ID) => (entity: ScopedEntity): boolean =>
  !belongsToAccount(entity, accountId)           // complement: keep rows NOT in this account

On the read side, the store exposes useScopedData() which filters AppData down to the active account (memoised so unrelated mutations don't re-render every consumer) — and its view-layer twin useActiveScopedData(), which additionally hides archived/soft-deleted rows for the 15 normal view components (export keeps the raw seam). On the write side, every update/delete first resolves the row through a guard that returns null for a missing id (a stale-id no-op) but throws for a cross-account id (a real integrity violation). The active account id is in store state but is never persisted — it is re-chosen at every load.

Load-bearing rule

Switching accounts resets the undo/redo history and all per-account UI state. Undo must never cross an account boundary, and the previous tenant's filters/selection don't apply to the next. This is enforced in setActiveAccount, and in server mode the switch additionally runs the hydration orchestrator so the diff snapshot and the data are always the same account (page 05 — the #1 invariant of the sync layer).

Schema ≠ deployment shape

The schema is multi-tenant; the default deployment is single-company. The server refuses to create a second account unless CAPACITYLENS_MULTI_ACCOUNT=1 (in every auth mode, create-time only), /api/auth/me reports canCreateAccount, and the picker hides "New company" at the cap — fail-open on the client, because the server is the enforcer. Dev and the demo build stay uncapped. In auth-on deployments the tenant boundary is real: reads are membership-gated per account and the whole-dataset read is closed (page 12).

The entity-extension path (the rails)

This is the architecture's signature mechanism and the thing a sibling most benefits from copying wholesale. Adding a field or a whole entity is normally a silent-data-loss minefield because the same fact is restated in many parallel lists. Here every list is checked against the types, so a miss is a build failure, not a runtime bug.

The parallel lists that all describe "the set of entities/fields":

ListWhereChecked how
ScopedEntityKey / SCOPED_KEYSshared typesThe type is the source; the array is the runtime mirror.
KNOWN_KEYSshared data/migrate.tsDerived: ['accounts', ...SCOPED_KEYS] — can't fall out of sync.
the sanitizeImportedRecord switchshared lib/sanitizeImport.tsdefault: const _exhaustive: never = key — a missing case won't compile.
FK_TARGETshared domain/mutations.tsTyped Record<string, ScopedEntityKey> — a bad table name won't compile.
UPSERT_ORDERclient data/syncOps.tsExclude<keyof AppData, TableKey> extends never — a missing table won't compile.
TABLES / CREATE_ORDER / SCOPED_ORDER + column specsserver tables.tsCheckColumns (bidirectional) + Exclude<…> extends never.
full per-entity fixturesshared data/fixtures.tsTyped as the entity interface + round-tripped through server tests — a dropped field fails the gate.

So the procedure for adding a field is a fixed, on-rails sequence:

  1. Add it to the shared type (entities.ts). If it's constrained (enum, clamp), add the rule there too.
  2. Set it in fixtures.ts to a non-default value. (The fixture is typed as the entity, so this is also where TS reminds you it's required.)
  3. Add the column to server/src/tables.ts. The CheckColumns type fails the build until you do; the runtime auto-ALTER adds the DB column on next open.
  4. Teach the sanitiser in sanitizeImport.ts how to clean/clamp it (if it needs repair on import).
Do this

Trust the compiler. After step 1, run npm run gate + npm run gate:server and let the failures route you through steps 2–4. Each missed list produces a precise type error pointing at the file to fix. You physically cannot ship a half-wired field. Page 04 (sanitiser, fixtures) and page 07 (column specs) show the exact code of each checkpoint.

Don't route control tables through the rails

The extension path is for app data. Memberships and invites deliberately bypass every list above — they must never appear in SCOPED_KEYS, KNOWN_KEYS, fixtures, or the server's TABLES, or they would leak through generic CRUD and export. A control table gets its own schema module and its own narrow accessors (page 12).

Deliberately not done at this size

An architecture is as much what you refuse to build as what you build. These were considered and rejected for the current scale; each is a candidate only when a real constraint forces it.

  • No generic store/CRUD factory and no schema-driven forms. Each entity has a hand-written form and list. The repetition is small and the explicitness is worth more than the abstraction at nine entities.
  • No custom listbox to replace native <select>. OS popup misalignment is accepted in exchange for free accessibility and zero code.
  • No measurement-based row virtualization for font-scaling. Body rows keep fixed pixel heights; only the date header uses minHeight.
  • No import id-dedup beyond repair, no ORM in the server (raw node:sqlite), no Postgres (the TenantStore interface on page 07 is the documented swap point if that ever changes).
The modularity rule that governs refactors

Only pure extractions (the gate proves them safe) land freely. High-churn structural splits — slicing the store by concern, extracting a viewport hook, splitting the grid render — wait until a characterisation test is written first. You don't refactor a hot, under-tested module and hope; you pin its behaviour with a test, then move it. More on page 09.
