UI & components

The component layer is thin, hand-written, and accessible by default. A small primitive kit, one canonical form pattern, one canonical list pattern, a pure scheduler view-model, and a token-driven design system. No component library — and that is the point.

└── AppShell.tsx  CommandPalette.tsx  …

Conventions: PascalCase for component files; camelCase for pure logic modules (schedulerModel.ts, layout.ts). One folder per entity, each holding <Entity>Form.tsx + <Entity>List.tsx. Tests co-located (ClientForm.test.tsx beside ClientForm.tsx).

Why controls.ts is not .tsx

It exports plain style-constant objects, no components. If a style export sat next to a component export in the same module, the react-refresh/only-export-components lint rule would fire. Splitting non-component constants into a .ts file keeps Fast Refresh boundaries clean. The barrel ui.tsx carries a matching eslint-disable comment explaining that a re-export-only file is not itself a refresh boundary.

The primitive kit

Everything imports from '../common/ui' — never from the implementation sub-files. The Button is the template for the whole kit: a small variant union, a class map, and safe defaults.

src/components/common/dialogs.tsx
type ButtonVariant = 'primary' | 'ghost' | 'danger'
const buttonClasses: Record<ButtonVariant, string> = {
  primary: 'bg-brand-soft text-brand-soft-ink hover:bg-brand/25 shadow-sm',
  ghost: 'border bg-surface text-ink hover:bg-canvas',
  danger: 'bg-danger-soft text-danger-soft-ink hover:bg-danger/25 shadow-sm',
}
export function Button({ children, onClick, variant = 'primary', type = 'button', ... }) {
  return <button type={type} ... className={`... ${buttonClasses[variant]}`}>{children}</button>
}
Load-bearing rule

Button defaults to type="button". Only the one primary action in a form is type="submit". This pairs with the Modal's form wrapper so Enter submits every dialog while no other button accidentally submits it.

The Modal pattern — every dialog is a real form

Modal wraps its body and footer in a <form noValidate>. The primary footer button is type="submit", so Enter in any field submits. The form is rendered even when no onSubmit is given, to suppress implicit page navigation.

src/components/common/dialogs.tsx (Modal essence)
<div role="dialog" aria-modal="true" aria-labelledby={titleId} tabIndex={-1}>
  <form noValidate onSubmit={(e) => { e.preventDefault(); onSubmit?.() }}>
    <div className="space-y-3 p-4">{children}</div>
    {footer && <footer className="flex justify-end gap-2 border-t px-4 py-3">{footer}</footer>}
  </form>
</div>

The Modal also owns three behaviours every dialog inherits for free:

  • Focus management. On open, focus goes to the first [data-autofocus] target (or first focusable). Tab wraps inside the dialog. On close, focus restores to the trigger, falling back to <main> if the trigger left the DOM.
  • Backdrop close requires both mousedown and mouseup on the backdrop — so a drag that starts in an input and releases on the backdrop doesn't dismiss the dialog.
  • The unsaved-changes guard. The Modal listens for bubbling input/change events and aria-pressed toggle clicks; once dirty, Escape and backdrop show a notice instead of closing, and the dirty state is published to the store so the shell can also guard beforeunload. A guardDirty={false} prop opts confirmation-only dialogs out.
Do this

Build every new dialog on Modal + a ConfirmDialog-style footer. You get Enter-to-submit, focus trapping, the dirty guard, and ARIA labelling without writing any of it. A ConfirmDialog deliberately passes no onSubmit (so Enter can't accidentally confirm a destructive action) and its message ends with "You can undo this with ⌘Z.".

Field primitives — three visual states, ARIA wired

Every field is a <label> wrapping a FieldLabel (which owns the required asterisk) and the native control, with ARIA threaded through. The shared controlClass encodes three states:

src/components/common/fields.tsx
function controlClass(invalid?: boolean, required?: boolean) {
  if (invalid) return `${inputClass} border-danger ring-1 ring-danger`   // post-Save error: pops
  if (required) return `${inputClass} border-l-2 border-l-danger/60`     // at rest: quiet red edge
  return inputClass
}

// aria-label is the BARE label (the asterisk is aria-hidden, so it never leaks into the name).
<input aria-label={label} aria-required={required || undefined}
  aria-invalid={invalid || undefined}
  aria-describedby={invalid ? describedById : undefined} ... />
Load-bearing rule

aria-describedby is wired only when invalid — pointing at the error element — so navigating to the offending field re-announces the message. The accessible name is the bare label string, never the rendered text (which would include the asterisk). Native <select> is used directly; there is no custom listbox (OS popup misalignment is accepted for free a11y).

Colour fields use a preset swatch grid only — no free hex input — so a stored colour is always a valid 6-digit hex. Weekday/toggle controls use aria-pressed (they are toggles, not checkboxes), which also feeds the Modal's dirty guard.

The canonical form

Forms are self-contained: local useState per field, validation on submit only (never on blur/keystroke), one error at a time via useFieldError.

src/components/clients/ClientForm.tsx (essence)
const [name, setName] = useState(client?.name ?? '')
const [color, setColor] = useState(client?.color ?? DEFAULT_COLORS.client)
const { error, errorField, errorId, fail } = useFieldError()

const submit = () => {
  const trimmed = validateName(name, fail)   // calls fail('name', msg) and returns null on error
  if (!trimmed) return
  if (!validateHex(color, fail)) return
  if (client) updateClient(client.id, { name: trimmed, color })
  else addClient({ name: trimmed, color })
  onClose()
}

return (
  <Modal title={client ? 'Edit client' : 'Add client'} onClose={onClose} onSubmit={submit}
    footer={<><Button variant="ghost" onClick={onClose}>Cancel</Button>
              <Button type="submit">Save</Button></>}>
    <TextField label="Name" value={name} onChange={setName} required
      invalid={errorField === 'name'} describedById={errorId} />
    <FieldError id={errorId}>{error}</FieldError>
  </Modal>
)

useFieldError holds error + errorField + a stable errorId (from useId()) shared by the field's describedById and the <FieldError>. Validators return the cleaned value (or null after calling fail) and short-circuit; the store action runs only after every check passes.

The canonical list

Lists read scoped data, pull individual actions, and manage three pieces of modal state via one hook. The form and confirm dialogs are gated at the bottom of the component, never inline in a row.

src/components/clients/ClientList.tsx (essence)
const clients = useScopedData().clients
const deleteClient = useStore((s) => s.deleteClient)
const { creating, setCreating, editing, setEditing, confirming, setConfirming } = useCrudListState<Client>()

return (
  <ListPage title="Clients" addLabel="Add client" onAdd={() => setCreating(true)}>
    {clients.length === 0 ? <EmptyState>No clients yet.</EmptyState> : (
      <ul className="divide-y divide-line rounded border">
        {clients.map((c) => (
          <li key={c.id} data-testid="client-row" className="flex justify-between px-3 py-2">
            <span><ColorSwatch color={c.color} /> {c.name}</span>
            <span><Button variant="ghost" onClick={() => setEditing(c)}>Edit</Button>
                  <Button variant="danger" onClick={() => setConfirming(c)}>Delete</Button></span>
          </li>))}
      </ul>)}

    {creating && <ClientForm onClose={() => setCreating(false)} />}
    {editing && <ClientForm client={editing} onClose={() => setEditing(null)} />}
    {confirming && <ConfirmDialog title="Delete client?"
       message={`Delete "${confirming.name}" and all of its projects… You can undo this with ⌘Z.`}
       onConfirm={() => { deleteClient(confirming.id); setConfirming(null) }}
       onCancel={() => setConfirming(null)} />}
  </ListPage>
)
src/hooks/useCrudListState.ts
export function useCrudListState<T>() {
  const [creating, setCreating] = useState(false)
  const [editing, setEditing] = useState<T | null>(null)
  const [confirming, setConfirming] = useState<T | null>(null)
  return { creating, setCreating, editing, setEditing, confirming, setConfirming }
}

The scheduler view-model — a pure function

The complex screen is built by a pure function with no React: buildSchedulerModel(data, origin, dayWidth, days, utilStart, utilEnd, filters) returns a GroupModel[] tree of rows, bars, day-states, time-off blocks, and utilisation. It is exhaustively unit-tested in isolation because it has no hooks.

Why the view-model is separated from rendering

Two payoffs. Testability: all the layout/capacity/filter logic is a pure function you test with plain objects (page 08). Performance: it groups allocations and time-off by resource once up front (so building each row is a Map lookup, making it O(allocations + timeOff + resources) not O(resources × everything)), and keeps each row's slice referentially stable so React.memo on lanes and bars actually skips work. Fixed pixel constants live in one layout.ts object; row virtualization math lives in a pure virtualWindow.ts (prefix-sum built once, queried per scroll frame).

Styling & the design tokens

Tailwind v4. The entire palette is semantic CSS custom properties declared in a @theme block in src/index.css; component classes use only the semantic names (bg-surface, text-ink, text-muted, border-line) — never Tailwind scale names like text-gray-500.

src/index.css (the token system)
@theme {
  --color-canvas: var(--c-base);     /* page bg — named `canvas` not `base` on purpose:
  --color-surface: var(--c-surface);    --color-base would shadow Tailwind's text-base size util */
  --color-ink: var(--c-ink);  --color-muted: var(--c-muted);  --color-faint: var(--c-faint);
  --color-brand: var(--c-brand);  --color-danger: var(--c-danger);  /* …soft + soft-ink pairs… */
  --text-2xs: 0.6875rem;             /* one deliberate 11px size for dense labels */
  --shadow-pop: 0 10px 30px -10px rgb(0 0 0 / 0.3);
}
:root { --c-base: #f4f5f8; --c-ink: #1c2230; --c-brand: #6366f1; /* …light palette… */ color-scheme: light; }
:root[data-theme="dark"] { --c-base: #0e1016; --c-ink: #e7eaf0; --c-brand: #818cf8; /* … */ color-scheme: dark; }
Load-bearing rule

Dark mode is one attribute swap (data-theme="dark" on <html>), not per-component dark: classes. JS resolves light/dark/system to a concrete attribute; the CSS only ever sees light or dark. Because components use semantic tokens, they flip automatically. Every colour value is tuned to clear WCAG AA in both themes (the comments record the contrast adjustments).

Other token-level decisions: a global :focus-visible ring on all interactives; preset swatch palette only; two custom keyframes (floaty-pop, floaty-fade) for modal/toast entrances; and a global prefers-reduced-motion block that nulls all animation. The metric is always called "Utilisation", never "Load" — in labels, store keys (utilizationPrefs), and identifiers.

Accessibility conventions

Accessibility is a gate (page 08), so the patterns are consistent and pervasive:

PatternHow it's done
Active navReact Router NavLink → automatic aria-current="page".
Togglesaria-pressed (weekday picker, swatches) — not aria-checked; they're buttons, not checkboxes.
Command paletteFull combobox/listbox: role="combobox" + aria-expanded/aria-controls/aria-activedescendant; role="listbox" + role="option" + aria-selected.
Dialogsrole="dialog" + aria-modal + aria-labelledby; manual Tab-trap; focus restore.
Decorative elementsIcons, avatars, hatch patterns, drag grips, the required asterisk → aria-hidden (+ focusable="false" on SVGs).
Field errorsrole="alert" on the message; field's aria-describedby points at it when invalid.
Skip linksr-only focus:not-sr-only "Skip to content" → <main id="main" tabIndex={-1}>.
BarsOperable by keyboard: role="button" tabIndex={0} with an aria-label that announces both data and affordances (Enter to edit, arrows to move, etc.).
Toastsrole="alert" (error) vs role="status" (info).
The collapsed-sidebar rail trap, handled correctly

The icons-only rail's icons do not navigate — they re-open the menu — so they are aria-hidden + tabIndex={-1}, and the single labelled Collapse/Expand toggle is the only accessible control. When a decorative affordance overlaps a real one, hide the decoy from assistive tech and leave exactly one labelled path.

React/TS conventions

  • Function components only (the one class is ErrorBoundary, which must be). Props typed inline as an object literal in the signature; only exported shapes (Option, BarLayout) get a named interface.
  • React.memo on hot-path scheduler components only (AllocationBar, ResourceLane), wrapping a named function for DevTools. Parent callbacks are stabilised with useCallback so memo is effective.
  • Event handler naming: props are onClose/onAdd/onConfirm; internal handlers are named by action (submit, requestClose) — no handle prefix.
  • data-testid is kebab-case, named for the role (client-row, allocation-bar, over-marker). Functional data attributes like data-resource-id are used for drag/snapshot logic, distinct from testids.
  • The ref-for-latest-callback pattern in effects: store onClose in a ref updated each render, so an empty-deps focus-trap effect runs once but always calls the latest function — avoiding effect re-runs on every store mutation while a dialog is open.
