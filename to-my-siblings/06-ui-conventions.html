UI & components

The component layer is thin and accessible by default: a small house kit (common/) layered over shadcn/ui primitives (ui/), one canonical form pattern, one canonical list pattern, a pure scheduler view-model, compile-time strings, and a token-driven design system that holds WCAG 2.2 AA in both themes. The library does the plumbing; the house kit owns every behaviour that matters.

Conventions: PascalCase for component files; camelCase for pure logic modules (schedulerModel.ts, layout.ts). One folder per entity, each holding <Entity>Form.tsx + <Entity>List.tsx. Tests co-located (ClientForm.test.tsx beside ClientForm.tsx).

Why controls.ts is not .tsx

It exports plain style-constant objects, no components. If a style export sat next to a component export in the same module, the react-refresh/only-export-components lint rule would fire. Splitting non-component constants into a .ts file keeps Fast Refresh boundaries clean. The barrel ui.tsx carries a matching eslint-disable comment explaining that a re-export-only file is not itself a refresh boundary.

The two-layer kit: house common/ over shadcn ui/

The kit is hybrid. src/components/ui/ holds shadcn/ui primitives (Radix headless behaviour + cva variants + cmdk); src/components/common/ wraps them with the house API, behaviours and tokens. App code imports from '../common/ui' — never from the shadcn files directly. The division of labour:

LayerOwnsEditing rule
ui/ (shadcn)Layout scaffolding, focus/disabled plumbing, Radix behaviour (Popover, Tooltip, Separator, Dialog primitive), cmdk.Generated by shadcn add, then house-edited: rebind stock tokens to house tokens, delete what conflicts (e.g. the stock aria-invalid ring on Input, so the single house danger ring shows; the generated CommandDialog). Edits are committed — these files are code, not vendored artefacts.
common/ (house)The public API and every behaviour that matters: variants, ARIA wiring, the dirty guard, focus restore, permission gating, testids.This is where new UI work happens. Never let a raw shadcn component leak into a feature file.
src/components/common/dialogs.tsx (Button essence)
type ButtonVariant = 'primary' | 'ghost' | 'danger'
const VARIANT_CLASS: Record<ButtonVariant, string> = {
  primary: 'bg-brand-strong text-white hover:bg-brand-strong/90 shadow-xs',
  ghost: 'border bg-surface text-ink hover:bg-canvas shadow-xs',
  danger: 'bg-danger-soft text-danger-soft-ink hover:bg-danger-soft/80 shadow-xs',
}
// The house Button RENDERS the shadcn Button (size="sm") — shadcn does layout/focus/disabled,
// the house map does colour, cn() merges. Props: ariaLabel, describedById, testId, className.
export function Button({ children, onClick, variant = 'primary', type = 'button', ... }) {
  return <ShadButton size="sm" type={type} ... className={cn(VARIANT_CLASS[variant], className)}>{children}</ShadButton>
}
Load-bearing rule

Button defaults to type="button". Only the one primary action in a form is type="submit". This pairs with the Modal's form wrapper so Enter submits every dialog while no other button accidentally submits it.

List-row actions are icon-only; creates carry a leading +

Three kit members own this convention: EditButton (pencil, ghost) and DeleteButton (trash, danger) for list rows — icon-only, accessible name on the button (aria-label + title), glyph always aria-hidden, and Viewer-gated (they render null when the role can't edit) — and AddButton (+ plus a text label) for every create affordance. Dialog footers keep their text buttons. Build new list pages on these shells, not bare <Button>Edit</Button>; getByRole('button', {name}) stays stable because the name never left the button.

The Modal pattern — every dialog is a real form, portaled to body

Modal is built on the Radix Dialog primitive but keeps the house behaviours: it runs modal={false} (Radix's traps are neutralised so the house owns one dismiss path), re-adds aria-modal="true" on the content, labels via DialogPrimitive.Title, wraps body and footer in a <form noValidate> — and the whole tree is createPortal-ed to document.body.

src/components/common/dialogs.tsx (Modal essence)
createPortal(
  <DialogPrimitive.Root open modal={false}>
    <div className="backdrop">                          /* plain backdrop — Radix Overlay is null when non-modal */
      <DialogPrimitive.Content aria-modal="true" …>     /* house Escape/Tab-trap/focus-restore; Radix auto-handlers preventDefault-ed */
        <DialogPrimitive.Title>{title}</DialogPrimitive.Title>
        <form noValidate onSubmit={(e) => { e.preventDefault(); onSubmit?.() }}>
          <div className="space-y-3 p-4">{children}</div>
          {footer && <footer className="flex justify-end gap-2 border-t px-4 py-3">{footer}</footer>}
        </form>
      </DialogPrimitive.Content>
    </div>
  </DialogPrimitive.Root>,
  document.body)
Why the portal is load-bearing (not a style choice)

A dialog opened from inside the scheduler used to render as a DOM descendant of the grid's role="grid" — which makes the accessibility tree claim the dialog is a grid cell (axe: aria-required-children). Portaling to <body> means a dialog is never inside any composite widget's DOM. The primary footer button is type="submit", so Enter in any field submits; the form renders even when no onSubmit is given, to suppress implicit page navigation.

The Modal also owns three behaviours every dialog inherits for free:

  • Focus management. On open, focus goes to the first [data-autofocus] target (or first focusable). Tab wraps inside the dialog. On close, focus restores to the trigger, falling back to <main> if the trigger left the DOM.
  • Backdrop close requires both mousedown and mouseup on the backdrop — so a drag that starts in an input and releases on the backdrop doesn't dismiss the dialog.
  • The unsaved-changes guard. The Modal listens for bubbling input/change events and aria-pressed toggle clicks; once dirty, Escape and backdrop show a notice instead of closing, and the dirty state is published to the store so the shell can also guard beforeunload. A guardDirty={false} prop opts confirmation-only dialogs out.
Do this

Build every new dialog on Modal + a ConfirmDialog-style footer. You get Enter-to-submit, focus trapping, the dirty guard, and ARIA labelling without writing any of it. A ConfirmDialog deliberately passes no onSubmit (so Enter can't accidentally confirm a destructive action) and its message ends with "You can undo this with ⌘Z.".

Field primitives — three visual states, ARIA wired

Every field is a <label> wrapping a FieldLabel (which owns the required asterisk) and the control — shadcn's <Input>/<Textarea> for text, the native <select> for selects — with ARIA threaded through. The three visual states are an accent layered over a shared base:

src/components/common/fields.tsx
// Accent-only — layered over the shared controlBase via cn(). A separate selectClass
// exists for the native <select> (it isn't a shadcn component).
function fieldAccent(invalid?: boolean, required?: boolean) {
  if (invalid) return 'border-danger ring-1 ring-danger'    // post-Save error: pops
  if (required) return 'border-l-2 border-l-danger/60'       // at rest: quiet red edge
  return ''
}

// aria-label is the BARE label (the asterisk is aria-hidden, so it never leaks into the name).
<Input className={cn(controlBase, fieldAccent(invalid, required))}
  aria-label={label} aria-required={required || undefined}
  aria-invalid={invalid || undefined}
  aria-describedby={invalid ? describedById : undefined} ... />
Load-bearing rule

aria-describedby is wired only when invalid — pointing at the error element — so navigating to the offending field re-announces the message. The accessible name is the bare label string, never the rendered text (which would include the asterisk). Native <select> is used directly; there is no custom listbox (OS popup misalignment is accepted for free a11y).

Colour fields use a preset swatch grid only — no free hex input — so a stored colour is always a valid 6-digit hex (swatch buttons are 24×24px — the WCAG 2.2 target floor). Weekday/toggle controls use aria-pressed (they are toggles, not checkboxes), which also feeds the Modal's dirty guard. Single-select pill choosers use SegmentedControl (role="radiogroup"/role="radio" — a different semantic from the toggles).

Strings: every user-facing string is a compiled message

No string literals in components — every label, aria-label, title, screen-reader announcement and toast message is a Paraglide message function, imported from the single seam:

import { m } from '@/i18n'          // NEVER from '@/paraglide' directly

<TextField label={m.form_resource_name_label()} … />
<Button type="submit">{m.form_save()}</Button>
  • Keys are snake_case, area-namespaced: nav_*, settings_*, form_* (form/dialog chrome, entity-prefixed where it clarifies), dialog_* (whole-dialog copy), palette_*, scheduler_sr_*. The value is the source English.
  • Share a key only when it is the same string and the same action — form_save/form_cancel/form_delete/form_add/form_edit cover every form's buttons; anything else gets its own key.
  • Resolve at render, never at module load. A module-scope label list uses a getter (() => m.nav_schedule()) so the account-scoped locale switch (syncLocaleFromAccount, driven by Account.language) re-resolves on the next render instead of freezing at import time.
  • Interpolations are parameters (m.form_allocation_new_for({name})); singular/plural pairs are _one/_other keys selected in JS.
Why this matters even in English-only

A missing or typo'd key is a tsc error (the messages compile to typed functions — page 02), so the string catalogue can never silently rot. And because every string — including aria-labels and SR-only text — goes through the catalogue, the accessibility copy is as reviewable as the visible copy. Adding a second language later is a JSON file, not a component sweep.

The canonical form

Forms are self-contained: local useState per field, validation on submit only (never on blur/keystroke), one error at a time via useFieldError, all copy via m, and the store call wrapped so an integrity throw surfaces as a field error (errorMessage(e) — never a blank dialog).

src/components/clients/ClientForm.tsx (essence)
const [name, setName] = useState(client?.name ?? '')
const [color, setColor] = useState(client?.color ?? DEFAULT_COLORS.client)
const { error, errorField, errorId, fail } = useFieldError()

const submit = () => {
  const trimmed = validateName(name, fail)   // calls fail('name', msg) and returns null on error
  if (!trimmed) return
  if (!validateHex(color, fail)) return
  if (client) updateClient(client.id, { name: trimmed, color })
  else addClient({ name: trimmed, color })
  onClose()
}

return (
  <Modal title={client ? 'Edit client' : 'Add client'} onClose={onClose} onSubmit={submit}
    footer={<><Button variant="ghost" onClick={onClose}>Cancel</Button>
              <Button type="submit">Save</Button></>}>
    <TextField label="Name" value={name} onChange={setName} required
      invalid={errorField === 'name'} describedById={errorId} />
    <FieldError id={errorId}>{error}</FieldError>
  </Modal>
)

useFieldError holds error + errorField + a stable errorId (from useId()) shared by the field's describedById and the <FieldError>. Validators return the cleaned value (or null after calling fail) and short-circuit; the store action runs only after every check passes.

The canonical list

Lists read scoped data, pull individual actions, and manage three pieces of modal state via one hook. The form and confirm dialogs are gated at the bottom of the component, never inline in a row.

src/components/clients/ClientList.tsx (essence)
const clients = useScopedData().clients
const deleteClient = useStore((s) => s.deleteClient)
const { creating, setCreating, editing, setEditing, confirming, setConfirming } = useCrudListState<Client>()

return (
  <ListPage title="Clients" addLabel="Add client" onAdd={() => setCreating(true)}>
    {clients.length === 0 ? <EmptyState>No clients yet.</EmptyState> : (
      <ul className="divide-y divide-line rounded border">
        {clients.map((c) => (
          <li key={c.id} data-testid="client-row" className="flex justify-between px-3 py-2">
            <span><ColorSwatch color={c.color} /> {c.name}</span>
            <span><EditButton onClick={() => setEditing(c)} />      {/* icon-only, viewer-gated */}
                  <DeleteButton onClick={() => setConfirming(c)} /></span>
          </li>))}
      </ul>)}

    {creating && <ClientForm onClose={() => setCreating(false)} />}
    {editing && <ClientForm client={editing} onClose={() => setEditing(null)} />}
    {confirming && <ConfirmDialog title="Delete client?"
       message={`Delete "${confirming.name}" and all of its projects… You can undo this with ⌘Z.`}
       onConfirm={() => { deleteClient(confirming.id); setConfirming(null) }}
       onCancel={() => setConfirming(null)} />}
  </ListPage>
)
src/hooks/useCrudListState.ts
export function useCrudListState<T>() {
  const [creating, setCreating] = useState(false)
  const [editing, setEditing] = useState<T | null>(null)
  const [confirming, setConfirming] = useState<T | null>(null)
  return { creating, setCreating, editing, setEditing, confirming, setConfirming }
}

The scheduler view-model — a pure function, three distinct signals

The complex screen is built by a pure function with no React: buildSchedulerModel takes the data, the geometry, the day window, two date windows (visStart/visEnd and overStart/overEnd), the filters and the per-account feature toggles, and returns a GroupModel[] tree of rows, bars, day-states, time-off blocks, and utilisation. It is exhaustively unit-tested in isolation because it has no hooks. (geom is the ColumnGeometry below — it owns every pixel conversion, so the model never multiplies by a dayWidth scalar.)

The three over/utilisation signals — never merge them

(1) The per-day over-marker flags any day where allocated > available (strictly greater — at-capacity is not over), whole timeline, weekend-aware: a bar merely spanning a non-working weekend does no work there, so the weekend keeps its cool tint; the zero-capacity days that do read as over are a time-off day a working allocation covers, and a weekend an allocation opts into via ignoreWeekends. (2) The displayed utilisation % (per-person, per-discipline, overall) is a working-day ratio over the visible window — the 1/2/4/8-week zoom span anchored at the scroll left edge, recomputed day-quantized on zoom/pan (the left-edge day index, never per scroll pixel). (3) The overSoon flag is a working-day ratio over a fixed forward 14-day window from today (UTILIZATION_WINDOW_DAYS), zoom-independent. Three questions, three windows, three code paths — SchedulerGrid threads both windows into the model separately.

Why the view-model is separated from rendering

Two payoffs. Testability: all the layout/capacity/filter logic is a pure function you test with plain objects (page 08). Performance: it groups allocations and time-off by resource once up front (so building each row is a Map lookup, making it O(allocations + timeOff + resources) not O(resources × everything)), and keeps each row's slice referentially stable so React.memo on lanes and bars actually skips work. Fixed pixel constants live in one layout.ts object; row virtualization math lives in a pure virtualWindow.ts (prefix-sum built once, queried per scroll frame).

Variable-width timeline columns — the ColumnGeometry pattern

A timeline that draws every day at one fixed dayWidth is a uniform grid: every position is index × dayWidth and the pointer→date inverse is a single Math.floor((clientX − laneLeft) / dayWidth). That scalar is load-bearing: it shows up in the bar layout, the lane overlays, the today line, the date header, the zoom/resize scroll-anchor and the drag/resize math. The day you want one column a different width — CapacityLens's "minimise weekends", where Sat/Sun shrink to a date-number sliver — you must give up the uniform assumption across ~7 files at once. The pattern that makes that safe is to replace the scalar with one pure geometry object that every site asks instead of multiplying.

src/components/scheduler/columnGeometry.ts (the abstraction)
export interface ColumnGeometry {
  readonly widths: number[]     // per-day px width
  readonly offsets: number[]    // prefix sum, length n+1; offsets[i] = left edge of day i
  readonly totalWidth: number
  readonly minimiseActive: boolean
  x(index: number): number             // left edge of column i (clamped 0..n)
  widthOf(index: number): number       // 0 out of range
  spanWidth(s: number, e: number): number
  indexAt(px: number): number          // the EXACT inverse of x() — binary search
  xForDateInGeom(date: ISODate): number // extrapolates at full width off-window
  widthForDates(start: ISODate, end: ISODate): number
}
// Built once per render in SchedulerGrid, memoised on [days, dayWidth, minimiseWeekends, weekendWidth].
export function buildColumnGeometry(days: ISODate[], dayWidth: number,
  opts: { minimiseWeekends: boolean; weekendWidth: number }): ColumnGeometry

Everything that used to multiply by dayWidth now asks the geometry. With the feature off, every widths[i] === dayWidth, so x(i) === i×dayWidth, indexAt === floor(px/dayWidth) and totalWidth === days.length×dayWidth — byte-identical to the old uniform grid.

Sequence it as two commits: mechanical, then the feature

This is the whole trick for a wide-blast-radius refactor. Commit 1 (mechanical): introduce the geometry and thread it through every site, built with the flag hard-wired off so it is byte-identical — every existing unit test and E2E passes unchanged, which is the proof the threading didn't change behaviour. Commit 2 (the feature): add the device-global pref, flip the flag on, switch the header label, add the fit; the only new behaviour is the widths. A reviewer reads commit 1 for "is the wiring faithful?" and commit 2 for "is the feature right?" — separately. Keep the branch off any other scheduler work; it touches the files most scheduler tweaks do, and interleaving is the one avoidable risk.

The threading, atomically — each site trades the scalar for a geometry call:

Three invariants make it correct. Each is a bug class that passing unit tests will not catch — they only bite with real layout, real sub-pixel scroll, or a real pointer, so they need an E2E and a screenshot, not just green Vitest.

1 · One inverse, the exact inverse, shared by drag AND draw

indexAt must be the exact inverse of x() at every boundary — a click at offsets[i] → i, at offsets[i]−1 → i−1 — one binary search, unit-tested with the round-trip indexAt(x(i)) === i. The drag day-delta is indexAt(curr) − indexAt(start) (each endpoint snapped to its own column), never round(Δpx / dayWidth): with unequal column widths a single rounded delta lands on the wrong day as soon as the path crosses a narrow column. The lane's draw-to-create and the bar's drag take the same resolver, so the two inverses can't diverge.

2 · Integer-pixel widths, or the scroll-anchor drifts

Round the special column width to a whole pixel. A rem-based width is fractional (1.4 × 15.99 = 22.39), which makes fractional offsets (offset(Mon) = 1699.12) — but the browser stores scrollLeft as a whole number (1699). The scroll-anchor recovers the left-edge date with geom.indexAt(scrollLeft), and indexAt(1699) floors to the column before Monday, so the left-edge date drifts back a day (onto the narrow weekend) on every zoom flip. Whole-pixel widths (the day width is already floored) make the offsets integers and the round-trip exact. This one cost a debugging cycle — it is invisible to unit tests because jsdom has no real scroll.

3 · The live preview and the commit go through the SAME geometry

The drag preview (pixels, mid-gesture) and the committed bar (the rebuilt view-model) must both run the snapped dates through the same ColumnGeometry. Then a drag that crosses a narrow column lands exactly where it previewed — no jump on release. Two different pixel formulas diverge precisely where the widths are unequal, which is exactly the new case.

Don't re-anchor the scroll on a pan

The scroll-anchor preserves the left-edge date when the column widths change (zoom, container resize, toggling the feature): read the date under the previous geometry, re-locate it under the new one (a flat scrollLeft×ratio is wrong once widths vary). But a pan (Prev/Next week shifts the origin and rebuilds the days array) must not re-anchor — preserving scrollLeft while the window slides is exactly what advances the view by a week. Distinguish them by the day array's referential identity (days === prevDays ⇒ a width change, not a pan), and key the effect on the memoised geometry.

Make "N weeks" still mean N weeks — the weekend-aware fit

The zoom picks a dayWidth to fit weeks weeks. The naïve fit floor(avail / (weeks×7)) assumes 7 equal columns — but with two narrow columns a week occupies only 5×dayWidth + 2×weekendWidth, so the viewport under-fills and a "1-week" view creeps to ~1.5. Make the fit width-aware (weeks×(5×dayWidth + 2×weekendWidth) = avail, applied only while the special columns are genuinely narrower), and raise the max-column-width cap so the widened weekday columns can actually fill a normal screen. Leave the cap and the right edge simply under-fills — cosmetic, but the owner notices on the very first click.

What stays put: the weekend-aware date math (working-day counts, the gesture model) is untouched — it works on dates and working-day counts, not pixels. The pref itself is an ordinary device-global toggle (page 05): its own localStorage key, default on, never on the account and never in the exported dataset. The whole feature is a small pure module plus a textbook pref; the engineering is the disciplined threading and the three invariants above — and an E2E with a screenshot, because that is the only oracle that sees them.

Styling & the design tokens

Tailwind v4. The entire palette is semantic CSS custom properties declared in a @theme block in src/index.css; component classes use only the semantic names (bg-surface, text-ink, text-muted, border-line) — never Tailwind scale names like text-gray-500.

@custom-variant dark (&:is([data-theme="dark"] *));   /* shadcn `dark:` utilities track the house attribute */

@theme {

  --color-weekend: var(--c-weekend);  /* the non-working-day band — its OWN token, never the canvas */
  --color-line-soft: …;  --color-line-faint: …;  --color-ok: …;  /* …soft + soft-ink pairs… */

  color-scheme: light; }

Load-bearing rule

Dark mode is one attribute swap (data-theme="dark" on <html>), not per-component dark: classes. JS resolves light/dark/system to a concrete attribute; the CSS only ever sees light or dark. Because components use semantic tokens, they flip automatically. Every colour value is tuned to clear WCAG AA in both themes (the comments record the contrast adjustments).

Other token-level decisions: a global :focus-visible ring on all interactives; preset swatch palette only; two custom keyframes (capacitylens-pop for Modal/Popover/palette entrances, capacitylens-fade for the Modal backdrop — Sonner animates its own toasts); and a global prefers-reduced-motion block that nulls all animation. The four schedule status signals keep distinct colour families so none reads as another: over-capacity = saturated rose (bg-danger-cell — the cell is empty, so no text-contrast bound applies), weekend = the cool recessed --c-weekend band, today = brand, time-off = amber hatch. The metric is always called "Utilisation", never "Load" — in labels, store keys (utilizationPrefs), and identifiers.

Accessibility is a gate (page 08), so the patterns are consistent and pervasive:

The collapsed-sidebar rail trap, handled correctly

The icons-only rail's icons do not navigate — they re-open the menu — so they are aria-hidden + tabIndex={-1}, and the single labelled Collapse/Expand toggle is the only accessible control. When a decorative affordance overlaps a real one, hide the decoy from assistive tech and leave exactly one labelled path.

React/TS conventions

  • Function components only (the one class is ErrorBoundary, which must be). Props typed inline as an object literal in the signature; only exported shapes (Option, BarLayout) get a named interface.
  • React.memo on hot-path scheduler components only (AllocationBar, ResourceLane), wrapping a named function for DevTools. Parent callbacks are stabilised with useCallback so memo is effective.
  • Event handler naming: props are onClose/onAdd/onConfirm; internal handlers are named by action (submit, requestClose) — no handle prefix.
  • data-testid is kebab-case, named for the role (client-row, allocation-bar, over-marker). Functional data attributes like data-resource-id are used for drag/snapshot logic, distinct from testids.
  • The ref-for-latest-callback pattern in effects: store onClose in a ref updated each render, so an empty-deps focus-trap effect runs once but always calls the latest function — avoiding effect re-runs on every store mutation while a dialog is open.
