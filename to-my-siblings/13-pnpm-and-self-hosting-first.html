Reference · migration guide

Migrate your repo to pnpm, and make self-hosting the front door

CapacityLens did this on 2026-07-08. This page is the recipe for a sibling repo, written to be followed step by step without improvisation: every phase ends with a verification command, every known pitfall is named where you'll hit it, and the test changes are spelled out exactly. Do the phases in order; don't start a phase until the previous one's check passed.

What changes, and why

Two independent upgrades, done together because the second rewrites docs the first also touches:

  1. npm → pnpm. Same dependency graph, different installer. What you gain: a strict, non-hoisted node_modules (importing an undeclared package fails loudly — the package-manager version of our drift-proofing), faster installs, and an explicit workspace:* link between the app and the shared core instead of the ambiguous "*" (which npm would happily resolve from the public registry if the workspace link ever broke — a real supply-chain hazard for a package literally named @yourbrand/shared).
  2. Self-hosting docs lead with bare metal. Owner call (2026-07-08): the people who self-host this kind of tool run a Linux box with nginx and systemd; Docker is the least likely path. Keep the Compose stack fully documented, but as the final section, not the front door.
Definition of done

Root gate green · server gate green · full Chromium e2e green · pnpm run dev boots both halves (verify visually, not just by exit code) · no file in the repo invokes npm/npx outside append-only history · docs/self-hosting.md quick start has no Docker in it.

Phase 0 — Baseline & names

0.1 — Prove the repo is green BEFORE touching anything. If any of these fail now, stop and fix that first; otherwise you cannot tell migration breakage from pre-existing breakage.

git status          # must be clean — commit or stash everything first
npm run gate        # tsc + eslint + vitest + build, all green
npm run gate:server # needs Node 24 (nvm use 24)
npm run e2e         # full Chromium suite green

0.2 — Toolchain. You need Node 24 (nvm use 24; the server's node:sqlite requires it) and pnpm ≥ 10 (corepack enable gives you one; pnpm --version to confirm). Record the exact version — you'll pin it in a moment.

0.3 — Find YOUR names. Everything below uses CapacityLens names. Substitute yours, found here — do not guess:

Placeholder in this guideCapacityLens valueWhere yours is defined
<server-pkg>capacitylens-serverserver/package.json → "name"
<shared-pkg>@capacitylens/sharedshared/package.json → "name"
<pnpm-version>10.33.0pnpm --version on your machine

Phase 1 — Manifests

1.1 — Root package.json:

  • Add "packageManager": "pnpm@<pnpm-version>" right after "engines". This is what lets contributors get the right pnpm from a bare corepack enable.
  • Delete the "workspaces" array — pnpm ignores it; leaving it invites drift.
  • In dependencies: "<shared-pkg>": "*" → "<shared-pkg>": "workspace:*".
  • In every script: npm run X → pnpm run X, and the workspace forms:
npm formpnpm form
npm test -w <server-pkg>pnpm --filter <server-pkg> test
npm run type-check -w <server-pkg>pnpm --filter <server-pkg> type-check
npm start -w <server-pkg>pnpm --filter <server-pkg> start
npx playwright testpnpm exec playwright test
npm ci / npm ci --include=devpnpm install --frozen-lockfile

1.2 — server/package.json: same workspace:* change for <shared-pkg>.

1.3 — Create pnpm-workspace.yaml at the repo root:

pnpm-workspace.yaml
packages:
  - shared
  - server

# pnpm blocks dependency postinstall scripts by default; esbuild's is the one we
# trust and need (it links the platform binary Vite/Vitest shell out to).
onlyBuiltDependencies:
  - esbuild
Do not skip onlyBuiltDependencies

pnpm 10 refuses to run dependency install scripts unless approved. Without the esbuild entry the install "succeeds" but Vite/Vitest die at run time with a missing-binary error. If your sibling has other native deps, run pnpm ignored-builds after Phase 2 and approve only what's needed — approval is a trust decision, not a checklist item.

Phase 2 — Lockfile & install

nvm use 24
pnpm import                    # converts package-lock.json -> pnpm-lock.yaml, preserving resolutions
rm package-lock.json
rm -rf node_modules server/node_modules shared/node_modules
pnpm install
pnpm ignored-builds            # should list ONLY esbuild (already approved in Phase 1.3)
pnpm rebuild esbuild           # run its postinstall now that it's approved

Check: pnpm run gate — the whole root gate, green. If tsc or Vite now reports a module it can't find, that's pnpm's strictness catching a phantom dependency: some file imports a package no manifest declares (it worked before only because npm hoisted it). The fix is always the same — add the package to the package.json of the workspace that imports it. Never "fix" it with hoisting config.

Phase 3 — Everything that spawns npm

Scripts and configs invoke npm by name; grep, don't recall:

grep -rn "npm\|npx" scripts/ playwright.config.ts vite.config.ts
  • scripts/dev-fullstack.mjs — spawn('npm', …) → spawn('pnpm', …), and the api child's args change shape: ['run','dev','-w','<server-pkg>'] → ['--filter','<server-pkg>','run','dev']. The launcher's kill-tree/process-group comments mention npm — update them (pnpm is pnpm.cmd on Windows, so the shell:true rationale carries over unchanged).
  • scripts/e2e-all.mjs / scripts/e2e-browsers.mjs — spawnSync('npx', ['playwright','test',…]) → spawnSync('pnpm', ['exec','playwright','test',…]).
  • playwright.config.ts — every webServer command: string (dev:demo, start:e2e, dev:api, start:auth-e2e, dev:auth): npm run → pnpm run. The two server entries keep their cwd: './server' — a bare pnpm run in that cwd resolves the server workspace's scripts, so no --filter needed there.
  • Comment sweep: source comments that tell a human what to type (vite.config.ts's dev-proxy note, test-file comments naming npm run gate) should say pnpm now. Sweep with grep; leave nothing behind.
The ppnpm trap

If you do this sweep with overlapping find-and-replace patterns (e.g. both `npm run→`pnpm run and npm run dev→pnpm run dev), a string matching both becomes ppnpm. It happened during the CapacityLens migration. After any bulk replace, run grep -rn "ppnpm" . and expect zero hits.

Check: pnpm run dev boots both halves — curl http://localhost:8787/api/health returns {"ok":true} and :5173 serves the app. Then stop it.

Phase 4 — Test updates (required)

One test is guaranteed to break, and it breaks silently late (only after you delete package-lock.json, which happens after your first green gate — so a naive "gate was green, ship it" misses it).

4.1 — src/test/privacy-posture.test.ts, the transitive backstop. Its third test reads package-lock.json and scans every installed package name against the analytics/telemetry/email denylist. Port it to pnpm-lock.yaml. Do not add a YAML parser dependency to do it — a heavier dep tree is exactly what this test polices. pnpm's lockfile keys every resolved package as a 2-space-indented name@version: line, so a line scan is enough. Replace the test body with:

src/test/privacy-posture.test.ts — the transitive backstop, ported
it('the resolved lockfile installs no denylisted package (transitive backstop)', () => {
  const raw = readFileSync(join(REPO_ROOT, 'pnpm-lock.yaml'), 'utf8')

  const installed = new Set<string>()

  // pnpm lockfile (v9): the `packages:`/`snapshots:` sections key every resolved package as a
  // 2-space-indented `name@version:` line (optionally quoted, optionally with a `(peer@ver)`
  // suffix), e.g. `  '@babel/code-frame@7.26.2':` or `  use-sync-external-store@1.4.0(react@19.2.6):`.
  // A package pulled from git/a tarball/a local path instead of the registry keys as
  // `name@https://...`, `name@git+...`, `name@file:...`, or `name@link:...` — no digit after the
  // `@` — so a version-digit-only pattern silently skips those rows, letting a denylisted package
  // slip the scan via a git fork. We extract the NAME (everything before the specifier) line-by-line
  // rather than adding a YAML parser dependency — a heavier dep tree is exactly what this test polices.
  const keyLine = /^ {2}'?((?:@[^\s/']+\/)?[^\s@']+)@(?:\d|https?:|git\+|file:|link:)/
  for (const line of raw.split('\n')) {
    const match = keyLine.exec(line)
    if (match) installed.add(match[1])
  }

  // Non-vacuous: we must have parsed a real tree. (A lockfile format change that stops these
  // lines matching must fail HERE, loudly — not silently scan nothing and vacuously pass.)
  expect(installed.size).toBeGreaterThan(100)

  const offenders = [...installed].filter((name) => DENYSET.has(name))
  expect(offenders, `lockfile installs denylisted package(s): ${offenders.join(', ')}`).toEqual([])
})

Also update the file's header comment ("root lockfile") and the REPO_ROOT comment that names package-lock.json. The non-vacuous floor moves from > 0 to > 100 — with a line-regex scan, "found a handful" is indistinguishable from "regex rotted"; a real tree has hundreds of packages.

4.2 — Hunt for other lockfile readers. CapacityLens had exactly one; verify your sibling:

grep -rn "package-lock" --include="*.ts" --include="*.tsx" --include="*.mjs" src server shared scripts e2e

Every hit outside append-only history must be ported or deleted. Zero hits = done.

4.3 — What does NOT need changing. Vitest, Stryker, Playwright, and axe configs are package-manager-agnostic — no config edits beyond the webServer commands in Phase 3. Test counts stay identical; if a unit test newly fails, it's a phantom dep (Phase 2) or the lockfile test (4.1), nothing else.

Expect e2e flake under load, and don't misread it

The first full e2e run after migrating, CapacityLens saw 7/181 failures — all of which passed in isolation and in a clean re-run. Cause: the suite ran while installs and other work hogged the machine, not pnpm. Rule: a spec only counts as migration breakage if it fails in a quiet, isolated re-run (pnpm exec playwright test <spec> --project=chromium). Never chase a red suite from a busy machine.

Phase 5 — Docker files

Docker demotes to the alternative path in Part B, but it stays shipped and must stay correct. Three mechanical changes and one structural one in the Dockerfile:

  • deps stage: add ENV COREPACK_ENABLE_DOWNLOAD_PROMPT=0 + RUN corepack enable; copy pnpm-lock.yaml pnpm-workspace.yaml instead of package-lock.json (the workspace yaml carries the esbuild approval, so the in-image install runs the same postinstall policy as a host install); RUN npm ci → RUN pnpm install --frozen-lockfile.
  • build stage: npm run build → pnpm run build.
  • api stage: also corepack enable, and the structural bit — pnpm keeps real packages in node_modules/.pnpm and gives each workspace its own node_modules of relative symlinks into it, so the runtime image must copy all three trees together: 
    COPY --from=deps /app/node_modules ./node_modules
COPY --from=deps /app/shared/node_modules ./shared/node_modules
COPY --from=deps /app/server/node_modules ./server/node_modules
    and the entrypoint becomes CMD ["pnpm", "--filter", "<server-pkg>", "start"].
  • .dockerignore: update the header comment's npm ci / package-lock.json mentions.

Check: docker compose up --build on a machine that has Docker; confirm the api container goes healthy and the SPA serves on :8080. If no Docker runtime exists on your dev machine (true of the CapacityLens dev Mac), verify what you can — run the api image's exact entrypoint semantics on the host:

# must REFUSE (production interlock, auth off, no consent flag):
NODE_ENV=production CAPACITYLENS_DB=/tmp/guard.db pnpm --filter <server-pkg> start
# must BOOT and answer /api/health:
NODE_ENV=production CAPACITYLENS_ALLOW_OPEN_IN_PRODUCTION=1 CAPACITYLENS_DB=/tmp/guard.db \
  pnpm --filter <server-pkg> start

…and log honestly that the image build itself is unverified until someone runs it on a Docker host. Unverified-but-logged beats assumed-working.

Phase 6 — Docs sweep

Apply the Phase 1.1 mapping table to every living doc: README.md, CONTRIBUTING.md, CLAUDE.md, DECISIONS.md, DEFENSIVE-CODING.md, NEEDS-INPUT.md, server/README.md, docs/{deploy,self-hosting,runbook,production-plan,production-posture,privacy}.md, user-stories/REFERENCE.md. Where a doc lists prerequisites, add that pnpm arrives via corepack enable, pinned by packageManager. package-lock.json mentions become pnpm-lock.yaml.

Never rewrite history

docs/decisions-log.md and CODE_REVIEW.md are append-only records of what happened — entries that say npm run gate were true when written. Don't touch them, and don't "fix" historical statements ("0 npm vulnerabilities at review time") in any doc. Only live instructions change.

Check: grep -rn "npm " *.md docs user-stories server/README.md | grep -v pnpm — every remaining hit must be in append-only history or the literal string "npm registry".

Phase 7 — Full verification

All of these, in order, on Node 24. Do not skip any because "it passed earlier" — Phases 3–6 changed files after your earlier runs:

pnpm run gate            # root: tsc + eslint + vitest + build
pnpm run gate:server     # server type-check + tests
pnpm run e2e             # full Chromium suite (quiet machine — see Phase 4)
pnpm run dev             # boots api + web; screenshot the app, don't just curl it

Then the repo-wide residue greps: grep -rn "ppnpm" . (zero), grep -rln "spawn('npm'\|spawnSync('npx'" scripts/ (zero), and the Phase 6 docs grep.

Part B — Self-hosting-first docs

The insight that drives the restructure: the containers are packaging, not architecture. A deployment is (1) the API daemon on Node 24 and (2) dist/ behind any web server that proxies /api/* to it — the daemon deliberately does not serve the SPA (page 10). Docker merely wraps those two pieces. So the doc should teach the pieces first and the wrapper last.

B.1 — Restructure docs/self-hosting.md to this exact section plan (CapacityLens's finished version is the reference implementation — crib from it directly rather than re-deriving):

The interlock stays the star of the quick start

Whichever path leads, the pick-an-access-mode step goes before the first boot command, and the troubleshooting section keeps the "daemon exits / restart-loops → read the refusing-to-start line" entry. On bare metal the symptom is a systemd restart loop and the log is journalctl -u <service> -e; in Docker it's restart: unless-stopped churning and docker compose logs api. Same guard, two costumes.

B.2 — Sweep the cross-references. Grep for the places that call Docker the default and flip them: docs/production-posture.md (called self-hosting.md "the end-to-end Docker setup"), this handbook's page 10 deploy-shapes table ("default, recommended" moves to the bare-metal row), and any README framing. Verify every internal #anchor in the restructured doc still resolves — renumbering sections silently breaks them.

B.3 — DECISIONS.md gains one line under Deployment/self-host: bare metal is the primary path, Docker the packaged alternative, don't write new docs or features that assume Docker is present.

Log the decision

Per page 09: append to docs/decisions-log.md (read the tail, never the whole file) — one entry for the pnpm migration (name the esbuild approval, the workspace:* change, the privacy-test port, and exactly what was and wasn't verified), one for the docs flip. Promote both standing calls to DECISIONS.md: pnpm under Testing & process, bare-metal-first under Deployment / self-host. CapacityLens's 2026-07-08 entries are the template.
