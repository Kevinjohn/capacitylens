Auth & access control

Sessions, roles, memberships, invites, redaction, audit and erasure — the whole trust layer, and the seams that keep it swappable and keep the auth-off deploy byte-identical. This page exists because the layer is large; the principles it obeys are just two: everything hangs off a narrow port, and OFF is a guarantee.

The three modes, and the OFF guarantee

CAPACITYLENS_AUTH selects the mode at boot; an unknown value refuses to boot (AuthConfigError — fail closed, never "accidentally open"):

ModeWhat it isRequires
off (default)Trusted-local. No Better Auth constructed, no auth tables created, no login UI mounts; every request carries the synthetic DEMO_USER (emailVerified: true, a non-routable demo@capacitylens.local). The open dataset is the deliberate self-host default.—
passwordEmail + password sessions. Self-registration is closed by default (disableSignUp) — membership arrives by invite; CAPACITYLENS_ALLOW_OPEN_SIGNUP=1 is the interim trusted-instance escape.BETTER_AUTH_SECRET (≥ 32 chars) + BETTER_AUTH_URL
ssoGeneric OAuth2/OIDC via the Better Auth plugin, configured from CAPACITYLENS_SSO_* (discovery URL, or explicit authorization + token URLs).The secret/URL pair + the SSO vars

Native social providers (Google, Microsoft, GitHub) are additive in either on-mode: each is configured only when both its CAPACITYLENS_<PROVIDER>_CLIENT_ID and _SECRET are set — fail-closed-absent, no half-configured provider.

The OFF guarantee (the #1 invariant of this whole page)

Auth off means byte-for-byte the pre-auth behaviour. Concretely: no adapter is built at all; the route gate short-circuits to allow-all on its first line; the member endpoints return empty lists / inert no-ops; the client's AuthProvider passes straight through. The proof is the test suite: every pre-auth server and E2E test runs in off mode and must stay green. When you add any auth feature, the first question is "what does this do in off mode?" and the answer must be "nothing".

…but production refuses an accidental OFF

Under NODE_ENV=production the posture interlock (page 07) refuses to boot with auth off — the open dataset must be an explicit decision (CAPACITYLENS_ALLOW_OPEN_IN_PRODUCTION=1, which downgrades the refusal to a loud warning). Dev, e2e and trusted-LAN self-hosting keep OFF as a first-class supported mode.

The AuthAdapter port — one method, one contract

server/src/authAdapter.ts
// The provider-neutral session-verify port. Everything above auth (requireUser,
// /api/auth/me) depends ONLY on this — never on Better Auth directly.
export interface AuthAdapter {
  verifySession(headers: IncomingHttpHeaders): Promise<SessionUser | null>
}
// betterAuthAdapter wraps Better Auth's getSession. OFF mode builds NO adapter at all.
The null/throw contract

null means "no session" → 401. A thrown error means "the auth backend failed" → 503 — never swallowed into null. Collapsing the two would turn an auth outage into a silent mass logout, which is exactly the kind of quiet failure this codebase exists to avoid. Swapping Better Auth for anything else is one new adapter behind this port.

In auth-on deployments, a root-level requireUser preHandler walls off every /api/* route except /api/health (uptime) and /api/auth/* (the login machinery; /api/auth/me answers its own 401 with { authMode, error } so the client can tell "not signed in" from "server broken"). This is pinned by a table-driven 401-matrix test over every route family — the hosted posture serves zero unauthenticated app data.

SessionUser — the normalized principal

{ id, email, emailVerified, name }, with email and emailVerified both required. normalizeSessionUser is the single narrowing boundary from the provider's user object: a provider that omits verification is treated as unverified (?? false) — no provider allow-list, just a safety net. This matters because email-preauthorised invites bind only when emailVerified === true.

The client side: two providers, no client-side auth flag

  • AuthProvider (mounted in main.tsx, outside the router) asks GET /api/auth/me once at boot. The server-reported authMode is the only auth flag — there is no VITE_* auth var to drift from the server's reality. off → pass-through; 401 → the lazy LoginScreen. It re-checks when a write later hits persistError, so an expired session walls the app off instead of failing writes silently. It also carries the capability facts (canCreateAccount, multiAccount) — parsed fail-open, because the server is the enforcer.
  • PermissionProvider (inside the shell, around the app body) resolves the caller's role for the active account from GET /api/accounts. The load-bearing default: role === null → editable (OFF/local/not-fetched/no-provider all behave like today); it fails open on a fetch blip because the server 403 backstops, and failing closed could lock out a legitimate editor. The role resets on tenant switch so a prior account's role can't leak.
  • The cosmetic fake sign-in (demo polish) mounts only when authMode === 'off', so it can never stack with the real login wall. It is a device-global flag, not auth — the real seam stays src/auth/.

Roles & can() — the single pure authority

Both sides consume the same matrix from shared/src/domain/access.ts (page 04): Role = owner | admin | editor | viewer, Action = read | write | manageMembers | manageInvites | purge | transferOwnership, and one pure can(role, action) (no I/O, no session, no clock). The matrix: read = any member; write = editor+; manage members/invites and purge = admin+; ownership transfer = owner only. Writes that store a role validate it against the union and throw on an unknown value — an access level is never silently coerced.

The exclusion invariant (principle 8)

Control tables are structurally excluded from the app-data universe: never in shared AppData/SCOPED_KEYS, server TABLES/CREATE_ORDER/ SCOPED_ORDER, KNOWN_KEYS, fixtures, sanitizeImportedRecord, loadState, generic /api/:entity CRUD, or import/export. They are reached only through their own narrow accessors (upsertMember, getMemberRole, listMembershipsForUser, createInvite…), which the permissioned endpoints wrap. A leak would hand out live role-bearing tokens through a dataset export. Any future control table (billing, quotas — the parked entitlementsFor() seam) follows the same rules.

requirePermission — the auth-on route gate

One authorize(req, reply, accountId, action) seam in buildApp:

  • OFF = a no-op allow-all on its first line — resolveRole/can never run. (The #1 invariant, again.)
  • Auth-on: resolveRole → null (non-member) or can(role, action) false → 403 { error: 'Forbidden.' }. No 401/503 here — requireUser handled those upstream.
  • Gated: the per-account read (read), every scoped write + import (write), and POST /api/batch with a pre-scan — a batch touching an unauthorized account is rejected whole, one 403, no partial write.
  • Account create stays open (new-user onboarding has no membership) — but the single-company cap still applies, account update is gated write, and account hard-delete is gated purge on both vectors (the direct route and the batch op) because a delete cascades total tenant destruction.
  • Per-account hydration completes the model: a signed-in user picks a tenant from their memberships and loads only that slice; the cross-tenant whole-read is closed (page 05, page 07).

Invites — single-use, expiring, never emailed

  • Create (POST /api/invites, admin+ via manageInvites): mints a randomBytes(32) base64url token, 7-day default TTL, optional preauthEmail (stored normalized; malformed → 400). An owner-role invite requires an owner caller — closing the admin→owner escalation. The token is never logged and never read back: invites carry a separate non-secret id for list/revoke, and the list endpoint never returns tokens.
  • Accept (POST /api/invites/:token/accept, no authorize — membership is the output): unknown → 404, used → 409, expired → 410. Success upserts the membership and marks the invite used in one transaction; single-use is double-guarded (handler check + a WHERE usedAt IS NULL SQL backstop against a race).
  • Preauth: a non-null preauthEmail binds only a caller whose verified email matches (preauthInviteAllows, pure + unit-tested); a wrong-email or unverified caller gets 403 and the invite stays live for the right person. Nothing is ever emailed — the admin distributes the link; preauth only narrows who may redeem it.
  • The client route /invite/:token sits outside the tenant gate but inside AuthProvider, so an unauthenticated visit shows the login first and the post-login reload lands back on the same URL — the token survives the wall.

Member management — pure guards + a server-only backstop

The Settings → Members section drives five endpoints (list/patch/remove members, list/revoke invites) plus ownership transfer. The who-may-touch-whom rules are two pure shared guards used by both the client affordances and the server routes — canManageMemberRole(actor, target, next) and canRemoveMember(actor, target) (an admin may never grant owner or touch an owner). The one rule that needs DB I/O — never demote or remove the sole active owner — is enforced server-side only. transferOwnership is owner-only and atomic: promote target + demote caller to admin in one step. The UI self-gates: a 403 on the members read renders nothing, so a viewer or editor never sees the section.

All three read the same pure can(), so they cannot disagree about what a viewer may do.

Field redaction — the time-off note

A time-off note can be sensitive ("surgery"), so it is redacted server-side for editors and viewers: readSlice takes a required includeTimeOffNote (every caller decides; no silent default), and false strips the key before serialization — the note never crosses the wire to a role that may not see it. canSeeTimeOffNote(role) (owner/admin) is the pure rule; a test asserts the sentinel note is absent from the raw response body. Client-side hiding would not be redaction — the pattern to copy is "sensitive fields leave the server only for roles allowed to read them".

Per-tenant export & delete-with-erasure

  • Export is not a new route: the complete per-tenant backup is GET /api/state?accountId=X&includeInactive=1 (admin-gated; archived + soft-deleted included; control tables structurally excluded because readSlice never reads them).
  • Account hard-delete routes through erasure.ts, in one transaction: delete the account (FK-cascades the scoped data) → bulk-delete its member rows and invites (they have no FK, so they would otherwise leak) → for each affected user with no remaining membership anywhere, scrub the auth user row (name → "Removed member", email → a globally-unique deleted-…@invalid, image → null), unlink SSO, kill sessions. Multi-account members are retained untouched. The batch vector reuses the outer transaction (eraseAccountInTx — node:sqlite has no nested BEGIN).
  • The deliberate consequence: deleting your sole account erases your own identity and kills your session — the post-delete read-back is a 401, not an empty 200. "PII provably erasable" applied to the last-membership case; irreversible by design.

Cookies & session pinning

The single betterAuth({...}) call pins what a library default change could silently re-tune: sameSite: 'lax' (not strict — strict drops the cookie on the top-level OAuth redirect back from the IdP, breaking SSO; lax is safe for that callback), httpOnly: true, expiresIn: 7d / updateAge: 1d, and useSecureCookies tied to the threaded https flag (the same CAPACITYLENS_HTTPS that gates HSTS — auth.ts never reads the env itself). Secure must be false over plain HTTP or the browser drops the cookie and login loops — the default deploy and the E2E are HTTP. Better Auth's telemetry is explicitly disabled (telemetry: { enabled: false }) — part of the privacy proof (principle 14).

Porting this layer to a sibling

Copy it wholesale — it is domain-agnostic by construction. The parts that touch your domain are exactly three: the Role/Action matrix (keep the shape, rename nothing), which entities are lifecycle-managed, and which fields are role-redacted (CapacityLens has one: the time-off note — find your sibling's equivalent). Everything else — the port, the control tables, invites, the gate, erasure, the cookie pinning — transfers unchanged. Test it the same way too: the auth-backed E2E project boots a real password-mode server (page 08), and the OFF guarantee is proven by the entire pre-auth suite staying green.
