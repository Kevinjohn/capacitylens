The server

The Fastify + node:sqlite REST API that is the app's default persistence backend. It is deliberately a thin shell over the shared core — it owns transport, storage, access control and ops concerns, and delegates every domain rule to the same code the client runs. (The auth & access-control layer gets its own page: 12.)

Stack & the thin-shell rule

Fastify 5 (with its bundled pino logger, @fastify/helmet and @fastify/rate-limit), node:sqlite (DatabaseSync + the online backup API) — no ORM, no driver library — and Better Auth for the opt-in session layer. Node 24+ is required because node:sqlite needs no flag there. The server runs its TypeScript directly via tsx (no build), and consumes @capacitylens/shared as a workspace dependency.

Load-bearing rule

The server contains no copy of any domain rule. Validation, sanitisation, integrity, cascade, import, the permission matrix, and the lifecycle machine all come from @capacitylens/shared. The server's job is HTTP + SQLite + sessions + ops. When a request needs to validate, it calls the client's validators. "Server validation is client validation" is structural here, not a goal.

App structure & the route table

The app is built by a factory, buildApp(db, opts), returning a Fastify instance. The entry point calls it once; tests call it repeatedly against a :memory: DB via app.inject() (no separate HTTP client). Hooks register in a deliberate order, and routes live in a child plugin so the rate-limit plugin's onRoute hook can attach to them. The surface has grown well beyond generic CRUD:

Method · PathPurpose
GET /api/healthLiveness. Deep mode proves a DB read + surfaces audit: 'ok'|'degraded' (200/503). Exempt from rate limiting and from the auth wall. Used by Playwright's webServer probe and uptime monitors.
GET /api/auth/meIdentity + capabilities — exists in every mode so the client never forks. Reports authMode, the user, and the capability facts (canCreateAccount, multiAccount). Off → the demo identity; on → the session user (401 if none).
GET | POST /api/auth/*Better Auth's own machinery (login, sign-up, OAuth callbacks) — mounted only when auth is on.
GET /api/accountsThe tenant list for the picker: off = all accounts; auth-on = the caller's memberships, each with its role.
GET /api/state?accountId=One account's scoped slice — backs the client's per-account loadAll(accountId). Membership-gated when auth is on; ?includeInactive=1 is the admin/export read (archived + soft-deleted included). The no-arg whole-tree read survives only in off mode (auth-on → 400).
GET /api/meta{ hasData } from the init marker — backs hasExisting().
POST /api/:entity · PUT/PATCH/DELETE /api/:entity/:idGeneric CRUD. PUT is an idempotent upsert (URL/body id must match; ownsRow; optional optimistic concurrency); PATCH is a true partial merge then validate; scoped DELETE requires ?accountId=.
POST /api/batchThe write path the sync adapter uses. { ops: [...] } applied in one transaction, in op order; permission pre-scan rejects a mixed-tenant batch whole (page 12).
POST /api/import{ accountId, data } → reuses parseData + remapAndValidateImport → replaces that account's slice.
POST /api/orgsThe atomic org-create path: account + its built-in Internal client + an Owner membership, one transaction. Gated by first-run / off-mode / admin-of-some-account / bootstrap token (page 12).
POST /api/:entity/:id/{archive,unarchive,delete,purge}The lifecycle actions (resources/clients/projects) — the only routes that set or clear a tombstone; interlocks and role tiers enforced server-side (delete requires archived; purge requires admin + a 30-day tombstone).

POST /api/test/resetWipe (+ optional reseed). Gated by CAPACITYLENS_ALLOW_RESET=1; refused in production by a boot-guard.
Two ordering subtleties worth copying

(1) The CORS/preflight hook is on the root instance, not the routes child — OPTIONS requests match no route, and only root-level hooks run on the not-found path; moving it into the child once silently broke all cross-origin writes. (2) Inside /api/batch, loadState() is read inside the transaction, so a child op validates against a parent upserted earlier in the same batch.

Two-layer validation on every write

Every POST/PUT/PATCH/batch-op passes through two shared-core layers, so a hand-crafted request can't persist junk:

server/src/validate.ts
// Layer 1 — sanitizeWrite: requires a non-empty id, then repairs value-level fields
// using the SAME sanitiser the import path uses (one code path = parity). It also PINS
// the lifecycle tombstones (archivedAt/deletedAt) to the stored row — only the dedicated
// lifecycle routes may set one, and an unrelated edit can't silently CLEAR one.
export function sanitizeWrite(table: string, row: Record<string, unknown>) {
  assertIdPresent(row)                // 400 if id missing/null/'' — TEXT PRIMARY KEY would store NULL otherwise
  const copy = { ...row }            // never mutate input
  if (table === 'accounts') { /* hex colour, cleanText name, drop junk schedulingMode, sanitizeAccount */ return copy }
  if (isScopedKey(table)) return sanitizeImportedRecord(table, copy)   // shared core
  return copy
}

// Layer 2 — validateWrite: referential integrity + date ranges, via shared asserts —
// plus the singleton guards (no second builtin Internal client; no un-flagging it).
// Any thrown Error becomes a ValidationError (so routes return 400, not 500).
export function validateWrite(state: AppData, table: string, row: Record<string, unknown>): void

The tenant guard, ownsRow, makes accountId immutable: a PUT/PATCH that tries to re-home a row to another account is 409; a scoped DELETE without the owning accountId is 400, and against the wrong owner 404. On the write path this is defence-in-depth (the account is still client-asserted); on the read path, auth-on deployments now have real membership isolation via readSlice + authorize (page 12). Account settings writes carry one more guard: accountFieldsFrozen rejects a change to the frozen creation-time fields (language, timezone, weekStartsOn) — per-route 409, batch 400.

TenantStore — the scoped-read/write swap point

server/src/tenantStore.ts
export interface TenantStore {
  readSlice(accountId: ID): AppData   // per-account scoped read — WHERE accountId=? on all 8 tables
  write(accountId: ID, next: AppData): void
}
export function sqliteTenantStore(db: Db): TenantStore   // the single implementation today

readSlice takes two required options so no caller gets a silent default: includeInactive (lifecycle filtering — normal reads pass false, the admin/export read passes true) and includeTimeOffNote (role-based field redaction — page 12). The no-cross-tenant invariant (no unpredicated query) lives at this layer. A future per-agency-DB or Postgres backend swaps in here, behind the interface, with no route change — this is the documented answer to "when do we outgrow SQLite?".

The auto-ALTER, compile-checked column specs

This is the server end of the entity-extension rails (page 03). Each table's columns are declared as a typed tuple, checked bidirectionally against the entity type, and the DB is auto-ALTERed to add any missing columns on open.

server/src/tables.ts
export interface ColumnSpec { name: string; json?: boolean; optional?: boolean }

// Bidirectional check: every column name is a key of E, AND every key of E is a column.
type CheckColumns<E, Cols extends readonly ColumnSpec[]> =
  Cols[number]['name'] extends keyof E
    ? Exclude<keyof E, Cols[number]['name']> extends never ? true : never
    : never

const COLS_resources = [
  { name: 'id' }, { name: 'accountId' }, { name: 'kind' }, { name: 'name', optional: true },
  { name: 'role' }, { name: 'disciplineId', optional: true }, { name: 'employmentType' },
  { name: 'workingHoursPerDay' }, { name: 'workingDays', json: true },
  { name: 'projectId', optional: true }, { name: 'color' }, ...META,
] as const satisfies ColumnSpec[]

declare const _checkResources: CheckColumns<Resource, typeof COLS_resources>  // fails build on drift

The creation/scoped orders carry the same Exclude<…> extends never exhaustiveness guard seen in syncOps.ts, and SCHEMA_SQL's FK cascade rules mirror integrity.ts exactly (annotated in comments). On open, migrateSchema introspects the live DB and adds any column present in a spec but missing in the table — so step 3 of the field-addition procedure (page 03) is "add the column spec", and the DB upgrades itself.

as const satisfies ColumnSpec[] is the magic incantation

as const makes TS infer the literal tuple (so Cols[number]['name'] is a string literal union, not just string); satisfies ColumnSpec[] keeps the element shapes checked. Together they let CheckColumns compare names to entity keys exactly. Copy this idiom for any "list that must mirror a type".

Control tables and auth tables stay OUT of these lists

Better Auth's tables (user/session/…) and the control tables (account_members, invites — own module, controlTables.ts) live in the same SQLite file but are deliberately absent from TABLES/CREATE_ORDER/ SCOPED_ORDER and every other drift-proofing list, so they can never leak through generic CRUD, loadState, or import/export. The exclusion is the invariant (principle 8).

Row codec — arrays/booleans/optionals across the SQLite boundary

Pure functions translate between domain objects and SQLite rows, driven by the column specs:

Connection, migration & the init marker

server/src/db.ts
export function openDb(path: string): Db {
  const db = new DatabaseSync(path)
  db.exec('PRAGMA journal_mode = WAL;')   // concurrent reads; cp can't catch a torn write
  db.exec('PRAGMA busy_timeout = 5000;')  // wait for a lock instead of throwing SQLITE_BUSY
  renameLegacyActivityTables(db)           // the Task→Activity rename, BEFORE the schema runs
  db.exec(SCHEMA_SQL)                      // CREATE TABLE IF NOT EXISTS … (no-op on existing tables)
  migrateSchema(db)                        // add any missing optional columns; rebuild activities if needed
  assertSchemaCurrent(db)                  // LOUD startup throw on any required-column / NULL drift
  ensureControlTables(db)                  // account_members + invites — idempotent, EVERY DB incl. :memory:
  ensureInternalClients(db)                // the one-builtin-per-account invariant, backfilled
  if (!isInitialized(db) && !isEmpty(loadState(db))) markInitialized(db)  // backfill pre-marker DBs
  db.exec('PRAGMA foreign_keys = ON;')    // enforced AFTER migration (table rebuilds are FK-unsafe)
  return db
}
Two fail-loud details

assertSchemaCurrent throws at startup on any column drift or optional?-vs-NULL mismatch, instead of failing silently on a later write — the same "fail loud, fail early" instinct as the dev-server port pin. And foreign_keys is turned ON only after migration, because table rebuilds (drop/rename) are FK-unsafe; node:sqlite defaults it off.

The init marker is a _meta row, not a row count: a user who deletes all their data keeps initialized=1, so the demo never re-seeds (mirroring the client's seed-once rule). upsertRow excludes createdAt from its SET clause — createdAt is immutable, so a re-PUT must never rewrite it. Bulk operations (insertAll, wipe, replaceAccountSlice) run inside one tx().

server/src/txn.ts (the whole helper)
export function tx(db: Db, fn: () => void): void {
  db.exec('BEGIN')
  try { fn(); db.exec('COMMIT') } catch (e) { db.exec('ROLLBACK'); throw e }
}

One node:sqlite trap the erasure code documents: there is no nested BEGIN — a helper that must run inside an outer transaction gets an explicit …InTx variant instead of opening its own (see eraseAccountInTx, page 12).

Always-on hardening (no flag, by design)

Not everything is flag-gated. These ship on because an "off" default would preserve an unsafe silence (the principle-12 test); each is named with its rationale:

@fastify/helmet security headers
A strict minimal CSP for this JSON-only API (default-src/connect-src/base-uri 'self', frame-ancestors 'none', object-src 'none'), nosniff, Referrer-Policy: no-referrer, X-Frame-Options: DENY. The one exception is HSTS, gated behind CAPACITYLENS_HTTPS=1 — HSTS over plain HTTP is harmful, and this server usually sits behind a TLS-terminating proxy. The connect-src 'self' line is also the privacy proof's server half (principle 14) — a test fails if it ever widens.
Request/connection timeouts
REQUEST_TIMEOUT_MS = CONNECTION_TIMEOUT_MS = 30 s, always applied — a slow-loris connection or a hung handler can't pin a worker forever.
Null-body 400s
Generic POST/PATCH reject a missing or non-object body with 400 "A request body is required." instead of a confusing downstream error.
The audit log (ON by default)
An append-only JSONL sink writes one line per app-data mutation at each route's commit point: {ts,userId,accountId,action,entity,id,changedFields}. The #1 invariant: changedFields is field names only — never a value, row or body, so no tenant data reaches a log line. Fail-never: append cannot throw; a broken sink never fails the request — it latches "degraded", logs one redacted line, sets an x-capacitylens-audit-warning: true response header, and deep health reports audit:'degraded' while keeping ok: true. Opt out with CAPACITYLENS_AUDIT=off; the file rotates at CAPACITYLENS_AUDIT_MAX_MB (64). Server-mode only — the sink is built in index.ts, and buildApp defaults to a no-op sink so tests are byte-identical.
The production posture interlock
A pure evaluateProductionPosture(env) consulted at boot: a no-op unless NODE_ENV==='production' (dev/e2e/self-host keep the open auth-off posture as a first-class mode), but in production it refuses to start with auth off (a world-writable dataset must be a decision, not an accident) and warns loudly on HSTS-off and open self-signup. The only escape is the explicit CAPACITYLENS_ALLOW_OPEN_IN_PRODUCTION=1, which downgrades the refusal to a warning — never silences it.
bootGuard / graceful shutdown
resetForbidden: CAPACITYLENS_ALLOW_RESET=1 + production = exit 1 before listening. Shutdown: SIGTERM/SIGINT → app.close() (drain) → db.close() → exit 0; a second signal force-exits 1.

Flag-gated modules — default OFF, fail-closed parsing

auth.ts (Better Auth)
CAPACITYLENS_AUTH=off|password|sso. Off (the default): Better Auth is never initialised — no auth tables, no login machinery, every request carries the synthetic DEMO_USER. Any unknown value refuses to boot (AuthConfigError). On modes require BETTER_AUTH_SECRET (≥ 32 chars — boot refusal below) + BETTER_AUTH_URL. Full architecture on page 12.
backup.ts
CAPACITYLENS_BACKUP_DIR set → online backup() snapshots (capacitylens-<YYYYMMDD-HHmmss>.db) once at boot then every CAPACITYLENS_BACKUP_INTERVAL_MIN (60); prune to the newest CAPACITYLENS_BACKUP_KEEP (48). The timer is unref()'d; failed snapshots log and continue. Unset → no timer, no filesystem writes.
rate limit
CAPACITYLENS_RATE_LIMIT=<n>, parsed by /^\d+$/ (fail-closed: 0/non-numeric/unset = off, plugin not registered). /api/health exempt. Behind the loopback proxy it trusts X-Forwarded-For; on a directly-exposed host it ignores it (spoofable).
logging / health-deep
CAPACITYLENS_LOG=1 → Fastify's bundled pino (JSON per request; the invite-accept URL's bearer token is redacted by a request serializer — the only path-borne secret). CAPACITYLENS_HEALTH_DEEP=1 → health proves a SELECT 1 and surfaces the audit soft-signal.
optimistic concurrency
CAPACITYLENS_OPTIMISTIC_CONCURRENCY=1 → a stale PUT is 409 instead of last-writer-wins.
The OFF-guarantee, enforced by the test suite

"Off = byte-for-byte today" is not a claim, it's a test: the entire existing server + E2E suites run in off mode and must stay green. When you add a flagged feature, the proof that its default is inert is that every prior test passes unchanged.

The single-company cap (deployment shape, not authz)

By default a server instance serves one company. accountCreateCapped returns true when CAPACITYLENS_MULTI_ACCOUNT is off and any account exists; every create vector — generic POST /api/accounts, a batch PUT of a new account, POST /api/orgs — then 403s with SINGLE_COMPANY_CAP_MESSAGE, which names the fix (set the flag). It applies in every auth mode (even the bootstrap token doesn't bypass it) because it is a deployment-shape policy, not an authorization rule. Create-time only: an existing multi-company DB keeps serving. GET /api/auth/me reports canCreateAccount so the picker can hide "New company" — fail-open on the client, because the server is the enforcer. Dev (npm run dev) and the demo build are uncapped.

HTTP status & error conventions

CodeWhen
201 / 200 / 204Create / update-read / delete (delete is idempotent — deleting a missing row is also 204).
400Validation, missing/empty id, bad FK, missing required param (incl. the auth-on no-arg /api/state read), null body, bad op shape, SQLite /constraint failed/i, a frozen-field change via batch.
401No session in an auth-on deployment (the root requireUser wall — every /api/* except health and the auth machinery).
403Authorization denials: non-member or insufficient role ({ error: 'Forbidden.' }), the single-company cap, purge-tier gates, reset disabled, a preauth invite redeemed by the wrong/unverified email.
404 / 409 / 410Unknown entity/row or cross-account delete · accountId immutability, frozen-field change (per-route), stale overwrite, used invite, delete-before-archive · expired invite.
413 / 429 / 503Body too large / rate limited / DB read failed (deep health) or the auth backend failed — a thrown session verify is 503, never treated as "no session".
Load-bearing rule

500 bodies are always generic ({ error: 'Internal server error' }) with the real error logged server-side — never leak the raw SQLite message (it exposes column names/schema). A ValidationError (a class) carries a safe, authored message and maps to 400. DB constraint failures are classified 400 (caller fault) via an exact /constraint failed/i match, not a loose one that could misclassify an unrelated 500.
