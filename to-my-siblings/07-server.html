The optional server

An opt-in Fastify + node:sqlite REST API that plugs in behind the persistence seam. It is deliberately a thin shell over the shared core — it owns transport, storage, and ops concerns, and delegates every domain rule to the same code the client runs.

Stack & the thin-shell rule

Fastify 5 (with its bundled pino logger and @fastify/rate-limit), node:sqlite (DatabaseSync + the online backup API) — no ORM, no driver library. Node 24+ is required because node:sqlite needs no flag there. The server runs its TypeScript directly via tsx (no build), and consumes @floaty/shared as a workspace dependency.

Load-bearing rule

The server contains no copy of any domain rule. Validation, sanitisation, integrity, cascade, and import all come from @floaty/shared. The server's job is HTTP + SQLite + ops. When a request needs to validate, it calls the client's validators. "Server validation is client validation" is structural here, not a goal.

server/src/README + module comments capture the spirit
// Imports the same pure domain-core the client uses — @floaty/shared/domain/mutations,
// @floaty/shared/data/{migrate,seed,transfer}, /types/entities, /lib/* — so server-side
// validation, integrity, cascade and import rules are literally the client's code.

App structure & the route table

The app is built by a factory, buildApp(db, opts), returning a Fastify instance. The entry point calls it once; tests call it repeatedly against a :memory: DB via app.inject() (no separate HTTP client). Hooks register in a deliberate order, and routes live in a child plugin so the rate-limit plugin's onRoute hook can attach to them.

Method · PathPurpose
GET /api/healthLiveness. Deep mode proves a DB read (200 {ok,db:true} / 503). Exempt from rate limiting. Used by Playwright's webServer probe.
GET /api/auth/meIdentity — exists in every mode so the client never forks. Off → the demo identity; on → the session user (401 if none).
GET /api/stateWhole AppData tree — backs the client's loadAll().
GET /api/meta{ hasData } from the init marker — backs hasExisting().

Two ordering subtleties worth copying

(1) The CORS/preflight hook is on the root instance, not the routes child — OPTIONS requests match no route, and only root-level hooks run on the not-found path; moving it into the child once silently broke all cross-origin writes. (2) Inside /api/batch, loadState() is read inside the transaction, so a child op validates against a parent upserted earlier in the same batch.

Two-layer validation on every write

Every POST/PUT/PATCH/batch-op passes through two shared-core layers, so a hand-crafted request can't persist junk:

server/src/validate.ts
// Layer 1 — sanitizeWrite: requires a non-empty id, then repairs value-level fields
// using the SAME sanitiser the import path uses (one code path = parity).
export function sanitizeWrite(table: string, row: Record<string, unknown>) {
  assertIdPresent(row)                // 400 if id missing/null/'' — TEXT PRIMARY KEY would store NULL otherwise
  const copy = { ...row }            // never mutate input
  if (table === 'accounts') { /* hex colour, cleanText name, drop junk schedulingMode, sanitizeAccount */ return copy }
  if (isScopedKey(table)) return sanitizeImportedRecord(table, copy)   // shared core
  return copy
}

// Layer 2 — validateWrite: referential integrity + date ranges, via shared asserts.
// Any thrown Error becomes a ValidationError (so routes return 400, not 500).
export function validateWrite(state: AppData, table: string, row: Record<string, unknown>): void

The tenant guard, ownsRow, makes accountId immutable: a PUT/PATCH that tries to re-home a row to another account is 409; a scoped DELETE without the owning accountId is 400, and against the wrong owner 404. This is defence-in-depth, not isolation (the account is still client-asserted — page 10's Stage C is where that changes).

The auto-ALTER, compile-checked column specs

This is the server end of the entity-extension rails (page 03). Each table's columns are declared as a typed tuple, checked bidirectionally against the entity type, and the DB is auto-ALTERed to add any missing columns on open.

server/src/tables.ts
export interface ColumnSpec { name: string; json?: boolean; optional?: boolean }

// Bidirectional check: every column name is a key of E, AND every key of E is a column.
type CheckColumns<E, Cols extends readonly ColumnSpec[]> =
  Cols[number]['name'] extends keyof E
    ? Exclude<keyof E, Cols[number]['name']> extends never ? true : never
    : never

const COLS_resources = [
  { name: 'id' }, { name: 'accountId' }, { name: 'kind' }, { name: 'name', optional: true },
  { name: 'role' }, { name: 'disciplineId', optional: true }, { name: 'employmentType' },
  { name: 'workingHoursPerDay' }, { name: 'workingDays', json: true },
  { name: 'projectId', optional: true }, { name: 'color' }, ...META,
] as const satisfies ColumnSpec[]

declare const _checkResources: CheckColumns<Resource, typeof COLS_resources>  // fails build on drift

The creation/scoped orders carry the same Exclude<…> extends never exhaustiveness guard seen in syncOps.ts, and SCHEMA_SQL's FK cascade rules mirror integrity.ts exactly (annotated in comments). On open, migrateSchema introspects the live DB and adds any column present in a spec but missing in the table — so step 3 of the field-addition procedure (page 03) is "add the column spec", and the DB upgrades itself.

as const satisfies ColumnSpec[] is the magic incantation

as const makes TS infer the literal tuple (so Cols[number]['name'] is a string literal union, not just string); satisfies ColumnSpec[] keeps the element shapes checked. Together they let CheckColumns compare names to entity keys exactly. Copy this idiom for any "list that must mirror a type".

Connection, migration & the init marker

server/src/db.ts
export function openDb(path: string): Db {
  const db = new DatabaseSync(path)
  db.exec('PRAGMA journal_mode = WAL;')   // concurrent reads; cp can't catch a torn write
  db.exec('PRAGMA busy_timeout = 5000;')  // wait for a lock instead of throwing SQLITE_BUSY
  db.exec(SCHEMA_SQL)                      // CREATE TABLE IF NOT EXISTS … (no-op on existing tables)
  migrateSchema(db)                        // add any missing optional columns; rebuild activities if needed
  assertSchemaCurrent(db)                  // LOUD startup throw on any required-column / NULL drift
  if (!isInitialized(db) && !isEmpty(loadState(db))) markInitialized(db)  // backfill pre-marker DBs
  db.exec('PRAGMA foreign_keys = ON;')    // enforced AFTER migration (table rebuilds are FK-unsafe)
  return db
}
Two principle-12 details

assertSchemaCurrent throws at startup on any column drift or optional?-vs-NULL mismatch, instead of failing silently on a later write — the same "fail loud, fail early" instinct as the dev-server port pin. And foreign_keys is turned ON only after migration, because table rebuilds (drop/rename) are FK-unsafe; node:sqlite defaults it off.

The init marker is a _meta row, not a row count: a user who deletes all their data keeps initialized=1, so the demo never re-seeds (mirroring the client's seed-once rule). upsertRow excludes createdAt from its SET clause — createdAt is immutable, so a re-PUT must never rewrite it. Bulk operations (insertAll, wipe, replaceAccountSlice) run inside one tx().

server/src/txn.ts (the whole helper)
export function tx(db: Db, fn: () => void): void {
  db.exec('BEGIN')
  try { fn(); db.exec('COMMIT') } catch (e) { db.exec('ROLLBACK'); throw e }
}

Production-hardening modules — all flag-gated, default OFF

Each lives in its own module, is enabled by an env flag, and its unset state is exactly today's behaviour. The flag parsers are fail-closed.

auth.ts (Better Auth)
FLOATY_AUTH=off|password|sso. Off: Better Auth is never initialised — no auth tables, no /api/auth/* routes beyond the thin /api/auth/me, and every request carries a synthetic DEMO_USER. Any unknown value refuses to boot. On modes require BETTER_AUTH_SECRET + BETTER_AUTH_URL (boot refusal if missing). The auth library's own tables live in the same SQLite file but are deliberately outside the entity drift-proofing lists (commented as such).
backup.ts
FLOATY_BACKUP_DIR set → online backup() snapshots (floaty-<YYYYMMDD-HHmmss>.db) once at boot then hourly; prune to the newest FLOATY_BACKUP_KEEP (48). The timer is unref()'d (must not keep a draining process alive); failed snapshots log and continue. Unset → no timer, no filesystem writes.
bootGuard.ts
resetForbidden(env) = FLOATY_ALLOW_RESET==='1' && NODE_ENV==='production'. If true, the process exits 1 before listening — the destructive reset route can never be reachable in production.
shutdown.ts
On SIGTERM/SIGINT: await app.close() (drain in-flight) → db.close() → exit 0. A second signal force-exits 1. Injected app/db/exit make it unit-testable.
rate limit
FLOATY_RATE_LIMIT=<n>, parsed by /^\d+$/ (fail-closed: 0/non-numeric/unset = off, plugin not registered). /api/health exempt. Behind the loopback proxy it trusts X-Forwarded-For; on a directly-exposed host it ignores it (spoofable).
logging / health-deep
FLOATY_LOG=1 → Fastify's bundled pino (JSON per request); off → startup line + console.error on 500s. FLOATY_HEALTH_DEEP=1 → health proves a SELECT 1.
The OFF-guarantee, enforced by the test suite

"Off = byte-for-byte today" is not a claim, it's a test: the entire existing server + E2E suites run in off mode and must stay green. When you add a flagged feature, the proof that its default is inert is that every prior test passes unchanged.

HTTP status & error conventions

CodeWhen
201 / 200 / 204Create / update-read / delete (delete is idempotent — deleting a missing row is also 204).
400Validation, missing/empty id, bad FK, missing required param, bad op shape, SQLite /constraint failed/i.
401 / 403Auth required (on modes, no session) / reset disabled.
404 / 409Unknown entity or missing row, cross-account delete / accountId immutability or stale overwrite.
413 / 429 / 503Body too large / rate limited / DB read failed (deep health).
Load-bearing rule

500 bodies are always generic ({ error: 'Internal server error' }) with the real error logged server-side — never leak the raw SQLite message (it exposes column names/schema). A ValidationError (a class) carries a safe, authored message and maps to 400. DB constraint failures are classified 400 (caller fault) via an exact /constraint failed/i match, not a loose one that could misclassify an unrelated 500.
