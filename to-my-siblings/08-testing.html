Testing & the gate

Four test layers, two extra oracles, one green gate. The suite is the safety net that lets the codebase refactor fearlessly and the AI pair work autonomously. Copy the structure, the frozen clock, and the oracles into every sibling.

Vitest setup & the fixture factories

The setup file is minimal — jest-dom matchers and a cleanup between tests; no global mocks:

src/test/setup.ts
import { afterEach } from 'vitest'
import { cleanup } from '@testing-library/react'
import '@testing-library/jest-dom/vitest'
afterEach(() => { cleanup() })   // unmount trees + reset jsdom between tests

Test data is built by factory functions with Partial<T> overrides in src/test/fixtures.ts — every factory has valid defaults; a test overrides only the field that matters. The canonical reset for any store test sets data and the active account in one call:

src/test/fixtures.ts
const TS = '2026-05-01T00:00:00.000Z'
export function makeResourceDraft(over: Partial<Draft<Resource>> = {}): Draft<Resource> {
  return { kind: 'person', name: 'Test Person', role: 'Designer', employmentType: 'permanent',
    workingHoursPerDay: 8, workingDays: [1,2,3,4,5], color: '#6366f1', ...over }
}
export function resetStoreWithAccount(accountId = DEFAULT_ACCOUNT_ID): void {
  useStore.getState().replaceAll(makeAppData({ accounts: [makeAccount({ id: accountId })] }))
  useStore.getState().setActiveAccount(accountId)
}

Component tests — behaviour, not appearance

Queries go by semantic role or label, never CSS class or querySelector. Interactions use userEvent.setup(). Assertions cover store state, callback invocation, and accessible attributes. The ARIA wiring is tested explicitly:

it('associates the error with the offending field', async () => {
  const user = userEvent.setup()
  render(<ClientForm onClose={vi.fn()} />)
  await user.click(screen.getByRole('button', { name: 'Save' }))

  const nameInput = screen.getByLabelText('Name')
  expect(nameInput).toHaveAttribute('aria-invalid', 'true')
  const describedBy = nameInput.getAttribute('aria-describedby')
  expect(screen.getByRole('alert')).toHaveAttribute('id', describedBy)   // the link is real
})

Describe blocks are 'Component – mode'; it descriptions are full sentences that read as specifications. Every test that touches the store calls resetStoreWithAccount() in beforeEach.

Pure unit tests — local dataset factories

Pure tests build a fresh dataset per call and extract repetitive call/assert patterns into file-local helpers, so each it is one or two lines. Floating-point results use toBeCloseTo; arrays use toEqual; error messages use case-insensitive regex.

const build = (filters = emptyFilters()) => buildSchedulerModel(dataset(), start, 48, days, start, end, filters)
const barIds = (m) => m.flatMap((g) => g.rows).flatMap((r) => r.bars).map((b) => b.allocation.id).sort()

it('hideTentative removes tentative bars, but capacity still counts them', () => {
  const m = build({ ...emptyFilters(), hideTentative: true })
  expect(barIds(m)).toEqual(['a1', 'a3'])
  const r1 = m.flatMap((g) => g.rows).find((r) => r.resource.id === 'r1')!
  expect(r1.utilization).toBeCloseTo(0.6)   // filter is visual; load math is unaffected
})
Why pure view-models pay off in tests

Because buildSchedulerModel is a pure function (page 06), the hardest logic in the app — bar layout, capacity, utilisation, filtering — is tested with plain objects and zero rendering. Push logic out of components and into pure functions specifically so it can be tested this cheaply.

The Playwright projects

One config, three stack flavours, selected by file suffix. The config also boots all the dev servers it needs (three Vite instances + two server instances) with reuseExistingServer locally. A fourth, opt-in webkit project mirrors chromium for Safari coverage — see the next section.

ProjectMatchesStack under test
chromium*.spec.ts (ignores db/auth)The default localStorage app on :5173.
db-backed*.db.spec.tsThe SQLite server (:8787, reset-enabled, temp DB) + a Vite build with VITE_FLOATY_API on :5273 — exercises the real ServerSyncAdapter.
auth-backed*.auth.spec.tsA server booted with FLOATY_AUTH=password (fresh DB) + a Vite app on :5373 — the only place the flag-gated login screen exists.

Selector strategy in specs mirrors the component tests: getByRole > getByLabel > getByTestId > getByText > scoped [data-…] locators. There are no page-object classes — shared setup is one helper (openApp); everything else is file-local (e.g. a box() bounding-box helper for drag assertions). DB specs reset the server in beforeEach via the test-only reset endpoint, and can assert directly on GET /api/state to prove a write reached the DB.

Cross-browser: the opt-in Safari twin

Chromium is the default loop — fast, and where you live while building. Safari/WebKit coverage is a separate, opt-in command so the inner loop stays quick. A fourth project, webkit, mirrors chromium exactly (same testIgnore → the same core localStorage specs) but runs on devices['Desktop Safari']:

CommandRunsServers booted
npm run e2echromium + db-backed + auth-backed (Chromium)3 Vite + 2 servers
npm run e2e:webkitthe webkit project onlyVite only
npm run e2e:allthe whole matrix (Chromium + WebKit)3 Vite + 2 servers

The trick that keeps e2e:webkit cheap: it sets FLOATY_WEBKIT_ONLY, which both makes the webkit project exist and trims the config's webServer list to just the :5173 Vite dev server. The core specs never touch the SQLite or auth servers, so the WebKit run needs neither those servers nor Node 24 — it runs anywhere the app builds. (e2e:all sets FLOATY_WEBKIT instead: project on, full stack.) Both are env-gated spreads in the projects/webServer arrays, exactly like the rehearsal project — add a flavour the same way.

Why only the core specs get a WebKit twin

The db-backed and auth-backed specs exercise server round-trips and the persistence seam, not Safari rendering — re-running them on WebKit would re-test browser-agnostic plumbing. The cross-browser risk lives in the UI: pointer-drags, focus, page.clock, date fill(), and the a11y oracle. Those are the core specs, and they pass on WebKit unchanged — which is the point: keep specs browser-agnostic, never branch on user-agent. The portable-drag rule (page.mouse with steps, not HTML5 dragTo) is what lets the same gesture register on both engines.

The frozen clock — the single most important E2E decision

The seed data lives in early June 2026 and the scheduler is today-anchored (it opens scrolled to this week's Monday; the utilisation window runs forward 14 days from today). On any later wall-clock date the seed bars scroll off-screen and every spec that finds a bar by label rots. So the shared opener freezes the clock to a date inside the seed window before navigating:

e2e/helpers.ts
const FIXED_NOW = new Date('2026-06-03T12:00:00')   // the over-allocated day, inside the seed window

export async function openApp(page, company = 'Studio North', path = '/') {
  await page.clock.setFixedTime(FIXED_NOW)   // MUST precede goto — app reads it on first render
  await page.goto(path)
  await page.getByRole('button', { name: company, exact: true }).click()  // exact: avoids "Delete Studio North"
}
Load-bearing rule

The frozen date must stay in step with the seed dates. If you move the seed, move this constant. Noon is chosen so a host in any timezone still lands inside the window. The rule is documented at both ends — in helpers.ts and in the user-story REFERENCE.md — so neither can drift unnoticed.

The two oracles beyond "tests pass"

Role/DOM assertions prove behaviour but not appearance or accessibility. Two oracles cover the gap.

Screenshots — the visual oracle

The E2E suite captures real renders at key states (e.g. each zoom level) to test-results/ for human inspection. They are the oracle for visual correctness; they are not pixel-compared automatically (no snapshot assertions) — they are looked at when a visual change is suspected.

axe — the accessibility oracle

@axe-core/playwright runs on three scenarios and fails the suite on any serious/critical WCAG violation:

e2e/a11y.spec.ts (essence)
const WCAG = ['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa']
test.use({ reducedMotion: 'reduce' })   // sample SETTLED colours, not mid-fade

async function assertClean(page) {
  const results = await new AxeBuilder({ page }).withTags(WCAG).analyze()
  const blocking = results.violations.filter((v) => v.impact === 'serious' || v.impact === 'critical')
  expect(blocking, JSON.stringify(blocking.map((v) => v.id), null, 2)).toEqual([])  // failure msg = the list
}
// 1) light-mode scheduler  2) dark-mode (theme seeded via addInitScript)  3) a modal open
Three details that make the a11y oracle reliable

Dark mode is seeded by writing the theme key in addInitScript (it's a stored preference, not an OS media query). The modal scenario waits ~350 ms for the entrance animation to settle (mid-fade colours read as false low-contrast). The expect second argument is a JSON summary of violations, so a failure is self-explanatory in the terminal without opening a report.

The gate (and optional CI)

"Green" means three commands pass (recap from page 02): npm run gate (tsc -b + eslint + vitest + vite build), npm run gate:server (type-check + vitest in server/, separate because of node:sqlite), and npm run e2e. These three are the merge bar, run locally before pushing — they are plain npm scripts, so "green locally" is the whole bar.

House standard — CI is optional, off by default

These repos do not enable GitHub Actions by default, to avoid Actions cost on private repos. The bar is the local gate; run it when you need it. A workflow file is kept purely as a recipe for anyone who wants hosted CI — it runs the identical commands (on Node 24, plus a shared type-check, installing Chromium before E2E). Enable it deliberately. (On a public repo, Actions minutes are free, so turning it on there costs nothing.)

The recipe, if you switch it on (it would run on push + PR to main):

.github/workflows/ci.yml (optional)
- run: npm ci
- run: npm run gate
- run: npm run type-check --workspace=shared
- run: npm run gate:server
- run: npx playwright install --with-deps chromium
- run: npm run e2e

When Playwright runs with CI=1 set (in a hosted CI, or locally), the config switches on: forbidOnly (a committed .only fails the run rather than silently skipping the suite), 2 retries, 1 worker, and the github reporter.

The user-story ↔ E2E lock-step

Every capability has a manual user story (goal → why → numbered end-to-end steps → checkable ✅ acceptance criteria) that names its automated coverage, and the suite mirrors the stories 1:1. A single user-stories/REFERENCE.md is the source of truth for routes, control labels, data-testids, the seed data, and domain rules — both the stories and the specs lean on it.

Load-bearing rule

When the app changes, update REFERENCE.md first, then the affected stories, then the specs. The reference is what keeps a label rename from quietly breaking five specs. The full story/template system is on page 09.

Flakiness prevention, by construction

  • Frozen clock before navigation — deterministic regardless of run date.
  • strictPort: true in Vite — the suite can't silently hit the wrong port.
  • exact: true on role queries where a sibling shares a name substring.
  • steps: 8 on drag mouse-moves — enough intermediate events to register as a drag, not a click; positions asserted with a margin, never exact pixels.
  • beforeEach + server reset in DB specs — each test starts clean.
  • reducedMotion: 'reduce' + a single explicit settle wait in the modal a11y test — everything else relies on Playwright's auto-waiting locators (no scattered waitForTimeout).
  • expect.poll() for server-side write confirmation, not a fixed sleep.
