Directory tree

The repo is an npm-workspaces monorepo with three packages. Tests are co-located with the code they cover; only Playwright specs live apart in e2e/.

capacitylens-v1/
├── src/                      # the web app (the root workspace)
│   ├── auth/                 # AuthProvider, LoginScreen (flag-gated)
│   ├── components/           # per-entity folders + common/ + scheduler/
│   ├── data/                 # PersistenceAdapter seam + adapters + bootstrap
│   ├── hooks/                # reusable hooks (useCrudListState, useFieldError, …)
│   ├── lib/                  # app-only pure utils (theme, palette, capacity, fuzzy…)
│   ├── store/                # Zustand store, selectors, scoping seam
│   ├── test/                 # vitest setup + shared fixtures
│   ├── index.css             # Tailwind v4 @theme tokens (the design system)
│   ├── main.tsx              # entry: theme + bootstrap + createRoot
│   └── router.tsx
├── shared/                   # @capacitylens/shared — pure domain core (no DOM; Node-agnostic by convention)
│   └── src/{types,lib,data,domain}/
├── server/                   # capacitylens-server — optional Fastify + node:sqlite API
│   └── src/
├── e2e/                      # Playwright specs (*.spec.ts, *.db.spec.ts, *.auth.spec.ts)

├── docs/                     # decisions-log, production-plan, runbook, deploy, …

The three workspaces and why the split exists

WorkspacePackage nameRoleEnvironment
(root)capacitylensThe web app — React/Vite SPA.Browser
shared/@capacitylens/sharedThe pure domain core: types, validation, integrity, cascade, import-remap, migrate, seed, fixtures.Agnostic — no DOM lib, Node-agnostic by convention, no React
server/capacitylens-serverThe optional SQLite REST API.Node 24+
Why a shared workspace and not a folder import

Making the core a real package with its own name means the app and the server import it by the same specifier (@capacitylens/shared/...), the workspace linker guarantees they get the identical code, and the boundary is reinforced by config: the core's tsconfig omits the DOM lib ("lib": ["ES2023"]), so a browser API like document won't type-check. Node is a convention rather than a compiler wall — the tsconfig keeps "types": ["node"] (it shares the monorepo's Node toolchain), so staying off Node-specific APIs is a discipline we hold, not something the compiler forbids. The core leans on cross-platform globals only (e.g. crypto.randomUUID()). That is what makes "server validation is client validation" structurally true rather than merely intended.

Runtime dependencies, deliberately few

The whole app runs on five runtime libraries: react + react-dom, react-router-dom, zustand for state, date-fns for date math, and better-auth (only used when the auth flag is on). Tailwind is a dev/build dependency. No component library, no form library, no state-machine library, no icon package. Resist adding to this list — almost everything has been built from primitives on purpose (see the "deliberately not done" notes throughout).

shared/ and server/ package.json

The shared package is consumed as raw TypeScript source — there is no build step. The wildcard exports map routes every subpath straight to a .ts file, and the consumers' bundler/transpiler handle it.

  "name": "@capacitylens/shared",
  "version": "0.1.0",
  "private": true,
  "type": "module",

  "dependencies": { "date-fns": "^4.3.0" },
  "devDependencies": { "@types/node": "^22.10.0", "vitest": "^4.1.7" },

Install at the root, always

Run npm install from the repo root, never inside a workspace — the root install is what links @capacitylens/shared into both the app and the server. The server runs its TypeScript directly via tsx (no emit); the app is transpiled by Vite.

TypeScript configuration

The root uses project references (tsc -b) so a single build command type-checks the app and its build-tooling config as separate projects with separate settings.

tsconfig.json (the solution file)
{
  "files": [],
  "references": [
    { "path": "./tsconfig.app.json" },
    { "path": "./tsconfig.node.json" }
  ]
}

tsconfig.app.json covers src and shared/src with the DOM lib; tsconfig.node.json covers vite.config.ts with the Node types. Both are strict. The settings worth copying exactly:

tsconfig.app.json (compilerOptions)
{
  "target": "es2023",
  "lib": ["ES2023", "DOM"],
  "module": "esnext",
  "moduleResolution": "bundler",
  "allowImportingTsExtensions": true,
  "verbatimModuleSyntax": true,
  "moduleDetection": "force",
  "noEmit": true,
  "jsx": "react-jsx",
  "strict": true,
  "noUnusedLocals": true,
  "noUnusedParameters": true,
  "erasableSyntaxOnly": true,
  "noFallthroughCasesInSwitch": true,
  "include": ["src", "shared/src"]
}
moduleResolution: "bundler"
Matches how Vite resolves; lets you import .ts files without extension gymnastics.
verbatimModuleSyntax: true
Forces import type for type-only imports — a value-import of a type is a compile error. Keeps the emitted graph honest.
noUnusedLocals / noUnusedParameters
Dead code fails the type-gate, not just the linter.
noFallthroughCasesInSwitch
Pairs with the exhaustive-switch convention (page 04) so a forgotten break is caught.
erasableSyntaxOnly
Disallows TS-only runtime constructs (enums, namespaces, parameter properties) — everything must be plain erasable types, so the source stays bundler-portable.

The shared/ and server/ workspaces each have their own strict tsconfig.json with "types": ["node"] (no DOM) and noEmit. The server's sets verbatimModuleSyntax: false and esModuleInterop: true because it interops with CommonJS Node built-ins.

ESLint — flat config, type-aware on the app only

One flat config covers all three packages, layering rules by file glob. The crucial addition over recommended is enforcing the un-awaited-promise discipline on the app source.

eslint.config.js
import js from '@eslint/js'
import globals from 'globals'
import reactHooks from 'eslint-plugin-react-hooks'
import reactRefresh from 'eslint-plugin-react-refresh'
import tseslint from 'typescript-eslint'
import { defineConfig, globalIgnores } from 'eslint/config'

export default defineConfig([
  globalIgnores(['dist', 'coverage', 'playwright-report', 'test-results', '**/node_modules', '.claude/worktrees/**']),

  // Baseline for every TS file in every package (web, shared, server).
  { files: ['**/*.{ts,tsx}'], extends: [js.configs.recommended, tseslint.configs.recommended] },

  // The web app is the only React package — React/Fast-Refresh rules + browser globals here only.
  { files: ['src/**/*.{ts,tsx}'],
    extends: [reactHooks.configs.flat.recommended, reactRefresh.configs.vite],
    languageOptions: { globals: globals.browser } },

  // Node packages: Node globals.
  { files: ['server/**/*.ts', 'shared/**/*.ts'], languageOptions: { globals: globals.node } },

  // Type-aware linting for the app source only (it's the code in the web tsconfig project).
  { files: ['src/**/*.{ts,tsx}'],
    languageOptions: { parserOptions: { projectService: true, tsconfigRootDir: import.meta.dirname } },
    rules: {
      // Catch an un-awaited / un-voided promise instead of letting it float silently.
      '@typescript-eslint/no-floating-promises': 'error',
      '@typescript-eslint/no-misused-promises': 'error',
    } },
])
Load-bearing rule

no-floating-promises + no-misused-promises are the reason the codebase uses an explicit void for every intentional fire-and-forget async call (see page 05). A floating promise without void is a lint failure. Keep this rule on; it is what makes the void convention enforceable rather than aspirational.

Vite — bind loud, fail loud

The Vite config carries the shared alias, the embedded Vitest config, and two deliberately "fail-loud" server settings.

vite.config.ts (server block + alias)
resolve: {
  alias: { '@capacitylens/shared': fileURLToPath(new URL('./shared/src', import.meta.url)) },
},
// Bind IPv4 loopback explicitly. Node 17+ resolves `localhost` to ::1 (IPv6) first, so
// the default host would listen on ::1 only and a browser hitting 127.0.0.1 gets
// connection-refused (blank page, no console error). strictPort: a squatted 5173 (incl.
// by a sibling repo on the same port) FAILS at startup instead of silently using 5174.
server: { host: '127.0.0.1', port: 5173, strictPort: true },
Why these two settings matter to siblings specifically

floaty-schedule and delivery-diary claim the same port 5173. Without strictPort, starting a second one silently lands on 5174 while your browser stares at the wrong port's white page. With it, the second one exits with a clear error and you kill the squatter. Both the host pin and strictPort exist to convert a silent blank-page-with-empty-console into a loud, diagnosable failure — principle 12 in action.

The Vitest config is embedded in the same file (one config, no second file):

vite.config.ts (test block)
test: {
  environment: 'jsdom',
  setupFiles: ['./src/test/setup.ts'],
  css: true,
  include: ['src/**/*.{test,spec}.{ts,tsx}', 'shared/**/*.{test,spec}.ts'],
  exclude: ['e2e/**', 'node_modules/**', 'dist/**'],
  coverage: {
    provider: 'v8',
    reporter: ['text-summary', 'html'],
    include: ['src/**/*.{ts,tsx}', 'shared/src/**/*.ts'],
    exclude: ['**/*.test.{ts,tsx}', 'src/test/**', 'src/main.tsx', 'src/router.tsx', 'src/vite-env.d.ts'],
  },
}

The Playwright config defines three project flavours (plus an opt-in WebKit/Safari twin) and boots its own dev servers; it is covered in full on page 08.

index.html — CSP, theme guard, and the never-white placeholder

Three things live in the HTML shell, each deliberately. Keep all three when you copy it.

  1. A defence-in-depth CSP meta tag that is safe to ship in HTML (object-src 'none'; base-uri 'none'). The full script-src policy belongs in a host response header, not the app.
  2. A theme FOUC guard: an inline script that sets data-theme before first paint so dark-mode users never flash light. It mirrors readStoredTheme()/resolveTheme() in src/lib/theme.ts (default light).
  3. The static placeholder inside #root plus a <noscript> banner (principle 12).
index.html (theme guard)
<script>
  (function () {
    try {
      var p = localStorage.getItem('capacitylens/theme') || 'light'
      var dark = p === 'dark' || (p === 'system' && matchMedia('(prefers-color-scheme: dark)').matches)
      document.documentElement.dataset.theme = dark ? 'dark' : 'light'
    } catch (e) { /* storage/matchMedia unavailable — leave the default light palette */ }
  })()
</script>

The green gate

"Done" has a precise definition. Three commands, all green:

CommandRunsCovers
npm run gatetsc -b → eslint . → vitest run → vite buildThe app + the shared core.
npm run gate:servertsc --noEmit + vitest run in server/The optional server (separately — see why).
npm run e2eplaywright test (boots its own dev servers)The three Chromium projects end-to-end. Safari/WebKit is the opt-in e2e:webkit (page 08), not part of the merge bar.
Why the server is out of the root gate

The server imports node:sqlite, a Node 24 built-in unavailable in the jsdom environment Vitest runs in. Importing it (even transitively) would crash the root test run immediately. Keeping the server separate also means the root gate can run on any recent Node, while the server gate requires 24. These three commands are the merge bar, run locally before pushing; hosted CI is optional and off by default in these repos (see page 08).

vite build is part of the gate (not just tsc) because the production bundle catches import-resolution and bundler-time errors that the type-checker alone misses.

The stack & versions

React 19 + TypeScript ~6 + Vite 8. Zustand 5 for state, Tailwind v4 for styling, React Router 7 for navigation. Vitest 4 + Testing Library for unit/component, Playwright for E2E, @axe-core/playwright for the a11y oracle. Node 24+ is required (.nvmrc + engines) because the server uses node:sqlite unflagged — on Node 24 it needs no --experimental-sqlite.

Starting a sibling

Copy the three package.json files, the five tsconfigs, eslint.config.js, vite.config.ts, playwright.config.ts, index.html, and the dotfiles. Rename the packages and the @capacitylens scope to your own. You now have an empty app with the identical toolchain and an identical green gate — before writing a line of domain code.
