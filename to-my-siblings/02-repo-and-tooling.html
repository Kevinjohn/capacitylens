Directory tree

The repo is an npm-workspaces monorepo with three packages. Tests are co-located with the code they cover; only Playwright specs live apart in e2e/.

capacitylens-v1/

│   ├── auth/                 # AuthProvider, LoginScreen, PermissionProvider (server-driven; inert when auth is off)
│   ├── components/           # per-entity folders + common/ + ui/ (shadcn) + scheduler/

The three workspaces and why the split exists

WorkspacePackage nameRoleEnvironment
(root)capacitylensThe web app — React/Vite SPA.Browser
shared/@capacitylens/sharedThe pure domain core: types, validation, integrity, cascade, import-remap, migrate, seed, fixtures, access & lifecycle.Agnostic — no DOM lib, Node-agnostic by convention, no React
server/capacitylens-serverThe SQLite REST API — the default persistence backend.Node 24+
Why a shared workspace and not a folder import

Making the core a real package with its own name means the app and the server import it by the same specifier (@capacitylens/shared/...), the workspace linker guarantees they get the identical code, and the boundary is reinforced by config: the core's tsconfig omits the DOM lib ("lib": ["ES2023"]), so a browser API like document won't type-check. Node is a convention rather than a compiler wall — the tsconfig keeps "types": ["node"] (it shares the monorepo's Node toolchain), so staying off Node-specific APIs is a discipline we hold, not something the compiler forbids. The core leans on cross-platform globals only (e.g. crypto.randomUUID()). That is what makes "server validation is client validation" structurally true rather than merely intended.

Runtime dependencies: few, and each with a named job

The list grew when the UI adopted shadcn/ui (page 06), but the discipline survived in a different form: every runtime dependency has one named job, and nothing overlaps. The core four: react/react-dom, react-router-dom, zustand (state), date-fns (date math). The auth client: better-auth (inert when auth is off). The shadcn layer: radix-ui (headless primitives), lucide-react (icons, always aria-hidden), cmdk (the command palette), sonner (toasts — the hand-rolled Toast is retired), and the styling glue class-variance-authority / clsx / tailwind-merge. Still deliberately absent: a form library, a state-machine library, an ORM, a fetch wrapper. Resist additions that duplicate a job something on this list already does.

The full-stack dev orchestrator

npm run dev is no longer bare Vite. It runs scripts/dev-fullstack.mjs, which gives you the default (server-backed) app in one command:

  1. Pre-flight: checks ports 8787 (API) and 5173 (web) are free — fail loud, not lucky.
  2. Spawns the API: npm run dev -w capacitylens-server with PORT=8787, CAPACITYLENS_SEED_DEMO=1 (dev gets the demo dataset) and CAPACITYLENS_MULTI_ACCOUNT=1 (dev is uncapped).
  3. Spawns the web app: npm run dev:web with CAPACITYLENS_DEV_API_PORT=8787; Vite's dev /api proxy forwards to the API, so the browser sees a same-origin backend exactly like production.
  4. Dies together: either child exiting tears the whole stack down (each child runs in its own process group for a clean SIGTERM).

The backend-free path is npm run dev:demo — Vite only, VITE_CAPACITYLENS_DEMO=1, the localStorage demo build. dev:api/dev:auth are the E2E flavours on their own ports (page 08). Full-stack dev needs Node 24 (the server's node:sqlite).

Do this in a sibling

Copy scripts/dev-fullstack.mjs as-is. The one-command full stack is what keeps the server-backed default honest in daily work — if devs have to boot two terminals, they drift back to the demo build and stop exercising the real persistence path.

Install at the root, always

Run npm install from the repo root, never inside a workspace — the root install is what links @capacitylens/shared into both the app and the server. The server runs its TypeScript directly via tsx (no emit); the app is transpiled by Vite.

The Paraglide compile step (why every script starts the same way)

Strings are compiled, not looked up (principle 13). messages/en.json + project.inlang/ are the committed source; paraglide:compile generates typed message functions into src/paraglide/, which is gitignored. Because the generated code must exist before tsc, Vite, or Vitest can run, the compile is prefixed into build, test, gate, coverage and every dev:* script. The Vite plugin (paraglideVitePlugin) keeps it hot in dev.

Don't

Don't commit src/paraglide/, don't edit it, and don't import it directly — components import m from the single src/i18n seam (page 06). And don't strip the paraglide:compile && prefixes to "simplify" the scripts: a fresh clone's npm run gate fails without them.

TypeScript configuration

The root uses project references (tsc -b) so a single build command type-checks the app and its build-tooling config as separate projects with separate settings.

tsconfig.json (the solution file)
{
  "files": [],
  "references": [
    { "path": "./tsconfig.app.json" },
    { "path": "./tsconfig.node.json" }
  ]
}

tsconfig.app.json covers src and shared/src with the DOM lib; tsconfig.node.json covers vite.config.ts (plus the privacy-posture test, which runs in Node) with the Node types. Both are strict. The settings worth copying exactly:

tsconfig.app.json (compilerOptions)
{
  "target": "es2023",
  "lib": ["ES2023", "DOM"],
  "types": ["vite/client"],
  "module": "esnext",
  "moduleResolution": "bundler",
  "allowImportingTsExtensions": true,
  "verbatimModuleSyntax": true,
  "moduleDetection": "force",
  "noEmit": true,
  "jsx": "react-jsx",
  "strict": true,
  "skipLibCheck": true,
  "allowJs": true,
  "noUnusedLocals": true,
  "noUnusedParameters": true,
  "erasableSyntaxOnly": true,
  "noFallthroughCasesInSwitch": true,
  "paths": { "@/*": ["./src/*"] },
  "include": ["src", "shared/src"]
}
moduleResolution: "bundler"
Matches how Vite resolves; lets you import .ts files without extension gymnastics.
verbatimModuleSyntax: true
Forces import type for type-only imports — a value-import of a type is a compile error. Keeps the emitted graph honest.
allowJs: true
Paraglide generates .js with JSDoc types; this lets tsc read them, which is what makes a missing message key a compile error.
paths: { "@/*": ["./src/*"] }
The shadcn convention — mirrored by a Vite alias and components.json. Use it for app-internal imports; shared-core imports keep the package specifier.
noUnusedLocals / noUnusedParameters
Dead code fails the type-gate, not just the linter.
noFallthroughCasesInSwitch
Pairs with the exhaustive-switch convention (page 04) so a forgotten break is caught.
erasableSyntaxOnly
Disallows TS-only runtime constructs (enums, namespaces, parameter properties) — everything must be plain erasable types, so the source stays bundler-portable.

The shared/ and server/ workspaces each have their own strict tsconfig.json with "types": ["node"] (no DOM) and noEmit. The server's sets verbatimModuleSyntax: false and esModuleInterop: true because it interops with CommonJS Node built-ins.

ESLint — flat config, type-aware on the app only

One flat config covers all three packages, layering rules by file glob. The crucial addition over recommended is enforcing the un-awaited-promise discipline on the app source.

eslint.config.js
import js from '@eslint/js'
import globals from 'globals'
import reactHooks from 'eslint-plugin-react-hooks'
import reactRefresh from 'eslint-plugin-react-refresh'
import tseslint from 'typescript-eslint'
import { defineConfig, globalIgnores } from 'eslint/config'

export default defineConfig([
  globalIgnores(['dist', 'coverage', 'playwright-report', 'test-results', '**/node_modules', '.claude/worktrees/**', 'src/paraglide']),

  // Baseline for every TS file in every package (web, shared, server).
  { files: ['**/*.{ts,tsx}'], extends: [js.configs.recommended, tseslint.configs.recommended] },

  // The web app is the only React package — React/Fast-Refresh rules + browser globals here only.
  { files: ['src/**/*.{ts,tsx}'],
    extends: [reactHooks.configs.flat.recommended, reactRefresh.configs.vite],
    languageOptions: { globals: globals.browser } },

  // Node packages: Node globals.
  { files: ['server/**/*.ts', 'shared/**/*.ts'], languageOptions: { globals: globals.node } },

  // Type-aware linting for the app source only (it's the code in the web tsconfig project).
  { files: ['src/**/*.{ts,tsx}'],
    languageOptions: { parserOptions: { projectService: true, tsconfigRootDir: import.meta.dirname } },
    rules: {
      // Catch an un-awaited / un-voided promise instead of letting it float silently.
      '@typescript-eslint/no-floating-promises': 'error',
      '@typescript-eslint/no-misused-promises': 'error',
    } },

  // shadcn primitives export variants alongside components — exempt them from the Fast-Refresh rule.
  { files: ['src/components/ui/**'], rules: { 'react-refresh/only-export-components': 'off' } },
])
Load-bearing rule

no-floating-promises + no-misused-promises are the reason the codebase uses an explicit void for every intentional fire-and-forget async call (see page 05). A floating promise without void is a lint failure. Keep this rule on; it is what makes the void convention enforceable rather than aspirational.

Vite — bind loud, fail loud

The Vite config carries the two aliases, the dev /api proxy, the Tailwind and Paraglide plugins, the embedded Vitest config, and two deliberately "fail-loud" server settings.

vite.config.ts (plugins, alias + server block)
plugins: [react(), tailwindcss(), paraglideVitePlugin({ project: './project.inlang', outdir: './src/paraglide' })],
resolve: {
  alias: {
    '@capacitylens/shared': fileURLToPath(new URL('./shared/src', import.meta.url)),
    '@': fileURLToPath(new URL('./src', import.meta.url)),   // the shadcn `@/*` convention
  },
},
// Bind IPv4 loopback explicitly. Node 17+ resolves `localhost` to ::1 (IPv6) first, so
// the default host would listen on ::1 only and a browser hitting 127.0.0.1 gets
// connection-refused (blank page, no console error). strictPort: a squatted 5173 (incl.
// by a sibling repo on the same port) FAILS at startup instead of silently using 5174.
server: {
  host: '127.0.0.1', port: 5173, strictPort: true,
  // Same-origin /api in dev, exactly like the production nginx proxy.
  proxy: { '/api': { target: `http://localhost:${process.env.CAPACITYLENS_DEV_API_PORT ?? 8787}`, changeOrigin: true } },
},
Why these two settings matter to siblings specifically

floaty-schedule and delivery-diary claim the same port 5173. Without strictPort, starting a second one silently lands on 5174 while your browser stares at the wrong port's white page. With it, the second one exits with a clear error and you kill the squatter. Both the host pin and strictPort exist to convert a silent blank-page-with-empty-console into a loud, diagnosable failure — principle 16 in action.

index.html — CSP, theme guard, and the never-white placeholder

Three things live in the HTML shell, each deliberately. Keep all three when you copy it.

  1. A defence-in-depth CSP meta tag that is safe to ship in HTML (object-src 'none'; base-uri 'none'). The full script-src policy belongs in a host response header, not the app (the API server ships its own full CSP via helmet — page 07).
  2. A theme FOUC guard: an inline script that sets data-theme before first paint so dark-mode users never flash light. It mirrors readStoredTheme()/resolveTheme() in src/lib/theme.ts (default light).
  3. The static placeholder inside #root plus a <noscript> banner (principle 16). The placeholder's wordmark colour is AA-checked — even the pre-hydration splash is inside the accessibility standard.
index.html (theme guard)
<script>
  (function () {
    try {
      // The legacy-key fallback covers devices that stored a theme before the rebrand.
      var p = localStorage.getItem('capacitylens/theme') || localStorage.getItem('floaty/theme') || 'light'
      var dark = p === 'dark' || (p === 'system' && matchMedia('(prefers-color-scheme: dark)').matches)
      document.documentElement.dataset.theme = dark ? 'dark' : 'light'
    } catch (e) { /* storage/matchMedia unavailable — leave the default light palette */ }
  })()
</script>

The green gate

"Done" has a precise definition. Three commands, all green:

CommandRunsCovers
npm run gateparaglide:compile → tsc -b → eslint . → vitest run → vite buildThe app + the shared core.
npm run gate:servertsc --noEmit + vitest run in server/The server (separately — see why).
npm run e2eplaywright test (boots its own dev servers)The three Chromium projects end-to-end. WebKit and Firefox are the opt-in e2e:webkit / e2e:firefox / e2e:browsers (page 08), not part of the merge bar.
Why the server is out of the root gate

The server imports node:sqlite, a Node 24 built-in unavailable in the jsdom environment Vitest runs in. Importing it (even transitively) would crash the root test run immediately. Keeping the server separate also means the root gate can run on any recent Node, while the server gate requires 24. These three commands are the merge bar, run locally before pushing; there is deliberately no committed GitHub Actions workflow — the local gate is the enforcement mechanism (see page 09).

vite build is part of the gate (not just tsc) because the production bundle catches import-resolution and bundler-time errors that the type-checker alone misses.

The stack & versions

React 19 + TypeScript ~6 + Vite 8. Zustand 5 for state, Tailwind v4 for styling, React Router 7 for navigation, shadcn/ui (radix-ui + cva + lucide) for primitives, Sonner for toasts, Paraglide (inlang) for compile-time i18n, Better Auth 1.6 for sessions. Vitest 4 + Testing Library for unit/component, Playwright for E2E, @axe-core/playwright for the a11y oracle. The server: Fastify 5 + @fastify/helmet + @fastify/rate-limit on node:sqlite. Node 24+ is required (.nvmrc + engines on all three manifests) because the server uses node:sqlite unflagged — on Node 24 it needs no --experimental-sqlite.

Starting a sibling

Copy the three package.json files, the five tsconfigs, eslint.config.js, vite.config.ts, playwright.config.ts, index.html, components.json, .env.example, project.inlang/ + messages/en.json, the scripts/ folder, and the dotfiles. Rename the packages and the @capacitylens scope to your own. You now have an empty app with the identical toolchain and an identical green gate — before writing a line of domain code.
