Deployment & ops

Bare metal (nginx + systemd) as the primary deploy with one reproducible two-service Docker image as the packaged alternative, a boot-time posture interlock that makes unsafe production configurations a decision rather than an accident, a flag register with named exceptions, and backups whose restore path is continuously tested. The discipline is "ship the small thing safely, and never build ahead of need".

The one structural fact that frames everything

The client env is inlined at build time, and an empty env means same-origin server mode — there is no localStorage fallback. A server build whose backend is missing boots into a "can't reach the server" screen, on purpose (a silent fallback would read as data loss when the server returned). The only way to get a browser-local build is the explicit demo flag. So: the server is a hard dependency of the default build, and which build you have is always knowable from the Settings build stamp (below).

The Docker Compose deploy (the packaged alternative)

One multi-stage Dockerfile (Node 24 for the build and the api target; nginx:alpine for web), one docker-compose.yml, one nginx.conf:

Browser ──HTTP(S)──> web (nginx, :${WEB_PORT:-8080})
                       ├── /          → the built SPA (empty env ⇒ same-origin /api)
                       └── /api/*     → api:8787   (same-origin, so CORS stays fail-closed)
                     api (Fastify + node:sqlite, CAPACITYLENS_HOST=0.0.0.0, NODE_ENV=production)
                       ├── /data/capacitylens.db        (named volume capacitylens-db)
                       └── /backups/…                   (named volume capacitylens-backups)
                       HEALTHCHECK → /api/health
  • The server stays API-only — nginx serves the SPA; the Fastify app never gains a static-file concern (no @fastify/static).
  • Same-origin by construction: the SPA is built with an empty VITE_CAPACITYLENS_API, and nginx proxies /api — so CAPACITYLENS_CORS_ORIGIN stays unset and CORS stays fail-closed.
  • SPA fallback + cache split in nginx.conf: try_files … /index.html for deep links; hashed /assets/ immutable for a year; index.html no-cache so a deploy is picked up immediately.
  • docker compose up --build -d, browse http://localhost:8080. That's the quick start.
The quick start must actually boot — test your own docs

The documented no-auth quick start once crash-looped: the api image sets NODE_ENV=production, and the posture interlock (below) correctly refused to boot with auth off. The fix was to make the documented default honest — the compose file carries the explicit CAPACITYLENS_ALLOW_OPEN_IN_PRODUCTION=1 for the trusted-LAN quick start, with a comment telling you to remove it when you turn auth on. The lesson for a sibling: run your own quick start from scratch; a deploy doc that has never been executed is fiction.

First run: empty, one company

A fresh real instance starts empty — no demo data — at the create-your-company screen (auto-seed is demo/dev-only: CAPACITYLENS_SEED_DEMO=1). And by default the instance serves one company: the server refuses to create a second account (403 naming the fix) unless CAPACITYLENS_MULTI_ACCOUNT=1; the picker hides "New company" at the cap. The multi-tenant schema is intact underneath — the cap is a deployment shape, not a migration (page 07).

The production posture interlock

At boot, a pure evaluateProductionPosture(env) is consulted. It is a no-op outside NODE_ENV=production — dev, E2E and trusted-LAN self-hosting keep the open auth-off posture as a first-class mode. In production it:

  • Refuses to start with auth off (the dataset would be world-readable and writable). The only escape is the explicit CAPACITYLENS_ALLOW_OPEN_IN_PRODUCTION=1, which downgrades the refusal to a loud warning — never silences it.
  • Warns loudly (still boots) on HSTS-off (CAPACITYLENS_HTTPS!=1 — expected when TLS terminates at a proxy) and on open self-signup.
Why an interlock and not a flag

This is one of the named exceptions to "everything ships behind a flag, default off" (principle 12): defaulting a safety guard off defeats it. Like the reset boot-guard, it is deliberately always-on; the escape hatch exists, is explicit, and shows up in the logs. An unsafe production posture is thereby always a decision with a paper trail, never an accident of an unset variable.

Bare-metal / Forge notes (the same pieces, by hand)

  1. Persistent DB path outside the deploy tree so git pull never touches it (CAPACITYLENS_DB=/home/forge/capacitylens-data/capacitylens.db).
  2. Run the server as a managed daemon (npm start --workspace=server); add a daemon restart to the deploy script so new code actually runs.
  3. Reverse-proxy /api → 127.0.0.1:8787. Same-origin, so CORS stays moot and fail-closed (leave CAPACITYLENS_CORS_ORIGIN unset — the localhost defaults never match a real origin).
  4. Build with an empty VITE_CAPACITYLENS_API for same-origin, or set it to the API's origin (never /api — the client composes ${API_BASE}/api/…, so /api would double-prefix).
  5. HTTPS via Let's Encrypt; then CAPACITYLENS_HTTPS=1 so HSTS and the Secure session cookie switch on together.
  6. Host-level security headers for the SPA (the API ships its own via helmet — page 07). Ship a full CSP as Report-Only first, click through the whole app, then enforce; style-src 'unsafe-inline' is required because the scheduler positions bars with inline styles.
  7. Cache headers: hashed /assets/ immutable; index.html no-cache.
  8. npm ci --include=dev in the deploy script — NODE_ENV=production would otherwise skip vite/tsc and the build dies with "vite: not found".

The static demo deploy (explicit opt-in)

The old default, retained as a preview vehicle: build with VITE_CAPACITYLENS_DEMO=1, ship dist/ to any static host with the SPA fallback rewrite, done. No backend, no login, self-seeding, every visitor an island. Do not use it for anything shared — and never assume a plain npm run build produces it (a plain build is a server build now).

Client build flags & the build stamp

Client flags are build-time Vite env (VITE_*), inlined into the bundle: VITE_CAPACITYLENS_API (empty = same-origin server; an origin to retarget), VITE_CAPACITYLENS_DEMO (the localStorage demo build — wins over the API var), VITE_CAPACITYLENS_BUILD_SHA, VITE_CAPACITYLENS_FEEDBACK_MAILTO. The build stamp is a deliberate safety device: which-build-is-this is invisible from the UI otherwise, so Settings shows a one-line footer — build <sha> · server or build <sha> · local — and the post-deploy smoke test asserts it says server (a stray demo flag in a deploy pipeline would otherwise ship a no-backend build that looks identical).

deploy script (server mode, the exports)
# VITE_CAPACITYLENS_API stays EMPTY for a same-origin deploy (the default);
# set it only to point at a DIFFERENT origin — and never to "/api".
export VITE_CAPACITYLENS_BUILD_SHA="$(git rev-parse --short HEAD)"  # the tell-tale stamp
export VITE_CAPACITYLENS_FEEDBACK_MAILTO="<owner address>"
npm run build

Backups, restore, and demo reset

Backups are a server feature (flag CAPACITYLENS_BACKUP_DIR), not a host cron — because WAL mode means a raw cp can catch a torn state. The daemon writes WAL-safe online backup() snapshots (capacitylens-<stamp>.db) once at boot then every CAPACITYLENS_BACKUP_INTERVAL_MIN (60), keeping the newest CAPACITYLENS_BACKUP_KEEP (48) — RPO ≤ 1 h.

Load-bearing rule — a backup that's never been restored is a hope

The restore path is continuously verified in-repo: restore.drill.test.ts runs the full cycle on every gate:server — snapshot → make a post-snapshot edit → corrupt the live DB → restore (copy the snapshot over it + remove the -wal/-shm sidecars, which would replay stale frames) → verify the seed row is back and the post-snapshot edit is gone (the point-in-time loss is real, proving the file was genuinely replaced). The manual sequence lives in docs/runbook.md; the on-host drill before real users remains the operator's step. Snapshot before each demo session; restoring is the reset-to-clean button (which is why the destructive reset route stays forbidden in production).

Monitoring, the audit log & rollback

  • Uptime is purely operational: an external monitor hits /api/health (with deep health on, a 200 proves the DB answers and carries the audit soft-signal audit:'ok'|'degraded'; a 503 means the DB is broken while the process lives). Plus a disk alert sized against backup retention.
  • The audit log (on by default) is the accountability surface: one JSONL line per mutation, field-names-only, rotated at 64 MB, fail-never (a broken sink degrades health-softly and sets a response header, never fails a request). Where it lives: CAPACITYLENS_AUDIT_FILE, default beside the DB — include it in what you back up.
  • Rehearsal before cutover: run the db-backed E2E specs against a real production build behind a local /api proxy (scripts/serve-dist.mjs) with the host's flags ON — proving ON-state behaviour before it faces users.
  • Rollback is a data decision. Export before any rollback: curl … /api/state?accountId=<id>&includeInactive=1 > export-$(date +%F).json (the complete per-tenant backup — admin-gated when auth is on). Rolling back to a good state within server mode is the restore sequence above. And note the old escape hatch is gone: rebuilding without the API var does not return the app to localStorage — it is still a server build.

The staged plan — triggers, not a schedule (updated for what shipped)

Why "triggers, not schedules"

Each stage carries real cost (a conflict UI, a security review, a rewrite). Building them before the need exists is speculative complexity in a deliberately-small product. The plan names the exact condition that makes each stage worth its cost, so the team can defer with confidence rather than gold-plating. Note how the shipped stages validated the pattern: the seams (adapter, auth port, tenant store) were built early and cheap, which is exactly what made each later stage a bounded change.

Security posture, stated honestly

  • Two postures, both deliberate. Auth-off = the trusted-local self-host default: an open, shared dataset (fine on a LAN or behind your own gate; the production interlock stops it being an accident on a public host). Auth-on = a real boundary: every /api/* request needs a session, reads are membership-gated per account, the whole-dataset read is closed, roles gate writes, sensitive fields are server-redacted.
  • The honest remainder: on the write path, accountId is still client-asserted; ownsRow plus the membership 403s are defence-in-depth. Deriving it from the session is the named final step of Stage C — until then, don't advertise hostile-multi-tenant isolation.
  • Headers: the API ships its own full CSP + security headers via helmet (page 07); the SPA's full policy belongs in the host/nginx config; the app HTML carries the safe object-src/base-uri baseline.
  • Privacy is test-enforced (principle 14): no-egress is proven by the dependency denylist test and the connect-src 'self' no-widening assertion; the statement is docs/privacy.md.
Don't

Do not expose real, isolated user data on the auth-off model, and do not treat auth-on as hostile-grade isolation while the write path stays client-asserted. The line moves when session-derived accountId lands with isolation tests and a security review — that gate is unchanged.
