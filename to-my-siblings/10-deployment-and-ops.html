Deployment & ops

Two deploy modes, a flag register that keeps every behaviour change opt-in, real backup and restore drills, and a staged migration plan where each later stage is gated on a product trigger — not a schedule. The discipline here is "ship the small thing safely, and never build ahead of need".

Two deploy modes

ModeWhat shipsDataWhen
Static SPA (default)The Vite build (dist/) on a static host. No backend.Each browser's own localStorage — per-browser islands.Solo use, demos, the simplest possible deploy.
Server modeThe build (with VITE_FLOATY_API baked in) + the Fastify daemon behind an /api proxy + persistent SQLite.One shared, server-persisted dataset, last-writer-wins.A shared dataset across people/devices.
The one structural fact that frames everything

VITE_FLOATY_API is inlined at build time with no localStorage fallback. A build either targets the server or it doesn't. So in server mode the server is a hard dependency (server down ⇒ app down), and rollback = rebuild without the flag, which strands whatever is on the server. Rollback is a data decision, not just a redeploy (see below).

Static SPA deploy (the default)

The reference target is Laravel Forge + DigitalOcean, but any static host works. The essentials:

  • Site type: Static HTML; web directory: /dist (the Vite output, not /public).
  • Deploy script runs the build on the host:
deploy script (static)
git pull origin $FORGE_SITE_BRANCH
# --include=dev guards against NODE_ENV=production skipping vite/tsc/tailwind
# ("vite: not found"). Prepend `export NODE_ENV=development` if the build still fails.
npm ci --include=dev
npm run build   # tsc -b && vite build  ->  dist/
Load-bearing rule — the SPA fallback

The app uses real URLs (createBrowserRouter), so the host must rewrite unknown paths to index.html or refreshing a deep link like /projects 404s: location / { try_files $uri $uri/ /index.html; }. Then enable HTTPS (Let's Encrypt). Each visitor's data lives in their own browser — there is no shared dataset in this mode.

Server cutover (the controlled-demo architecture)

Moving to a shared dataset is a cutover plus hardening, not a build — the server and the sync adapter already exist. The target architecture:

Browser ──HTTPS──> Nginx (Let's Encrypt, Basic Auth, security headers)
                     ├── /          → dist/  (Vite build, VITE_FLOATY_API baked in)
                     └── /api/*      → 127.0.0.1:8787  (Fastify daemon, node:sqlite)
                                         └── /home/.../floaty.db (WAL) + online backups

The cutover steps (operations, mostly no code):

  1. Persistent DB path outside the deploy tree so git pull never touches it (FLOATY_DB=/home/forge/floaty-data/floaty.db).
  2. Run the server as a managed daemon (npm start --workspace=server); add a daemon restart to the deploy script so new code actually runs.
  3. Reverse-proxy /api → 127.0.0.1:8787. Same-origin, so CORS stays moot and fail-closed (leave FLOATY_CORS_ORIGIN unset — the localhost defaults never match a real origin).
  4. Flip the build onto the server: export VITE_FLOATY_API=https://<site> before npm run build.
  5. Gate it with Nginx Basic Auth over the whole site — one htpasswd entry per user (attribution in access logs, per-person revocation). The API is destructive and unauthenticated; it must not be naked on the internet.
  6. Security headers in the host config, not the app (the app ships only the safe object-src/base-uri CSP). Ship the full CSP as Report-Only first, click through the whole app, then enforce. Note style-src 'unsafe-inline' is required because the scheduler positions bars with inline styles.
  7. Cache headers: hashed /assets/ immutable for a year; index.html no-cache — or testers get stuck on a stale build after a deploy.

The flag register — every behaviour change, opt-in

This is principle 10 made operational. One table is the single source of truth for every flag, its default (unset = today's behaviour), and what the production host actually sets.

The three deliberately un-flagged changes

A change is not flagged when an "off" default would preserve a defect: the Node-version pin (an environment prerequisite, not behaviour), graceful shutdown (only changes termination — today it can kill a request mid-transaction), and the reset boot-guard (a safety interlock on an existing flag; defaulting a guard off defeats it). Document the exceptions with rationale, as the register does.

Client build flags & the build stamp

Client flags are build-time Vite env (VITE_*), inlined into the bundle. The build stamp is a deliberate safety device: a build that silently lost VITE_FLOATY_API reverts to localStorage and looks identical otherwise. So Settings shows a one-line footer — build <sha> · server or build <sha> · local — and the post-deploy smoke test asserts it says server.

deploy script (server mode, the exports)
export VITE_FLOATY_API="https://<site>"                       # server mode — REQUIRED
export VITE_FLOATY_BUILD_SHA="$(git rev-parse --short HEAD)"  # the tell-tale stamp
export VITE_FLOATY_FEEDBACK_MAILTO="<owner address>"
npm run build

Backups, restore, and demo reset

Backups are a server feature (flag FLOATY_BACKUP_DIR), not a host cron — because WAL mode means a raw cp can catch a torn state. The daemon writes online backup() snapshots once at boot then hourly, keeping the newest 48 (RPO ≤ 1 h).

Load-bearing rule — a backup that's never been restored is a hope

Run the restore drill once before real use: stop the daemon, copy a snapshot over the live DB, remove the WAL sidecars (floaty.db-wal/-shm — stale ones would replay old frames over the restored file), restart, verify in the app. Document the exact command sequence in a one-page runbook.md. Snapshot before each demo session; restoring it is the reset-to-clean button (which is why the destructive reset route stays forbidden in production).

Monitoring: an uptime check on /api/health (with deep health on, a 200 proves the DB answers; a 503 means the DB is broken while the process lives) plus a disk alert sized against backup retention. A production-shaped rehearsal runs the db-backed E2E specs against a real production build served behind a local /api proxy (scripts/serve-dist.mjs) with the droplet's flags ON — proving the ON-state behaviour before the cutover.

Rollback is a data decision

Export before you roll back

Rebuilding without VITE_FLOATY_API returns the app to browser-local storage but strands the server data (nothing migrates back). Before flipping, export it: curl -su <user> https://<site>/api/state > export-$(date +%F).json. Rolling back to a good state without leaving server mode is the restore sequence above.

The staged migration plan — triggers, not a schedule

Beyond the shared demo, further stages are conditional. Each is gated on a product trigger; you do not build ahead of the trigger.

StageWhatTrigger
0Static localStorage deploy.Today's default.
Near-termShared dataset on the DB, behind Basic Auth (the cutover above).People need one shared, durable dataset.
BOptimistic concurrency — but build the client 409/conflict UI first, then turn on FLOATY_OPTIMISTIC_CONCURRENCY. Move off any experimental driver. Tested restore.The shared dataset starts mattering / people co-edit.
C the big oneReal auth + per-account isolation: app login, derive accountId from the session (never the client), server-enforce ownership on every read/write, isolation tests + a security review as the gate.Real users in separate trust domains who must not see each other's data.
DProductised "push my localStorage to the server" import flow.Real users with existing local data to carry over.
EPostgres. Note: no ORM seam, so this is a data-layer rewrite, not a dialect swap.SQLite concurrency/scale/managed-backup needs justify the ops cost.
Why "triggers, not schedules"

Each stage carries real cost (a conflict UI, a security review, a rewrite). Building them before the need exists is speculative complexity in a deliberately-small product. The plan names the exact condition that makes each stage worth its cost, so the team can defer with confidence rather than gold-plating.

Security posture, stated honestly

  • No app-level auth by default. The demo's gate is Nginx Basic Auth; the dataset is shared, last-writer-wins.
  • Auth is wired but OFF (FLOATY_AUTH, Better Auth): off = byte-for-byte today, no auth tables, no login UI. The server-reported authMode is the only auth flag — there is no client-side auth flag.
  • Session ≠ isolation. Turning auth on gates requests, but accountId stays client-asserted (ownsRow is defence-in-depth) until Stage C derives it from the session.
  • CSP: object-src/base-uri ship in the app; the full policy belongs in a host header.
Don't

Do not expose real, isolated user data on the shared/no-auth model. Until Stage C lands, the Basic Auth gate is the only thing between the destructive API and the internet — anyone with the URL + credentials can edit or wipe the shared dataset. That line moves only when Stage C is built and security-reviewed.
