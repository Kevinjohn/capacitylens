Process & workflow

How knowledge is captured and how change is governed. This is the part that lets a small team (and an AI pair) keep moving fast without the decisions evaporating — and it is designed so that keeping the docs current costs almost nothing.

The docs map — split by access pattern

Documentation is split not by topic but by how it is read and written, so each file stays cheap to keep current. Know which file a fact belongs in.

DECISIONS.mdA slim, present-tense digest of standing calls that still constrain the code.Read whole (it's short). Edit a line only when a load-bearing call changes.

Why the digest/log split is the whole trick

A single growing decisions file becomes either too long to load or too lossy to trust. Splitting it into a short digest you always read and a long log you only append to means: the digest never costs much to load, the log never costs much to write, and the source code is always the final authority with the digest as its index. This is the cheapest durable-memory pattern the repo has, and the single most valuable process habit to carry into a sibling.

CLAUDE.md — the one file an agent must internalise

Every repo carries a short CLAUDE.md that an AI pair (or a new human) reads first. It is deliberately tiny — "keep this file short; it loads every session" — and contains: the one-sentence product statement and non-goals; the architecture in one breath; the load-bearing invariants ("don't break"); the docs map; how to log a decision; and the definition of the green gate.

Do this in a sibling

Write the sibling's CLAUDE.md first, before any feature work, by adapting Floaty's. The sections are the same; the contents change (the product statement, the entity names, the invariants specific to the new domain). It is the contract the rest of the repo is built to honour.

Logging a decision (keep it cheap)

The procedure is three steps, optimised so the common case is one appended line:

  1. Append one line to docs/decisions-log.md — - 2026-06-13 — <area> — <what changed> (<sha>). Full rationale only for load-bearing calls.
  2. Append by reading the file tail (open it near EOF), never the whole file — that locality is the entire point of the split.
  3. If the call constrains future work, promote it to DECISIONS.md as one line. When a promoted call later changes, edit that line so the digest never drifts from the code.
Load-bearing rule

Two directions, two files. New facts flow down into the append-only log; standing constraints get promoted up into the digest and edited in place when they change. Never let the digest accumulate dated history, and never let the log become something you have to read whole.

Open questions are flagged, not silently resolved

NEEDS-INPUT.md holds genuine product questions for the owner, grouped by status (parked, resolved, genuinely open). The discipline: when you hit a question that is the owner's to answer — a scope call, a UX trade-off, a posture decision — you flag it here rather than guessing and baking the guess into code. Resolved questions are marked with the date and who decided, and load-bearing resolutions are promoted to DECISIONS.md.

Why this matters for an AI pair specifically

An autonomous agent will happily resolve an ambiguous product question by inventing a reasonable-looking answer. That is exactly the failure this file prevents: ambiguity about the product is escalated, not resolved unilaterally. Ambiguity about the code (which the agent can verify) is resolved by reading the source. Knowing which is which is a core competence here.

The user-story system

Every capability has a runnable manual story under user-stories/<area>/. The shape is fixed by a template: goal (one sentence) → why (the value) → how (numbered, end-to-end steps from a defined precondition) → acceptance criteria (checkable ✅ assertions, including edge/negative cases). Each story names its linked E2E test (or the unit test/manual check when no dedicated spec exists), so manual scripts and the automated suite stay in lock-step.

user-stories/TEMPLATE.md (the shape)
# US-XXX-NN — <short title>
**Area:** <area> · **Persona:** <who> · **Linked E2E:** `e2e/<file>.spec.ts` → "<test title>"

## Goal     <one sentence — the outcome the user wants>
## Why      <the value/motivation>
## How (end-to-end)
**Precondition:** <starting state>
1. <action>  2. <action>  3. <observed result>
## Acceptance criteria
- ✅ <a concrete, checkable assertion>
- ✅ <edge/negative case where relevant>

And the single source of truth they all depend on:

Load-bearing rule — REFERENCE-first

user-stories/REFERENCE.md pins the exact current facts every story and spec depend on: routes, control labels (accessible names), data-testids, the first-run seed data, and the domain rules a tester must know. When the app changes, update REFERENCE.md first, then the affected stories, then the specs. This ordering is what stops a relabelled button from quietly breaking the suite.

The modularity rule — when you may refactor

The codebase is opinionated about when structural change is allowed, because a refactor of a hot, under-tested module is how regressions slip in:

  • Pure extractions land freely. Moving a pure function to its own file, lifting a constant, splitting out a tested helper — the green gate proves these safe, so they need no special ceremony.
  • High-churn structural splits wait for a characterisation test. Slicing the store by concern, extracting a viewport hook, splitting the grid's render — these touch hot, stateful code. The rule: write a characterisation test that pins the current behaviour first, then do the split, and watch the test stay green.
Do this

Before a risky refactor, ask: "does the gate already prove this safe?" If yes, go. If no, write the test that makes it provable, then go. Never refactor a hot module and rely on eyeballing the diff.

Git & the bar for merging

  • Branch off main for work; commit or push only when asked. Don't commit straight to the default branch.
  • The merge bar is the full green gate: npm run gate + npm run gate:server + npm run e2e, plus any user-visible change reflected in REFERENCE.md and the stories. Run it locally before pushing — hosted CI is optional and off by default (see page 08).
  • Commit messages are conventional-ish and scoped (feat(client): …, fix(scheduler): …, docs: …) — see the recent history for the house style.
  • One decisions-log line per landed change, with its commit ref.

Adopting the process in a sibling

The process files are as copyable as the code. To stand up a sibling's process layer:

  1. Copy CLAUDE.md, DECISIONS.md, NEEDS-INPUT.md, and the docs/ + user-stories/ skeletons.
  2. Rewrite the product statement and non-goals for the new domain (page 01's rule).
  3. Rewrite user-stories/REFERENCE.md for the new routes/labels/seed.
  4. Start the decisions-log fresh; promote the standing architectural calls (which you inherited from this handbook) into the sibling's DECISIONS.md so they are explicit there too.
