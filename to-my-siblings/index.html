The Floaty house style

To my siblings

This is the complete architectural, coding, and process handbook for Floaty — the original repo. It exists so its sibling projects (floaty-schedule, a shift rota; delivery-diary, a deliverables tracker; and any future relatives) can adopt the same conventions exactly, down to the small decisions. If you are a developer or an AI agent standardising a sibling repo: read this front to back, then keep it open as a reference.

What these repos share

Floaty and its siblings are all the same machine re-skinned for a new domain: a local-first, browser-based scheduler with a pure domain core, an optional SQLite server behind one seam, multi-tenancy by account, and a relentless testing gate. The domain changes (people×allocations vs. shifts vs. deliverables); the architecture, conventions, and process do not. This handbook is the part that does not change.

How to use this handbook

Each page is self-contained but they build on each other. Pages 01–02 orient you. Pages 03–07 are the build: architecture down through every layer. Pages 08–10 are how we keep it correct and ship it. Page 11 is the cheat-sheet you will keep returning to.

Throughout, four kinds of callout flag what matters most:

Load-bearing rule

A decision the code depends on. Copy it verbatim into the sibling. Breaking it breaks an invariant.

Why

The reasoning behind a decision. When you are tempted to do it differently, read this first.

Do this

The positive pattern to follow.

Don't

An anti-pattern that was considered and rejected, or a footgun that has bitten this codebase before.

The ten ideas everything else follows from

If you internalise nothing else, internalise these. Every page is an elaboration of one of them.

  1. Be deliberately small. One product solves one problem well. Non-goals are written down and defended. New scope is rejected by default.
  2. Local-first, server-optional, one seam. The app runs with zero backend on localStorage; an optional server drops in behind a single PersistenceAdapter interface and nothing else changes.
  3. One source of truth, shared so it can't drift. A pure domain core (@floaty/shared) is imported by both client and server. Server validation is client validation — the same code, not a re-implementation.
  4. Forms reject; import and server repair. The same rule has two enforcement modes: a person fixes their mistake at a form; a machine import or a direct API write strips/repairs/drops instead.
  5. Extending an entity is drift-proofed. Parallel lists are exhaustiveness-checked against the types, so a new field that misses a list fails the build instead of silently dropping.
  6. Multi-tenant by account, through one seam. Every entity carries an accountId; the active account is chosen on load and never persisted; all scoped reads go through one hook.
  7. Device-global vs. account data are separate. Theme, display toggles, sidebar state live in their own localStorage keys, never in the exported dataset.
  8. Two oracles beyond "tests pass". Screenshots are the visual oracle; axe is the accessibility oracle. Role/DOM assertions prove behaviour, not appearance.
  9. Every behavioural change ships behind a flag, default OFF. Unset flag = byte-for-byte today's behaviour. Flag parsing is fail-closed.
  10. Decisions are documentation. A slim present-tense digest of standing calls, an append-only dated log, open questions flagged not silently resolved, and manual user stories kept 1:1 with the E2E suite.

A note on the code samples

Code shown here is drawn from the live Floaty source and is faithful to its patterns, but it is meant to teach the convention, not to be copy-pasted byte-for-byte. The authoritative source is always the repo itself; configuration files (package.json, tsconfig, eslint, vite, playwright) are reproduced exactly because they are meant to be copied.
