The CapacityLens house style

The CapacityLens Handbook

This is the complete architectural, coding, and process handbook for CapacityLens — the original repo. It exists so its sibling projects (floaty-schedule, a shift rota; delivery-diary, a deliverables tracker; and any future relatives) can adopt the same conventions exactly, down to the small decisions. If you are a developer or an AI agent standardising a sibling repo: read this front to back, then keep it open as a reference.

What these repos share

CapacityLens and its siblings are all the same machine re-skinned for a new domain: a browser SPA over a pure domain core, server-backed by default (a Node + node:sqlite API behind one persistence seam, with an in-browser localStorage build as an explicit demo opt-in), opt-in session auth behind one adapter port, multi-tenancy by account, and a relentless testing gate. The domain changes (people×allocations vs. shifts vs. deliverables); the architecture, conventions, and process do not. This handbook is the part that does not change.

Updated for the open-source posture (2026-07)

This edition reflects the big inversions since the first edition: persistence flipped from local-first to server-backed by default; a real auth & access-control layer (Better Auth, roles, invites, audit) landed behind seams that keep the auth-off deploy byte-identical; deployments default to one company per instance; the UI adopted shadcn/ui + Tailwind and passed a WCAG 2.2 AA sweep; and strings compile through Paraglide so a missing key fails the build. Where a rule inverted, the old rule is called out so a sibling mid-migration knows which side of history it is on.

How to use this handbook

Each page is self-contained but they build on each other. Pages 01–02 orient you. Pages 03–07 and 12 are the build: architecture down through every layer, then the server and its auth/access layer. Pages 08–10 are how we keep it correct and ship it. Page 11 is the cheat-sheet you will keep returning to.

Throughout, four kinds of callout flag what matters most:

Load-bearing rule

A decision the code depends on. Copy it verbatim into the sibling. Breaking it breaks an invariant.

Why

The reasoning behind a decision. When you are tempted to do it differently, read this first.

Do this

The positive pattern to follow.

Don't

An anti-pattern that was considered and rejected, or a footgun that has bitten this codebase before.

The twelve ideas everything else follows from

If you internalise nothing else, internalise these. Every page is an elaboration of one of them.

  1. Be deliberately small. One product solves one problem well. Non-goals are written down and defended. New scope is rejected by default.
  2. Server-backed by default, one seam, demo opt-in. An empty env means the same-origin SQLite server (/api); VITE_CAPACITYLENS_API overrides the origin; VITE_CAPACITYLENS_DEMO=1 is the only route to the in-browser localStorage build. Both are implementations of a single PersistenceAdapter interface and nothing else knows which is active.
  3. One source of truth, shared so it can't drift. A pure domain core (@capacitylens/shared) is imported by both client and server. Server validation is client validation — the same code, not a re-implementation.
  4. Forms reject; import and server repair. The same rule has two enforcement modes: a person fixes their mistake at a form; a machine import or a direct API write strips/repairs/drops instead.
  5. Extending an entity is drift-proofed. Most parallel lists are exhaustiveness-checked against the types, so a new field that misses one fails the build; the FK map only type-checks its values, so a missed FK field is caught by the round-trip fixtures instead of silently dropping.
  6. Multi-tenant schema, single-company deployments. Every entity carries an accountId; the active account is chosen on load and never persisted; all scoped reads go through one hook. But a self-hosted instance serves one company by default — creating a second is refused unless CAPACITYLENS_MULTI_ACCOUNT=1.
  7. Auth is a seam, and OFF is a guarantee. Better Auth sits behind a one-method AuthAdapter port; roles flow through one pure can(role, action); control tables (members, invites) are structurally excluded from app data. CAPACITYLENS_AUTH unset means byte-for-byte the trusted-local behaviour — no auth tables, no login UI.
  8. Device-global vs. account data are separate. Theme, display toggles, sidebar state live in their own localStorage keys, never in the exported dataset.
  9. Two oracles beyond "tests pass". Screenshots are the visual oracle; axe is the accessibility oracle (WCAG 2.2 AA is a gate, not an aspiration). Role/DOM assertions prove behaviour, not appearance.
  10. Every behavioural change ships behind a flag, default OFF. Unset flag = byte-for-byte today's behaviour. Flag parsing is fail-closed. The few deliberate exceptions (safety interlocks, the on-by-default audit log) are named and justified in the flag register.
  11. Strings compile; a missing key fails the build. i18n is Paraglide (inlang): messages are typed functions compiled from messages/<locale>.json, so a removed or typo'd key is a tsc error, never a runtime miss.
  12. Decisions are documentation. A slim present-tense digest of standing calls, an append-only dated log, open questions flagged not silently resolved, and manual user stories kept 1:1 with the E2E suite.

A note on the code samples

Code shown here is drawn from the live CapacityLens source and is faithful to its patterns, but it is meant to teach the convention, not to be copy-pasted byte-for-byte. The authoritative source is always the repo itself; configuration files (package.json, tsconfig, eslint, vite, playwright) are reproduced exactly because they are meant to be copied.
