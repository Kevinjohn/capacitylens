Domain core — @floaty/shared

The pure, environment-agnostic package that both the app and the server import. This is where the rules live exactly once. If you copy one thing carefully into a sibling, copy the shape of this package and its conventions.

Module organisation

Four sub-directories, each with a strict rule about what belongs there. The split is by how much of the dataset a module touches, which is also the dependency order.

DirHoldsThe rule
types/Types, enums, type-derived constants, and the clamps that are part of the domain contract.Zero dependencies. No behaviour beyond type guards and contractual clamps.
lib/Pure utilities keyed on the domain types — one file per concern (id, strings, dates, colour, integrity, sanitize, scheduling-days).Pure functions. May import types/. Minimal cross-lib imports.
data/Whole-dataset lifecycle: migrate, seed, export/import transfer, test fixtures.Knows about AppData as a whole and schema versions. Imports types/ + lib/.
domain/Whole-AppData mutations + the tenancy predicate — the logic that must be identical on client and server.Pure. Imports everything below. No store, no React, no I/O.

Consumed as raw source: the package's exports map ("./*": "./src/*.ts") means import { migrate } from '@floaty/shared/data/migrate' resolves straight to the .ts file. No build, no dist/, no drift between built and source.

types/ — the shapes and the contractual helpers

Beyond the entity interfaces (page 03), types/entities.ts holds the enums, their runtime mirror arrays, the clamp helpers, and the empty/seed sentinels. The runtime arrays exist so the server's sanitiser can reject a junk enum value without re-typing the union.

shared/src/types/entities.ts (enums + runtime lists)
export type SchedulingMode = 'hourly' | 'days' | 'blocks'
// Runtime list of the valid scheduling modes — the single source the server's
// sanitiser uses to reject a junk `schedulingMode` on a direct account write.
export const SCHEDULING_MODES: SchedulingMode[] = ['hourly', 'days', 'blocks']

The clamp helpers live here (not in lib/) because they are the domain's contractual ceiling/floor, imported identically by the store's write boundary and the import sanitiser:

shared/src/types/entities.ts (clamps)
export const MAX_HOURS_PER_DAY = 24

/** Clamp an ALLOCATION's hours/day into [0, 24]; non-finite → 0. 0 is legal (a
 *  'blocks' booking carries 0 load). Shared by the store write + the import sanitiser. */
export function clampHoursPerDay(h: number): number {
  return Number.isFinite(h) ? Math.max(0, Math.min(h, MAX_HOURS_PER_DAY)) : 0
}

/** Clamp a RESOURCE's working hours/day to (0, 24]. Unlike an allocation, a resource
 *  must work a POSITIVE number of hours, so junk / <= 0 falls back to a normal 8h day. */
export function clampWorkingHoursPerDay(h: number): number {
  return Number.isFinite(h) && h > 0 ? Math.min(h, MAX_HOURS_PER_DAY) : 8
}
Why two clamps, not one

They encode different domain rules: an allocation may carry zero load (a pure booking block), but a resource with zero capacity is nonsensical, so its clamp floors to 8 rather than 0. Putting both in types/ and importing them everywhere means the form, the store, the import path, and the server can never disagree on the ceiling. There is no "import version" and "form version" of a clamp — there is one clamp.

Also here: SCHEMA_VERSION (bumped when the persisted shape changes; drives migrate), emptyAppData() (the all-empty dataset), and isEmpty(data) — the single definition of "genuinely empty" shared by the client bootstrap and the server's init-marker backfill so the two "is this empty?" checks can't drift.

lib/ — pure utilities, one concern per file

id.ts
newId() wraps crypto.randomUUID(). All id generation goes through it, so call sites never touch the crypto API directly.
strings.ts
Text hygiene — the denylist (below), MAX_NAME_LENGTH (100) / MAX_NOTE_LENGTH (1000), hasDisallowedChars() (forms test), cleanText() (import strips).
dateMath.ts
All date arithmetic, wrapping date-fns so call sites use whole-day integer offsets on date-only ISO strings, never millisecond Date math (which causes DST off-by-ones). Includes the hot-path isWithin.
color.ts
Bar/text colour resolution, WCAG contrast math (contrastRatio, readableTextColor, ensureBarColors), and isHexColor().
integrity.ts
Referential-integrity validators (return ValidationResult) + the pure cascade-delete transforms.
sanitizeImport.ts
Value-level record repair for the import/server path; contains the exhaustive switch.
schedulingDays.ts
Conversions between stored [start, end, hoursPerDay] and the user-facing "days of work over a span" inputs.

The text denylist (allow-most, deny-specific)

Text validation is a denylist, not an allowlist: it rejects emoji, symbol-other, control/format/surrogate/private/unassigned codepoints, and keycap/variation-selector marks — but allows all letters, marks, digits, punctuation, whitespace, currency and math. So José, Müller, O'Brien & Co, CJK and €£+= all pass.

Do this

One regex, one definition, used by client and server. The form calls hasDisallowedChars() to reject; the import/server call cleanText() to strip. Deny only what is genuinely hostile (it deliberately does not ban all nonspacing marks, which would strip legitimate decomposed accents). An allowlist would reject the next language someone types.

The date hot path: string compare, not re-parse

shared/src/lib/dateMath.ts
// Zero-padded YYYY-MM-DD strings sort chronologically, so a plain string compare is
// exact AND avoids three parseISO calls — and this is the scheduler's hottest path.
// The padding assumption is not just convention: validateDateRange rejects any
// non-YYYY-MM-DD date at EVERY write boundary, guaranteeing it.
export function isWithin(date: ISODate, start: ISODate, end: ISODate): boolean {
  return date >= start && date <= end
}
A pattern worth stealing: pay validation once to make the hot path free

The shortcut (compare strings) is only correct because an invariant is enforced at the boundary (all stored dates are zero-padded ISO). The codebase repeatedly trades a one-time validation cost for a cheaper steady state, and documents the dependency at both ends — the comment on isWithin names the validator it relies on. When you optimise, write the invariant down where the next person will look.

Validation: reject vs. repair, one rule each

This is principle 4 in code. Each constrained field has one definition, two enforcement modes:

shared/src/lib/strings.ts (module header — the contract, stated)
// Text hygiene for user-entered free text. Two surfaces use it:
//   - the FORMS reject disallowed input via hasDisallowedChars (so the user fixes it);
//   - the IMPORT + SERVER write paths can't show a form error, so they STRIP it via
//     cleanText (consistent with sanitizeImport's repair-don't-reject rule).
// One definition, imported by client + server, so the two paths can never drift.

Validators that a form consumes return a uniform result type — they never throw:

shared/src/lib/integrity.ts
export interface ValidationResult { ok: boolean; errors: string[] }

// Both ends required, both real calendar dates, not reversed. The single source
// enforced at every write boundary (and the guarantor of isWithin's string compare).
export function validateDateRange(startDate?: ISODate | null, endDate?: ISODate | null): ValidationResult

The store/server boundary wraps the same validators in thin throwing asserts, because at a write boundary an invalid value is exceptional:

shared/src/domain/mutations.ts
export function assertDateRange(startDate?: ISODate, endDate?: ISODate): void {
  const v = validateDateRange(startDate, endDate)
  if (!v.ok) throw new Error(v.errors[0])
}
Load-bearing rule

Validators return ValidationResult; asserts throw. Forms call validators and render the error; the store and server call asserts and let the boundary reject. Never throw from a validator a form consumes, and never swallow an assert at a write boundary.

Real-calendar validity is checked by a round-trip, which catches rolled-over dates like 2026-02-30 that a regex alone would pass:

export function isValidISODate(s: unknown): s is ISODate {
  if (typeof s !== 'string' || !/^\d{4}-\d{2}-\d{2}$/.test(s)) return false
  const parsed = parseDate(s)
  return !Number.isNaN(parsed.getTime()) && toISODate(parsed) === s
}

Integrity & cascade — pure transforms

The cascade-delete functions are pure: they take AppData and return a new AppData, never mutating. They encode the domain's deletion semantics, and the required-vs-optional FK distinction shows up directly:

  • deleteResourceCascade — removes the resource + its allocations + its time-off.
  • deleteActivityCascade — removes the activity + its allocations.
  • deleteProjectCascade — removes the project, its phases, its activities (and those activities' allocations), and unbinds (does not delete) placeholders bound to it.
  • deleteClientCascade — the project cascade across all the client's projects, in a single pass using Set membership (not N recursive calls).
  • deletePhaseCascade / deleteDisciplineCascade — non-destructive: they ungroup (null the optional FK on) activities / resources rather than deleting them.
Load-bearing rule

Deleting a required-parent removes the children (cascade). Deleting an optional-parent ungroups the children (set-null). The import repair (below) mirrors this exactly — a dangling required FK drops the row; a dangling optional FK unbinds it. The two halves of this symmetry must always move together.

Where timestamps are bumped — and aren't

Cascade and integrity functions never touch updatedAt. Timestamp bumping on an edited row is the store's job (page 05). The core computes the next data shape; the orchestrator owns identity and time. Keeping that division clean is what lets the same cascade run on the server.

data/ — lifecycle: migrate, seed, transfer, fixtures

migrate.ts — turn any blob into current-shape AppData

Two guards protect the import path before any migration runs, so a non-Floaty file can never wipe data:

export const KNOWN_KEYS: string[] = ['accounts', ...SCOPED_KEYS]   // derived — can't drift

// True if the candidate has at least one known table that IS an array.
export function looksLikeFloaty(value: unknown): boolean

// True if a known key is PRESENT but NOT an array (structural damage from a
// truncated/hand-edited file). migrate would silently coerce it to []; this
// gate hard-rejects instead so the loss is surfaced, not swallowed.
export function hasNonArrayKnownTable(value: unknown): boolean

migrate(raw) then: returns emptyAppData() for non-objects; detects the schema version (absent = legacy); unwraps { schemaVersion, data } or accepts bare AppData; applies version steps (e.g. an old boolean → an enum); and normalises every known table to an array. It never throws on bad data — it repairs or empties.

A migration that needs no migration step

When Floaty added accountId to every scoped entity (v2→v3), it did not write a migration function — it changed the live localStorage key from floaty/v2 to floaty/v3, orphaning the old data, and the import path stamps accountId on every incoming row. Bumping the storage key is a legitimate "migration" for a local-first app where the old data is demo-only. Decide per change: transform, or bump-and-orphan.

transfer.ts — export/import with caps

serializeData wraps AppData as { schemaVersion, data } pretty JSON. parseData is the only import-path function that throws — and only for structural failures the user must see before any change: not JSON, not Floaty (looksLikeFloaty fails), structurally damaged (hasNonArrayKnownTable), over the record cap (MAX_IMPORT_RECORDS = 200,000), or zero records. The caps guard against a self-DoS / JSON bomb.

seed.ts & fixtures.ts — two different jobs

seed.ts is the demo dataset shown on first run: two companies (one rich enough to demonstrate stacking, over-allocation, placeholders and time-off; one near-empty to prove account switching), with stable literal ids (r-tyler, p-acme) and inline comments explaining the scenario each row demonstrates. fixtures.ts is the opposite: every optional field set to a non-default value, used as the drift-proof round-trip oracle in server tests.

The import-remap algorithm

remapAndValidateImport(data, accountId, incoming, now) in domain/mutations.ts is the import pipeline shared by the in-app import and the server's /api/import. It is the most intricate piece of the core; understand its five phases and you understand the repair philosophy.

  1. Per-table id maps. One Map<sourceId, freshId> per entity type. Every incoming row gets a brand-new id (imports never trust source ids). Cross-table id collisions are safe because each FK resolves against its own table's map.
  2. FK remap + value sanitise. Each row is given its fresh id, stamped with the active accountId and now timestamps, has each FK remapped through the right table's map, and is passed through sanitizeImportedRecord for value-level repair.
  3. Referential repair, parent-before-child. Walk the dependency tree top-down so each child sees the surviving parent set: projects (drop if clientId didn't survive) → phases (drop if projectId gone) → resources (unbind dangling optional disciplineId/projectId) → activities (unbind dangling optional projectId; clear phaseId if its phase is gone or belongs to a different project) → allocations (drop if resourceId or activityId missing, or the date range / placeholder rule fails) → timeOff (drop if resourceId missing or range invalid).
  4. Assemble. For each scoped table, keep the OTHER accounts' rows (via notInAccount) and append the repaired incoming rows.
  5. Return counts. An honest { imported, skipped } — a row that merely had an optional FK unbound still counts as imported; only dropped rows are skipped.
Load-bearing rule

The repair order is the FK dependency order. A child is repaired only after its parent's survivors are known. Required FK dangling → drop the row (mirrors cascade). Optional FK dangling → unbind it (mirrors set-null). This is the exact mirror of the cascade rules in integrity.ts; the two are designed as a pair.

Drift-proofing: the exhaustive switch & the typed map

The sanitiser dispatches per entity key and ends with a never assignment, so adding a ScopedEntityKey without a case fails the build:

shared/src/lib/sanitizeImport.ts
export function sanitizeImportedRecord(key: ScopedEntityKey, rec: Record<string, unknown>) {
  switch (key) {
    case 'resources': /* clamp hours, dedup workingDays, clean text … */ break
    case 'allocations': /* clampHoursPerDay, normalize dates … */ break
    /* … every other case … */
    default: {
      // Exhaustiveness: a new ScopedEntityKey without a case here won't compile.
      const _exhaustive: never = key
      void _exhaustive
    }
  }
  return rec
}

And the FK→table mapping is a typed record, so a typo'd or renamed table is a compile error, not a silent remap to undefined:

const FK_TARGET: Record<string, ScopedEntityKey> = {
  disciplineId: 'disciplines', projectId: 'projects', clientId: 'clients',
  phaseId: 'phases', resourceId: 'resources', activityId: 'activities',
}
The two exhaustiveness idioms to copy

(1) const _x: never = key; void _x in a switch default — for "handle every member of a union". (2) Record<string, UnionType> on a lookup map — for "every value must be a valid member". Page 05 and 07 show a third (Exclude<keyof X, Y> extends never) for "this list covers every key of that type". Between them, these lists are kept honest at compile time — with one caveat: idiom (2) checks the map's values, not that every FK field appears as a key, so a brand-new FK field missing from the map is caught by the round-trip fixtures rather than the compiler.

Coding conventions in the core

AspectConvention
Types / interfacesPascalCase — AppData, ScopedEntity, ValidationResult.
Runtime constantsSCREAMING_SNAKE_CASE — SCOPED_KEYS, MAX_HOURS_PER_DAY, FK_TARGET.
FunctionscamelCase — remapAndValidateImport, deleteClientCascade.
Boolean predicatesPrefixed is/has/belongs/not — isValidISODate, hasDisallowedChars, belongsToAccount.
ExportsNamed only. No default exports anywhere in the package.
PurityEvery lib/ and domain/ function returns new data via spread; the one in-place mutator (sanitizeImportedRecord) is documented as such and operates on a fresh copy.
ErrorsValidators return results; asserts throw; import/migrate never throw on bad data (only parseData throws, for structural failures).
Comment style: explain WHY, at both ends of a dependency

Every file opens with a 3–6 line block stating its role, which surfaces consume it, and the invariant it enforces. Comments justify decisions and name edge cases; they don't narrate the code. When a constant is used by two divergent paths, the comment says so ("used by client and server, so the two can't drift"). This density is not optional — it is how the next reader (human or AI) learns the why without spelunking. See page 11 for the full comment-style guide.
