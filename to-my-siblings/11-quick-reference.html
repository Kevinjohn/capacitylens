"Clear local data" wipes exactly the prefixed keys — never a blind localStorage.clear(). Auth sessions live in the Better Auth cookie, never in storage.

Comment style (the DEFENSIVE-CODING.md digest)

  • Intricate files open with a short header stating their role, who consumes them, and the invariant they enforce.
  • Comments explain why, for a junior — not what. If the code is clear, no comment. If a decision is non-obvious, the rationale and the rejected alternative go in a comment.
  • Every exported symbol gets TSDoc — highest priority in shared/ (it is the published API) — with preconditions and @throws where a throw has meaning.
  • Dual-surface contracts are stated: when a constant/function is used by two divergent paths (form vs import, client vs server), say so ("used by client and server, so the two can't drift").
  • Performance shortcuts name the invariant they depend on, at both ends (e.g. isWithin's string-compare comment names the validator that guarantees zero-padding).
  • Degenerate-case guards are always explained (why this clamp, why this fallback, why this iteration cap) — and deliberately unguarded hot paths carry a guard-comment saying not to wrap them.

The golden rules (load-bearing, consolidated)

Architecture

One persistence seam (PersistenceAdapter) — no code knows which backend is live; the choice is isDemoMode() alone (empty API var = same-origin server, the default). One auth port (AuthAdapter: null = 401, throw = 503) — nothing above it touches the provider. The core imports nothing up the stack and no environment API. Dependencies point inward. One cast for the scoped-table union, in one named seam (scopedTables).

Data & tenancy

Every entity carries id + createdAt + updatedAt; scoped entities also carry accountId. The active account is never persisted. The tenancy predicate is defined once. Switching accounts resets undo + per-account UI. The store owns ids, timestamps, and history; the core computes next-data only.

Validation

One rule per field, two modes: forms reject (validators return ValidationResult); import + server repair (sanitisers + asserts that throw at the boundary). Required-FK dangling → drop the row; optional-FK dangling → unbind. Import repair mirrors cascade, exactly.

Extension

Adding a field flows on rails: types → fixtures (non-default value) → server column spec (auto-ALTER) → sanitiser. Most parallel lists are exhaustiveness-checked, so a miss fails the build, not silently at runtime; the FK map is the exception — the compiler checks its values are valid table names, and the fixtures' round-trip catches a missed field.

UI

Every dialog is a <form noValidate> via Modal; Enter submits; only the primary button is type="submit". Semantic colour tokens only; dark mode is one data-theme swap. Native <select>. Preset swatches only. Accessibility is a gate. "Utilisation", never "Load".

Server & ops

Server is a thin shell over the core — no duplicated rules. 500 bodies are generic; never leak SQLite errors. Every behaviour change ships behind a fail-closed flag, default OFF = today's behaviour — with the named exceptions: safety interlocks (production posture, reset boot-guard, timeouts, helmet) are always-on, and the audit log is on-by-default. Single company per instance unless CAPACITYLENS_MULTI_ACCOUNT=1; a real deploy first-runs empty. Backups must have a tested restore (and the drill is a test). Rollback from server mode strands data — export first.

Auth & access

OFF is a guarantee: no auth tables, no login UI, the gate allow-alls on line one; the entire pre-auth suite green in off mode is the proof. One pure can(role, action) is the whole matrix, consumed by server routes and client affordances alike. Control tables never enter AppData, fixtures, generic CRUD or export. Sensitive fields are redacted server-side, per role. The server 403 is authoritative; client gating is UX; the store's viewer guard no-ops, never throws.

Process

Decisions: append one line to the dated log (tail-read), promote load-bearing calls to the digest, edit the digest line when they change. Flag open product questions; resolve code questions by reading source. REFERENCE.md first, then stories, then specs. Pure extractions land freely; structural splits need a characterisation test first. The merge bar is the full green gate.

The three exhaustiveness idioms

These are how the parallel lists are kept honest. Memorise them.

// (1) Handle every member of a union — in a switch default:
default: { const _exhaustive: never = key; void _exhaustive }

// (2) Every value of a lookup map must be a valid union member:
const FK_TARGET: Record<string, ScopedEntityKey> = { projectId: 'projects', /* … */ }

// (3) This list covers every key of that type:
type _Missing = Exclude<keyof AppData, TableKey>
const _complete: _Missing extends never ? true : never = true

// (3b) Bidirectional column-spec check (server tables.ts), enabled by `as const satisfies`:
type CheckColumns<E, C extends readonly ColumnSpec[]> =
  C[number]['name'] extends keyof E
    ? (Exclude<keyof E, C[number]['name']> extends never ? true : never) : never

Anti-patterns — considered and rejected

Don't add abstraction this small product doesn't need

No generic store/CRUD factory, no schema-driven forms, no custom listbox replacing native <select>, no ORM, no form/fetch/state-machine library. Every runtime dependency has one named job (the shadcn layer included — page 02); reject any addition that duplicates a job something already does. Explicit hand-written forms/lists at this scale beat the abstraction.

Don't hand-roll what the kit already owns

Don't hand-roll a toast — surface via setNotice (the Sonner bridge; the old Toast was retired). Don't build a dialog off Modal, a list row off the icon-button shells, or a string outside the message catalogue. Don't import a shadcn ui/ file from feature code — go through common/.

Don't reintroduce silent failures

Don't drop the dev-server host pin or strictPort (silent wrong-port white page). Don't remove the JS-less placeholder. Don't let the server fail on a later write instead of throwing at startup on schema drift. Don't leak raw SQLite errors in 500 bodies.

Don't merge distinct signals or mix data with preferences

Keep conceptually distinct signals on separate code paths (and name them differently). Never put theme or display toggles in AppData/export. Never persist the active account id.

Don't bypass the rails

Don't add a field to the type and skip the fixture/column/sanitiser. Don't put a floating promise without void. Don't validate from a form by throwing, or swallow an assert at a write boundary. Don't refactor a hot module without a characterisation test.

Don't grow the scope

Reject features that add process or granularity unless the owner has changed the product statement. Mobile views, budgets, timesheets, hour-granularity, per-seat gating are non-goals — cite them.

Runbook: standardise a new sibling repo

An ordered, do-this-then-that sequence a developer or an agent can follow. Each step ends green. Remember the shape you are building: server-backed by default, demo build opt-in — the server is part of the product, not an appendix.

  1. Copy the toolchain (page 02): the three package.jsons, five tsconfigs, eslint.config.js, vite.config.ts, playwright.config.ts, index.html, components.json, .env.example, project.inlang/ + messages/en.json, the scripts/ folder (dev orchestrator + e2e runners), the Docker trio, and the dotfiles. Rename the app, the @scope/shared package, the env-var prefix, and shared/src/brand.ts. Node 24, npm install at the root. Confirm an empty npm run gate passes (it compiles Paraglide first — that's expected).
  2. Write the product layer (page 01 + 09): the one-sentence problem statement and the non-goals, in the new CLAUDE.md and DECISIONS.md; copy DEFENSIVE-CODING.md verbatim. Copy the standing architectural decisions from this handbook into DECISIONS.md so they're explicit.
  3. Define the data model (page 03–04): write shared/src/types/entities.ts for the new domain — keep the Entity/ScopedEntity bases, the tenant-at-top shape, required-vs-optional FKs, the enum + runtime-list pattern, the clamps, SCOPED_KEYS/scopedTables, SCHEMA_VERSION, emptyAppData/isEmpty — plus lifecycle fields on the entities that need archive/delete.
  4. Port the core (page 04): bring over lib/ (id, strings, dateMath, color where relevant) and domain/access.ts + domain/lifecycle.ts (near-verbatim), then write the domain's integrity validators + cascade transforms, the sanitiser (with the exhaustive switch), tenancy.ts, and mutations.ts (asserts + remapAndValidateImport). Write its unit tests as you go.
  5. Port the server (page 07 — it is on the default path, not optional): copy tables.ts (rewrite column specs for the new entities — the CheckColumns guard will tell you when they match), rowCodec, db.ts, txn.ts, validate.ts, tenantStore.ts, app.ts, and the hardening modules. It is a thin shell — most of it is unchanged.
  6. Build the store & seam (page 05): the Zustand store (Draft/Patch, stamp/touch, mutate, undo, requireAccount, action patterns, the viewer guard), selectors, useScopedData/useActiveScopedData, the PersistenceAdapter with both implementations (ServerSyncAdapter + syncOps for the default build, LocalStorageAdapter for the demo build), and persist.ts (bootstrap, debounce/flush, the switch orchestrator).
  7. Wire auth & access (page 12): copy the whole layer — authAdapter.ts, auth.ts, controlTables.ts, membership.ts, the authorize gate, erasure — plus the client's src/auth/ (AuthProvider, LoginScreen, PermissionProvider). Adapt only the role matrix, the lifecycle entity list, and any role-redacted fields. Verify the OFF guarantee: every pre-auth test green with CAPACITYLENS_AUTH unset.
  8. Build the UI (page 06): copy common/ + ui/ wholesale (kit + Modal + fields + icon buttons), src/i18n/, index.css tokens, the canonical form + list per entity, the domain's main view. Keep the a11y patterns and the WCAG 2.2 conventions; put every string in messages/en.json from day one.
  9. Stand up testing (page 08): setup.ts, fixtures, the frozen-clock openApp (set the constant to your seed window), REFERENCE.md, the a11y oracle + the bespoke WCAG 2.2 tests, the db-backed and auth-backed projects. Write component + E2E coverage 1:1 with user stories.
  10. Set the merge bar (page 02/08): the bar is gate + gate:server + e2e green, run locally before pushing; e2e:browsers before a release. There is deliberately no committed CI workflow — the local gate is the enforcement mechanism.
  11. Deploy & operate (page 10): the Docker Compose two-service deploy is the default path (empty first run, one company, posture interlock); the static build is the demo. Turn on the ops flags (log, deep health, rate limit, backups) for anything real, and run the restore drill.
The end state

A sibling that a stranger could mistake for CapacityLens under the hood: same layers, same conventions, same gate, same process — solving a different problem. That is what "standardise" means here, and it's the whole reason this handbook exists. Welcome to the family.
