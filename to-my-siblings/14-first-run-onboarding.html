Build a first-run onboarding: a state-driven checklist + a loose tour

CapacityLens shipped this on 2026-07-07 (commit df3483c): a "Getting started" card on the main view that ticks itself off as the account fills up, and a five-stop "Show me around" spotlight tour. This page is the recipe for a sibling repo. The pattern transfers verbatim; the content — which steps, which tour stops, which user story — is yours to derive, and §1 is the worksheet that derives it. Do the phases in order; each ends with a check.

The two-part pattern, and the two rules that make it work

Onboarding is two separate artefacts with two different jobs:

ArtefactJobCharacter
Getting-started checklist (card on the main view)The how: the 3–5 actions that take an empty account to a useful one.State-driven — each step ticks itself off by reading the active account's real scoped data.
"Show me around" tour (driver.js spotlights)The where: point at the main view, the toolbar, and the nav destinations.Loose — it never navigates, never opens forms, never waits on the user doing something.
Rule 1 — the checklist is derived, never scripted

A scripted "now click here" flow breaks the moment the user wanders off mid-flow, and lies the moment it gets out of step with reality. Deriving each step as a boolean off scoped data means the card is always honest: do the step anywhere, by any means (form, import, API), and it ticks. This is why the derivation lives in a pure, unit-tested function and not in the component.

Rule 2 — the tour looks, it doesn't touch

The tour's steps set disableActiveInteraction: true and never navigate: a stray click on a spotlighted nav link would change the page underneath the overlay. Task-by-task guidance is the checklist's job; mixing the two produces the brittle walkthroughs everyone dismisses unread.

§1 — Design worksheet: derive YOUR content first

Do this on paper before writing code. Answer the four questions, then fill the table — the table is your spec for §2–§7 and for the user story.

  1. What is the shortest path from empty to "the main view shows something useful"? List the entities a user must create, in dependency order. That list, capped at ~5, is your checklist. Each step must be checkable as a boolean off scoped AppData — if you can't write the predicate, it's not a step.
  2. Which built-in/seeded rows must NOT count? CapacityLens excludes the builtin Internal client (isBuiltinClient) — every account has it, so it isn't "your first client". Find your equivalents (a default team? a seeded category?) or the card will start life half-ticked and teach nothing.
  3. Where does each step happen? Steps whose page is elsewhere become <Link>s to that route. A step that happens on the main view itself (the final "use it" step, usually) gets no link — a hint line instead. CTAs navigate; they must NOT try to open another page's Add modal (those are local component state — reaching across is the anti-pattern).
  4. What are the five-ish tour stops? The main view, its toolbar, and the 2–3 nav destinations a new user needs. Anchors: existing data-testids for in-view elements, data-nav="<route>" for sidebar links (§5).

CapacityLens's filled-in worksheet, as the worked example:

Sketches for the siblings — validate against your own domain model before adopting; the predicates must name your tables:

  • Shift-rota sibling: add a location/venue → add your people → define a shift pattern → put someone on the rota (final step, on the grid, hint not link).
  • Deliverables sibling: add a client → add a deliverable → log your first update/entry (final step, on the diary view, hint not link).
Write the user story from the worksheet, before the code

Per page 09, user-stories/REFERENCE.md is updated first. Draft your US-NAV-style story straight from the table: precondition (clean state), the card appears on an empty account with all steps pending, doing step 1 by its normal page ticks it, the tour drives all stops, Dismiss hides and persists. CapacityLens's US-NAV-13-getting-started.md is the template — same sections (Goal / Why / How / Result), same 1:1 link to the e2e spec.

§2 — The pure derive function

One tiny file in src/lib/ (NOT in the component): an interface with one boolean per step, and a function from scoped AppData to it. The real one, whole:

src/lib/gettingStarted.ts
import { isBuiltinClient } from '@capacitylens/shared/data/internalClient'
import type { AppData } from '@capacitylens/shared/types/entities'

/** Which onboarding steps the active account has completed. */
export interface GettingStartedSteps {
  client: boolean
  project: boolean
  person: boolean
  assign: boolean
}

/** Derive the checklist's per-step completion from an account's scoped AppData. The
 *  built-in Internal client doesn't count — every account has it, it's not "your first client". */
export function deriveGettingStartedSteps(data: AppData): GettingStartedSteps {
  return {
    client: data.clients.some((c) => !isBuiltinClient(c)),
    project: data.projects.length > 0,
    person: data.resources.length > 0,
    assign: data.allocations.length > 0,
  }
}

Swap the fields and predicates for your worksheet rows. Keeping it pure and in src/lib matters twice: it's trivially unit-testable, and that directory is inside the mutation-testing scope (page 08) — Stryker will tell you if your tests don't actually bite.

Check: a unit test file next to it (see §7.1) runs green.

§3 — The dismissal device pref

Dismissal is a device preference, not account data: own localStorage key, never in AppData/export, never on the undo stack (the page 05 invariant — same class as theme and introSeen). Three touch points, all mirroring introSeen exactly:

  1. src/lib/displayPrefs.ts: a <prefix>gettingStartedDismissed key with readStoredGettingStartedDismissed() / writeStoredGettingStartedDismissed() via the existing readBoolPref/writeBoolPref helpers ('on'/'off' strings, default false so the card shows on first contact, storage-unavailable degrades to the default — the documented non-tenant-device-pref exception to surface-not-swallow).
  2. src/store/useStore.ts: a gettingStartedDismissed field initialised from the reader, and a setGettingStartedDismissed action that persists then set()s. Plain set, NOT mutate — a device-global view flag must never enter undo/redo or the export.
  3. TSDoc on both stating the two-part visibility contract: this flag records only an explicit dismissal; the card also self-hides (without touching the flag) once every step is complete.

Check: the displayPrefs unit tests gain read/write/default cases for the new pref (CapacityLens added 24 lines of test for exactly this).

§4 — The checklist card

One component (src/components/GettingStarted.tsx), rendered at the top of the main view (CapacityLens: above the scheduler toolbar in SchedulerView). The load-bearing parts:

  • Visibility — one early return: if (dismissed || allDone || activeRole === 'viewer') return null. Dismissed is the §3 pref; allDone is derived (a seeded/established account never sees the card); a Viewer is excluded because every CTA is a write affordance they can't complete — the same rule that hides the list pages' Add buttons (page 12).
  • Step rows: done = check icon + struck-through label plus a screen-reader-only "done" prefix (gs_step_done_sr — the strike-through is invisible to a screen reader); pending = a <Link> to the step's page, or plain text + hint for the in-place final step.
  • Footer: a primary "Show me around" button (startTour, §5) and a ghost "Dismiss" button (setGettingStartedDismissed(true)).
  • Hooks & naming: data-testids getting-started, getting-started-tour, getting-started-dismiss; an aria-labelled <section>; every string a Paraglide gs_* key (CapacityLens added ~24 gs_*/tour_* messages) — no literals, per page 06.
  • Data source: useActiveScopedData() — the card reads the ACTIVE account through the scoping seam like everything else; never the raw tables.

Check: with a fresh empty account the card shows all-pending; with the demo seed it never appears.

§5 — The tour

Library: driver.js. Chosen over Shepherd and Intro.js because they're AGPL/commercial while driver.js is MIT, ~5 kB, zero dependencies — re-run this license check in your own year before adopting anything else. One module, src/lib/tour.ts, exporting startTour():

  • Steps built at CALL time, inside startTour() — Paraglide messages resolve when invoked, so the active locale applies (the same deferred-resolution rule as AppShell's nav links). A module-level steps array would freeze the boot-time locale.
  • Anchors: in-view stops reuse existing data-testids (scheduler-grid, scheduler-toolbar — yours will differ); nav stops use a new data-nav="<route>" attribute added to BOTH sidebar variants — the open-menu NavLinks and the collapsed icon rail — so [data-nav="/x"] matches whichever is rendered. Comment it as a tour anchor, NOT an e2e selector (specs keep using role/name).
  • Nav stops pin side: 'right' — auto placement drops the popover below the small link, covering the neighbouring nav rows it's pointing at.
  • Config: showProgress with a translatable progressText (driver.js interpolates its own {{current}}/{{total}} tokens — pass them through the Paraglide message as literals), translated Next/Back/Done labels, and disableActiveInteraction: true (Rule 2).
  • Missing anchors degrade gracefully — driver.js renders a step whose element is absent as a centred popover rather than throwing. Don't add defensive existence checks; that's the library's documented behaviour.

Check: click "Show me around" — all stops highlight the right element, and during a stop, clicking the spotlighted nav link does not navigate.

§6 — The CSS import-order gotcha (read before styling)

driver.css must be imported BEFORE index.css, in main.tsx

index.css re-skins the tour popover onto the app tokens with a .driver-popover block (~60 lines: surface/ink/line/brand variables, both themes). Those overrides are equal-specificity with the vendor stylesheet, so cascade order decides. Importing driver.js/dist/driver.css from lib/tour.ts placed the vendor sheet after index.css in the bundle — and dark mode rendered white-on-white. The fix: import it in main.tsx, on the line above import './index.css', with a comment saying exactly why. This was caught by screenshot verification, not by the functional e2e — which is the whole argument for §8's visual pass.

§7 — Test plan (required)

7.1 — Unit (vitest, co-located):

  • src/lib/gettingStarted.test.ts — empty data → all false; each entity present → its step true independently; the exclusion case (a builtin-only client list does NOT tick the client step); all-populated → all true. One case per worksheet row plus one per exclusion.
  • src/lib/displayPrefs.test.ts — the new pref: default false when unset/garbage, round-trips true/false, storage-unavailable degrades to default.

7.2 — E2E (Playwright, one spec file, four specs). This shape maps 1:1 onto the user story; keep the names parallel to CapacityLens's e2e/getting-started.spec.ts:

SpecAsserts
Seeded company never shows the cardOpen the demo-seeded account → getting-started testid absent (the derived all-done self-hide).
Empty company shows it; a completed step ticks itself offCreate a fresh company → card with every step pending; do step 1 via its normal page; return → that row struck through, others pending (state-driven, not scripted).
"Show me around" runs the loose tourClick the tour button → popover with progress counter; Next through every stop; Done closes; URL never changed (the never-navigates rule, asserted).
Dismiss hides the card and persistsDismiss → card gone; reload → still gone (device pref survives); the account's data untouched.

7.3 — REFERENCE.md first, then the story. Add the three testids, the data-nav anchors, and the new gs_*/tour_* keys to user-stories/REFERENCE.md before writing the story or the spec (page 09's REFERENCE-first rule). Then write your US-NAV story from the §1 worksheet with its Linked-E2E line naming all four specs.

7.4 — Mutation note. The derive function sits in the Stryker scope. After the feature lands, an on-demand pnpm run mutation should show no surviving mutants in gettingStarted.ts — if one survives, your 7.1 cases assert too little (e.g. you asserted "some step is true" instead of "exactly this step").

§8 — Verification

pnpm run gate     # unit tests incl. the new derive + pref cases
pnpm run e2e      # incl. the four new onboarding specs
pnpm run dev      # then LOOK at it — this feature has two visual-only failure modes

The visual pass is not optional: screenshot the card and an open tour popover in both light and dark themes (§6's white-on-white bug is invisible to every functional test), and run the axe pass over the view with the card mounted (the card is a labelled section with real links and buttons — it's in scope for the a11y oracle, page 08).

Log the decision

Per page 09: one docs/decisions-log.md entry naming the state-driven-not-scripted call, the driver.js license rationale, the CSS-order gotcha (so the next person greps and finds it), and the test counts; promote the standing calls (state-driven checklist, loose tour, driver.js, dismissal as a device pref) to DECISIONS.md › UI & product. CapacityLens's 2026-07-07 entry is the template.
