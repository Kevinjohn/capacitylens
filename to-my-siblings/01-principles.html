Principles & the why

Every concrete convention in this handbook descends from a small number of judgement calls. Understand these, and the rest is consequence. This page is the philosophy; later pages are the mechanics.

1. Deliberately small (the most important one)

CapacityLens solves exactly one problem: a helicopter view of who is busy, free, or overworked, week by week, for small agencies with a few staff and rotating freelancers. That is the entire product. It deliberately replaces a resourcing spreadsheet — not a full PSA suite.

The owner-confirmed non-goals are written down so they can be defended:

  • No budgets or money.
  • No timesheets.
  • No hour-granularity workflows (week granularity is the grain).
  • No mobile views (light mobile affordances are the single agreed exception).
  • No per-seat / per-feature gating (the entitlements seam exists but is parked, default-unlimited).
Load-bearing rule

Reject features that add process or granularity. When a request would grow the surface, the default answer is no, and the non-goal is cited. A sibling repo has its own one-sentence problem statement and its own non-goals list — write them first, in its CLAUDE.md and DECISIONS.md, before any feature work.

Why

Scope is the thing that kills small, sharp tools. Writing the non-goals down converts a thousand future "could we just…" conversations into one citation. It also makes the codebase legible: a reader can trust that the absence of timesheets is a decision, not an omission.

2. Server-backed by default, one seam, demo build opt-in

With zero configuration the app boots against the same-origin SQLite server: an empty env means the client talks to a Node + node:sqlite REST API at the relative /api. VITE_CAPACITYLENS_API overrides the backend origin (never /api itself — the client composes ${API_BASE}/api/…). npm run dev is full-stack: one orchestrator boots the API on :8787 and Vite on :5173 with an /api proxy between them.

The old default — the whole dataset as one AppData blob in localStorage, no backend, no login — still exists, but as an explicit demo opt-in: VITE_CAPACITYLENS_DEMO=1 is the only route to the in-browser build (key capacitylens/v3, self-seeding, npm run dev:demo). It is the preview build, the cross-engine E2E target, and the zero-infrastructure demo — not the product's default posture.

Load-bearing rule

There is exactly one persistence seam — the PersistenceAdapter interface. The server sync adapter and the localStorage adapter are two implementations of it, chosen by one ternary on isDemoMode(). No component, store action, or selector may know which one is active. Adding a third backend means adding a third implementation, never a second seam. And note the selection logic: an empty VITE_CAPACITYLENS_API still means server mode (same-origin) — only the demo flag selects localStorage.

Why the flip (and what survived it)

CapacityLens began local-first. When it grew real self-hosted deployments, real auth, and an open-source posture, "the server is optional" stopped being true operationally — a team's data belongs in a database with backups, not in one browser's storage. The flip inverted the default, not the architecture: because everything already went through the adapter seam, flipping was a one-ternary change plus documentation. That is the lesson to carry into a sibling — build the seam first, and defaults stay cheap to change.

3. One source of truth, shared so it cannot drift

The pure domain core — types, validation, integrity rules, cascade-delete, import-remap, migration, seed, the access matrix, the lifecycle machine — lives in a workspace package (@capacitylens/shared) that is imported by both the web app and the server. There is no second copy of the rules on the server. Server-side validation is literally the client's code running in Node.

Why

The classic failure mode of a client + server pair is rule drift: the client clamps hours to 24, the server forgets to, and a hand-crafted request stores garbage. By making it physically the same module, drift is not "discouraged" — it is impossible. The package has no environment dependencies (no React, no DOM, no Zustand, no localStorage) precisely so it can run in both places. The same discipline now covers permissions: the pure can(role, action) matrix lives in shared, so the server's route gate and the client's affordance gating cannot disagree either.

4. Forms reject; import and the server strip/repair

Every constrained field has one rule, enforced two ways depending on who is at the keyboard:

SurfaceModeBecause
A person at a formReject — show an inline error, keep the dialog openThey can fix it; teaching them is better than silently changing their input.
A machine importRepair or drop — clean text, clamp numbers, mint a fresh id, drop a record with a dangling required FKThere is no human to prompt; rejecting the whole file because one row is bad is worse than salvaging the rest.
A direct API write to the serverRepair, then reject — clean the same value-level fields (enums, colour, hours), but throw ValidationError → 400 on a dangling required FK rather than dropping the rowA single crafted write is one record, not a file of salvageable rows; a referential break should fail loudly, not be silently swallowed.
Load-bearing rule

The definition of each rule (max length, allowed characters, hours clamp) is a single shared function. The form calls it to test ("is this disallowed?"); the import path calls a sibling function to fix ("clean this"). They can never disagree because they read the same constants. See Domain core → Validation.

5. Extending an entity is drift-proofed by the compiler

Adding a field to an entity touches several parallel lists: the type, the import sanitiser's switch, the foreign-key map, the server's column spec and creation order. Forgetting one would normally be a silent data-loss bug. Here, most of those lists are exhaustiveness-checked against the types, so a missed list is a compile error. The one exception is the FK map (Record<string, ScopedEntityKey>): the compiler proves every value is a real table name, but not that every FK field is listed — that gap is caught by the fully-populated fixtures, which round-trip through the server tests so a dropped or un-remapped field fails the gate.

Do this

When you add a field, follow the path on rails: shared types → fixtures.ts (set the new field to a non-default value) → server tables.ts columns (the auto-ALTER adds the DB column) → the sanitiser. If you miss a step, the build tells you exactly where. Page 03 walks the whole path.

6. Multi-tenant schema, single-company deployments

Every domain entity carries an accountId. You pick a company on load; the dataset is scoped to it. The active account id is held in memory and never persisted — it is a runtime selection, not part of the data. All scoped reads go through one hook/function so a read filter, a write guard, and a cascade can never independently define "belongs to account X".

The deployment shape, however, is deliberately narrower than the schema: a self-hosted instance serves one company by default. The server refuses to create a second accounts row (a 403 that names the fix) unless CAPACITYLENS_MULTI_ACCOUNT=1 — in every auth mode, create-time only (an existing multi-company DB keeps serving). The picker hides "New company" at the cap; the demo build and npm run dev stay uncapped.

Why keep the schema multi-tenant at all?

Because the schema is the expensive part and the cap is the cheap part. Multi-tenancy baked into every entity, read seam, and cascade from day one means the hosted/multi-company future is a flag flip, not a migration. Meanwhile the single-company default matches what a self-hoster actually wants, and shrinks the blast radius of the open trust model. A sibling should copy both halves: schema multi-tenant, deployment single-tenant by default.

Posture, stated honestly (two trust models)

With auth off (the trusted-local self-host default), the account picker is convenience scoping, not a security boundary: accountId is client-asserted and the server's ownership check is defence-in-depth. With auth on, the boundary is real: every request needs a session, reads are membership-gated per account (a non-member gets 403), the whole-dataset read is closed, and roles gate writes. Know which posture a deployment is in, and never expose isolated user data on the OFF model.

7. Auth is a seam, and OFF is a guarantee

Session auth (Better Auth: password and SSO modes, social providers, invites) exists — but the whole stack hangs off two narrow seams, and the unset state is a contract, not an accident:

  • One port for session verification. AuthAdapter { verifySession(headers) } is the only thing the request pipeline knows about. null means no session → 401; a thrown error means the auth backend failed → 503, never swallowed to "no session".
  • One pure function for permissions. can(role, action) in shared is the single access matrix, used by the server's route gate and the client's affordances alike.
  • The OFF guarantee. CAPACITYLENS_AUTH unset/off means byte-for-byte the trusted-local behaviour: no auth tables are created, no login UI mounts, the permission gate short-circuits to allow-all on its first line. The server-reported authMode is the only auth flag — there is no client-side auth env var to drift.
Load-bearing rule

Everything above auth depends only on the port, never on Better Auth directly. Swapping the provider is one adapter; and every auth feature must keep the OFF deploy inert — if a change makes auth-off behave differently, it broke the guarantee. Page 12 is the full mechanics.

8. Control plane ≠ data plane

Memberships and invites are about accounts, but they are not account data. They live in their own control tables, created idempotently on every DB open, and are structurally excluded from the app-data universe: never in AppData, the scoped-table lists, the fixtures, generic CRUD, or import/export. They are reached only through their own narrow functions and permissioned endpoints.

Why

The generic entity plumbing is deliberately powerful — auto-ALTER columns, generic /api/:entity CRUD, whole-dataset export. If an invites table ever entered that universe, a dataset export would hand out live, role-bearing tokens, and a generic write could mint an owner. The exclusion is an invariant with a test, not a habit. Any future control table (billing, quotas — see the parked entitlements seam) follows the same rule.

9. Device-global preferences are separate from the dataset

Theme, utilisation display toggles, allocation-bar label toggles, sidebar collapsed-state, weekend-minimise and week-snap prefs are per-device preferences, not data. Each lives under its own localStorage key (all under the capacitylens/ prefix) and is excluded from AppData and from export/import.

Why

If theme were in the dataset, importing a colleague's export would flip your colour scheme, and the server would sync your personal display choices to everyone. A preference about this screen belongs to the screen. The default theme is light; system follows the OS; a flash-of-wrong-theme guard runs before first paint. Account-level settings (timezone, week start, scheduling mode, feature toggles like disciplines/placeholders/external) are the opposite: they are shared data, on the Account entity, synced and exported.

10. Keep conceptually distinct signals distinct

A recurring discipline: when things look similar but answer different questions, they get separate code paths and separate names, and merging them is explicitly forbidden. CapacityLens's canonical example is its three over/utilisation signals:

  • The per-day over-marker flags any day where allocated > available (strictly greater — at-capacity is not over), across the whole timeline, weekend-aware.
  • The displayed utilisation % is a working-day ratio over the currently visible window (the 1/2/4/8-week zoom span), recomputed day-quantized on zoom/pan.
  • The overSoon red flag is a working-day ratio over a fixed forward 14-day window from today, zoom/pan-independent.
Load-bearing rule

They answer different questions ("is anything booked on a day off?" vs. "how loaded is this person in what I'm looking at?" vs. "is trouble coming in the next fortnight?") so they are never merged. Your sibling will have its own such families — find them, name them distinctly, and write the "don't merge these" note where the next person will see it. Naming is part of this: it is always "Utilisation", never "Load", everywhere in CapacityLens.

11. Two oracles beyond "the tests pass"

Automated tests assert behaviour, not appearance — role and DOM assertions prove that a button does the thing, not that it looks right. Two additional oracles cover what assertions can't:

  • Screenshots are the visual oracle. The E2E suite captures real renders at key states for human inspection.
  • @axe-core/playwright is the accessibility oracle, run in light mode, dark mode, and with a modal open — failing the suite on any serious/critical WCAG violation.

Accessibility is therefore a gate, not an afterthought — and since the WCAG 2.2 AA pass it is a standard, not just an absence of axe violations: per-route titles, portaled modals, live-region announcements, 24px targets, a dual-tone focus ring. See UI → Accessibility and Testing → The two oracles.

12. Every behavioural change ships behind a flag, default OFF

Production-hardening and new server behaviours are introduced behind environment flags whose unset state is byte-for-byte today's behaviour. The flags are CAPACITYLENS_* on the server (runtime) and VITE_CAPACITYLENS_* on the client (build-time, inlined). Flag parsing is fail-closed: a non-numeric rate limit, a typo'd auth mode, a missing secret — each means "off" or "refuse to boot", never "accidentally on".

Do this

A handful of changes are deliberately not flagged — and each is named with its rationale in the flag register. Graceful shutdown and a Node version bump (an "off" default would preserve a defect). The production posture interlock (in NODE_ENV=production the server refuses to boot with auth off, because a world-writable dataset is never an acceptable accident; the only escape is an explicit override flag that downgrades it to a loud warning). And the audit log, which is on by default (CAPACITYLENS_AUDIT !== 'off') — an accountability mechanism you must opt out of. The test is: would an "off" default preserve a defect or an unsafe silence? If yes, don't flag it; fix it. Otherwise, flag it off.

13. Strings compile; a missing key is a build error

i18n is Paraglide (inlang), compile-time, English-only for now. Messages live in messages/<locale>.json (committed, the source of truth) and compile into typed functions in a gitignored src/paraglide/; every build, test, and dev script runs the compile first. Components import m from a single src/i18n seam — never the generated package directly — and resolve keys at render, never at module load (a module-scope label list uses a getter, so a locale switch re-resolves).

Why compile-time

A runtime string lookup fails at runtime, on the one screen you didn't test, as a raw key leaking into the UI. A compiled message is a typed function: a removed, renamed, or typo'd key is a tsc error at the gate. The keys are snake_case and area-namespaced (nav_*, form_*, dialog_*…), the value is the source English, and a key is shared between call sites only when it is the same string and the same action. Even the single-locale phase pays for itself: it forced every string through one seam, which is the hard part of i18n — adding a language later is data, not surgery.

14. Privacy is proven, not promised

The product phones home to no one — and that claim is enforced by tests, from both ends. A dependency denylist test fails the gate if an analytics/telemetry/email package ever enters the tree (exact names, including transitives via the lockfile); a server test asserts the CSP's connect-src 'self' never widens, so the browser cannot be made to egress to a third party. Better Auth's own telemetry is explicitly disabled; the product sends no email (invites are links the admin distributes). The human-readable statement lives in docs/privacy.md; the tests keep it honest.

Do this in a sibling

Copy the pattern, not just the claim: any posture you advertise ("no tracking", "no egress", "your data never leaves the box") should have a test that fails when it stops being true. A README sentence rots; a gate assertion doesn't.

15. Decisions are documentation, and documentation is cheap to keep

The repo treats its own decisions as a first-class artifact, split by access pattern so keeping it current is nearly free:

DECISIONS.md
A slim, present-tense digest of the calls that still constrain the code. Read whole; short by design.
docs/decisions-log.md
An append-only, dated history of every review/remediation round. Large — you grep it or tail it, never read it whole.
NEEDS-INPUT.md
Open product questions. You flag these, you don't silently resolve them.
CLAUDE.md
The short working-notes file loaded into the AI pair's context every session — the load-bearing invariants in one breath.
DEFENSIVE-CODING.md
The error-handling and commenting standard (see §17). Short; followed on every change.
user-stories/
Manual test scripts kept 1:1 with the Playwright specs, with one REFERENCE file as their single source of truth.

The full mechanics — how to log a decision in one line, when to promote it to the digest — are on page 09.

16. A JS-less load is never a silent white page

A small but characteristic decision that captures the house attitude to failure. The entire app is JS-rendered, so a browser that blocks scripts would otherwise show a blank page with an empty console — undiagnosable. So index.html ships a static placeholder inside #root ("Loading… if this doesn't go away, JavaScript isn't running") that React replaces on mount, plus a <noscript> banner for the JS-disabled case.

Why this is a principle, not a detail

The instinct it encodes — a failure should diagnose itself, never present as a mysterious blank — recurs everywhere in this codebase: the dev server binds loudly rather than silently on the wrong port; the server throws at startup on schema drift rather than failing on a later write; load errors route to a specific recovery screen (server unreachable vs. corrupt local bytes are two different screens) rather than an empty app. Carry the instinct, not just the placeholder.

17. Surface, never swallow (and know where try/catch is harmful)

The same self-diagnosing instinct, made into an error-handling rule. A catch exists only to do one of three things: re-throw with more context, route the error to a visible surface (a form field error, a Sonner toast via setNotice, a typed load-error that picks a recovery screen, an HTTP status), or degrade to a documented default — and that last one only for genuinely non-load-bearing state (device preferences). A catch {} that drops the error, or one that replaces a specific, already-authored message with a generic one, is the anti-goal: it turns a loud, fixable failure into invisible data corruption.

Load-bearing rule

When you re-throw from a catch, attach the caught error as { cause: e } so the full chain survives, not just a re-worded message. ESLint's preserve-caught-error enforces this for native errors; it does not fire on custom error classes or a bare catch {}, so make those carry the cause too (CapacityLens's LoadError, ValidationError and AuthConfigError take an ErrorOptions and forward it). This convention came from a sibling repo — exactly the kind of lesson this handbook is for.

The half a naive sweep gets wrong

"More try/catch" is not the goal — surfacing is. Wrapping a pure function, a hot render path, or a deliberate integrity throw (the store's last-line guards) hides corruption rather than catching it. Put the safety as a clamp in the pure core instead, and leave a guard-comment so the next contributor doesn't re-add the harmful wrapper. The full standard — the boundaries that get guarded, the two-tier error model, the only acceptable swallows — is a short repo doc, DEFENSIVE-CODING.md.
