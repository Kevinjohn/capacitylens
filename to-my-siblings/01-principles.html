Principles & the why

Every concrete convention in this handbook descends from a small number of judgement calls. Understand these, and the rest is consequence. This page is the philosophy; later pages are the mechanics.

1. Deliberately small (the most important one)

Floaty solves exactly one problem: a helicopter view of who is busy, free, or overworked, week by week, for small agencies with a few staff and rotating freelancers. That is the entire product. It deliberately replaces a resourcing spreadsheet — not Float, not a PSA suite.

The owner-confirmed non-goals are written down so they can be defended:

  • No budgets or money.
  • No timesheets.
  • No hour-granularity workflows (week granularity is the grain).
  • No mobile views (light mobile affordances are the single agreed exception).
  • No per-seat / per-feature gating.
Load-bearing rule

Reject features that add process or granularity. When a request would grow the surface, the default answer is no, and the non-goal is cited. A sibling repo has its own one-sentence problem statement and its own non-goals list — write them first, in its CLAUDE.md and DECISIONS.md, before any feature work.

Why

Scope is the thing that kills small, sharp tools. Writing the non-goals down converts a thousand future "could we just…" conversations into one citation. It also makes the codebase legible: a reader can trust that the absence of timesheets is a decision, not an omission.

2. Local-first by default, server-optional behind one seam

With zero configuration the app has no backend and no login. The whole dataset is one AppData blob in localStorage (under a versioned key). It never leaves the device. This is the default a new contributor, a demo, and the test suite all run against.

An optional Node + node:sqlite REST API plugs in behind the same PersistenceAdapter interface, switched on by a single build-time env var (VITE_FLOATY_API). When it is on, the app talks to the server through that one seam and nothing else in the app changes. Server mode is last-writer-wins with no per-user isolation — a deliberate, documented posture, not an oversight.

Load-bearing rule

There is exactly one persistence seam — the PersistenceAdapter interface. Local storage and server sync are two implementations of it. No component, store action, or selector may know which one is active. Adding a third backend means adding a third implementation, never a second seam.

3. One source of truth, shared so it cannot drift

The pure domain core — types, validation, integrity rules, cascade-delete, import-remap, migration, seed — lives in a workspace package (@floaty/shared) that is imported by both the web app and the server. There is no second copy of the rules on the server. Server-side validation is literally the client's code running in Node.

Why

The classic failure mode of a client + server pair is rule drift: the client clamps hours to 24, the server forgets to, and a hand-crafted request stores garbage. By making it physically the same module, drift is not "discouraged" — it is impossible. The package has no environment dependencies (no React, no DOM, no Zustand, no localStorage) precisely so it can run in both places.

4. Forms reject; import and the server strip/repair

Every constrained field has one rule, enforced two ways depending on who is at the keyboard:

SurfaceModeBecause
A person at a formReject — show an inline error, keep the dialog openThey can fix it; teaching them is better than silently changing their input.
A machine importRepair or drop — clean text, clamp numbers, mint a fresh id, drop a record with a dangling required FKThere is no human to prompt; rejecting the whole file because one row is bad is worse than salvaging the rest.
A direct API write to the serverRepair, then reject — clean the same value-level fields (enums, colour, hours), but throw ValidationError → 400 on a dangling required FK rather than dropping the rowA single crafted write is one record, not a file of salvageable rows; a referential break should fail loudly, not be silently swallowed.
Load-bearing rule

The definition of each rule (max length, allowed characters, hours clamp) is a single shared function. The form calls it to test ("is this disallowed?"); the import path calls a sibling function to fix ("clean this"). They can never disagree because they read the same constants. See Domain core → Validation.

5. Extending an entity is drift-proofed by the compiler

Adding a field to an entity touches several parallel lists: the type, the import sanitiser's switch, the foreign-key map, the server's column spec and creation order. Forgetting one would normally be a silent data-loss bug. Here, most of those lists are exhaustiveness-checked against the types, so a missed list is a compile error. The one exception is the FK map (Record<string, ScopedEntityKey>): the compiler proves every value is a real table name, but not that every FK field is listed — that gap is caught by the fully-populated fixtures, which round-trip through the server tests so a dropped or un-remapped field fails the gate.

Do this

When you add a field, follow the path on rails: shared types → fixtures.ts (set the new field to a non-default value) → server tables.ts columns (the auto-ALTER adds the DB column) → the sanitiser. If you miss a step, the build tells you exactly where. Page 03 walks the whole path.

6. Multi-tenant by account, through one read seam

Every domain entity carries an accountId. You pick a company on load; the dataset is scoped to it. The active account id is held in memory and never persisted — it is a runtime selection, not part of the data. All scoped reads go through one hook/function so a read filter, a write guard, and a cascade can never independently define "belongs to account X".

Posture, stated honestly

In the current trust model, the account picker is convenience scoping, not a security boundary. accountId is client-asserted; the server's ownership check is defence-in-depth, not isolation. This is fine for a trusted demo and is documented as such — turning it into a real boundary is a named future stage, not a default. Never expose isolated user data on this model.

7. Device-global preferences are separate from the dataset

Theme, utilisation display toggles, allocation-bar label toggles, and sidebar collapsed-state are per-device preferences, not data. Each lives under its own localStorage key and is excluded from AppData and from export/import.

Why

If theme were in the dataset, importing a colleague's export would flip your colour scheme, and the server would sync your personal display choices to everyone. A preference about this screen belongs to the screen. The default theme is light; system follows the OS; a flash-of-wrong-theme guard runs before first paint.

8. Keep conceptually distinct signals distinct

A recurring discipline: when two things look similar but answer different questions, they get separate code paths and separate names, and merging them is explicitly forbidden. Floaty's canonical example is its two "over-capacity" signals:

  • The per-day over-marker flags any work on a zero-capacity day.
  • Utilisation % is a working-day ratio over a fixed forward 14-day window from today.
Load-bearing rule

They answer different questions ("is anything booked on a day off?" vs. "how loaded is this person this fortnight?") so they are never merged. Your sibling will have its own such pairs — find them, name them distinctly, and write the "don't merge these" note where the next person will see it. Naming is part of this: it is always "Utilisation", never "Load", everywhere in Floaty.

9. Two oracles beyond "the tests pass"

Automated tests assert behaviour, not appearance — role and DOM assertions prove that a button does the thing, not that it looks right. Two additional oracles cover what assertions can't:

  • Screenshots are the visual oracle. The E2E suite captures real renders at key states for human inspection.
  • @axe-core/playwright is the accessibility oracle, run in light mode, dark mode, and with a modal open — failing the suite on any serious/critical WCAG violation.

Accessibility is therefore a gate, not an afterthought. See Testing → The two oracles.

10. Every behavioural change ships behind a flag, default OFF

Production-hardening and new server behaviours are introduced behind environment flags whose unset state is byte-for-byte today's behaviour. The flags are FLOATY_* on the server (runtime) and VITE_FLOATY_* on the client (build-time, inlined). Flag parsing is fail-closed: a non-numeric rate limit, a typo'd auth mode, a missing secret — each means "off" or "refuse to boot", never "accidentally on".

Do this

A handful of changes are deliberately not flagged — graceful shutdown, a production safety-interlock, a Node version bump. The test is: would an "off" default preserve a defect? If yes, don't flag it; fix it. Otherwise, flag it off. Document the exceptions with their rationale, as Floaty does in its flag register.

11. Decisions are documentation, and documentation is cheap to keep

The repo treats its own decisions as a first-class artifact, split by access pattern so keeping it current is nearly free:

DECISIONS.md
A slim, present-tense digest of the calls that still constrain the code. Read whole; short by design.
docs/decisions-log.md
An append-only, dated history of every review/remediation round. Large — you grep it or tail it, never read it whole.
NEEDS-INPUT.md
Open product questions. You flag these, you don't silently resolve them.
CLAUDE.md
The short working-notes file loaded into the AI pair's context every session — the load-bearing invariants in one breath.
user-stories/
Manual test scripts kept 1:1 with the Playwright specs, with one REFERENCE file as their single source of truth.

The full mechanics — how to log a decision in one line, when to promote it to the digest — are on page 09.

12. A JS-less load is never a silent white page

A small but characteristic decision that captures the house attitude to failure. The entire app is JS-rendered, so a browser that blocks scripts would otherwise show a blank page with an empty console — undiagnosable. So index.html ships a static placeholder inside #root ("Loading… if this doesn't go away, JavaScript isn't running") that React replaces on mount, plus a <noscript> banner for the JS-disabled case.

Why this is a principle, not a detail

The instinct it encodes — a failure should diagnose itself, never present as a mysterious blank — recurs everywhere in this codebase: the dev server binds loudly rather than silently on the wrong port; the server throws at startup on schema drift rather than failing on a later write; load errors route to a specific recovery screen rather than an empty app. Carry the instinct, not just the placeholder.

13. Surface, never swallow (and know where try/catch is harmful)

The same self-diagnosing instinct, made into an error-handling rule. A catch exists only to do one of three things: re-throw with more context, route the error to a visible surface (a form field error, a toast, a typed load-error that picks a recovery screen, an HTTP status), or degrade to a documented default — and that last one only for genuinely non-load-bearing state (device preferences). A catch {} that drops the error, or one that replaces a specific, already-authored message with a generic one, is the anti-goal: in a local-first app it turns a loud, fixable failure into invisible data corruption.

Load-bearing rule

When you re-throw from a catch, attach the caught error as { cause: e } so the full chain survives, not just a re-worded message. ESLint's preserve-caught-error enforces this for native errors; it does not fire on custom error classes or a bare catch {}, so make those carry the cause too (Floaty's LoadError and ValidationError take an ErrorOptions and forward it). This convention came from a sibling repo — exactly the kind of lesson this handbook is for.

The half a naive sweep gets wrong

"More try/catch" is not the goal — surfacing is. Wrapping a pure function, a hot render path, or a deliberate integrity throw (the store's last-line guards) hides corruption rather than catching it. Put the safety as a clamp in the pure core instead, and leave a guard-comment so the next contributor doesn't re-add the harmful wrapper. The full standard — the boundaries that get guarded, the two-tier error model, the only acceptable swallows — is a short repo doc, DEFENSIVE-CODING.md.
