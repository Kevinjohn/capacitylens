State & persistence

The Zustand store is the orchestrator: it owns identity, time, and history, and delegates every rule to the shared core. Persistence sits behind one adapter, debounced and flushed safely — and in server mode, hydrated per account through one carefully-guarded orchestrator. This page is the contract between the UI and everything below it.

The store's three zones

One Zustand store, created with plain create<StoreState>()((set, get) => …) — no middleware, no immer, no devtools. Its state divides into three zones with different lifecycles:

ZoneFieldsSnapshotted for undo?Persisted?
Datadata: AppDataYesYes (via the adapter)
UIui: SchedulerUI (zoom, dates, filters, selection, draw mode)NoNo
Transienthydrated, activeAccountId, past/future, persistError, loadError, connectionError, notice, dirtyForm, accountSummaries, previousAccountId, activeRole, srAnnouncement, draggingAllocationId, theme, device prefs (sidebar, weekends, snap, fake sign-in, intro)…NoNo (theme/prefs have their own keys)
Load-bearing rule

Only data is on the undo stack and only data goes through the adapter. UI and transient state must never land in either. The mechanism that enforces this is that only data mutations call mutate(); everything else calls set() directly. Note the two error fields are distinct on purpose: connectionError (server unreachable → retry screen) vs. loadError (corrupt local bytes → recovery screen).

Draft / Patch — callers never supply identity

Two utility types stop a caller from ever passing an id, an accountId, a timestamp — or the built-in-client flag. The store stamps all of those itself.

src/store/useStore.ts
// A Draft drops the server-owned fields (id/timestamps), accountId (the store stamps the
// active account) AND builtin (the Internal-client flag is never caller-settable).
export type Draft<T extends Entity> = Omit<T, 'id' | 'accountId' | 'createdAt' | 'updatedAt' | 'builtin'>
export type Patch<T extends Entity> = Partial<Draft<T>>

addResource(draft: Draft<Resource>), updateResource(id, patch: Patch<Resource>). The compiler refuses any attempt to pass identity from a component.

id, time & the mutate helper

Identity and time are minted in two tiny closures, and every data change funnels through one helper that also drives undo:

src/store/useStore.ts (inside the create closure)
const stamp = () => { const now = new Date().toISOString(); return { createdAt: now, updatedAt: now } }
const touch = () => new Date().toISOString()

const requireAccount = (): ID => {
  const id = get().activeAccountId
  if (!id) throw new Error('No active account — cannot mutate scoped data.')
  return id
}

const updateById = <T extends Entity>(list: T[], id: ID, patch: Partial<Omit<T, keyof Entity>>): T[] =>
  list.map((x) => (x.id === id ? { ...x, ...patch, updatedAt: touch() } : x))

// EVERY forward data mutation goes through mutate(): it snapshots prior data onto the
// undo stack (capped) and clears the redo stack. (undo/redo move between the stacks
// themselves; mutate is the only PUSHER of new history.)
const mutate = (producer: (d: AppData) => AppData) =>
  set((s) => ({ data: producer(s.data), past: [...s.past, s.data].slice(-HISTORY_LIMIT), future: [] }))
Load-bearing rule

The store — not the core, not the component — owns id minting (newId()), timestamps (stamp/touch), and history (mutate). The core computes the next data; the store assigns identity and records the transition. Keep this division: it is what lets the same core run server-side where the server owns its own ids/time.

Undo / redo — two stacks, capped

History is two arrays of full AppData snapshots. Undo pops past onto future; redo does the reverse. Both capped at HISTORY_LIMIT (50).

undo: () => set((s) => {
  if (s.past.length === 0) return {}
  const previous = s.past[s.past.length - 1]
  return { data: previous, past: s.past.slice(0, -1), future: [s.data, ...s.future].slice(0, HISTORY_LIMIT) }
}),

Undo/redo is wired to ⌘Z / ⌘⇧Z globally in the shell — the keyboard handler calls useStore.getState().undo() — and to visible Undo/Redo icon buttons on the schedule toolbar (undo-button/redo-button, disabled when their stack is empty).

Account switches reset history

setActiveAccount sets past: [] and future: [] (and clears per-account UI: filters, collapsed groups, selection, focus dates). Undo must never cross an account boundary. This is the read-side of the tenancy rule from page 03.

Action patterns

Three shapes cover almost everything. The pattern for a create with full integrity — note the entity is built first, then the viewer guard, then the shared-core asserts:

addAllocation: (input) => {
  const accountId = requireAccount()
  const e: Allocation = { ...input, hoursPerDay: clampHoursPerDay(input.hoursPerDay),
    id: newId(), accountId, ...stamp() }
  if (blockedByViewer()) return e            // a Viewer's write is an inert no-op + notice, never a throw
  assertAllocation(get().data, accountId, input.resourceId, input.activityId, input.hoursPerDay)  // shared core
  assertDateRange(input.startDate, input.endDate)                                                 // shared core
  mutate((d) => ({ ...d, allocations: [...d.allocations, e] }))
  return e                                  // returned so callers can chain off the new id
},

The pattern for an update — the stale-id no-op, and every assert re-run against the effective merged row, so the store rejects exactly what the server's PUT/PATCH merge would 400 on:

updateAllocation: (id, patch) => {
  if (blockedByViewer()) return
  const existing = findOwned(get().data, 'allocations', id)
  if (!existing) return   // stale id (e.g. drag committed after an undo) → silent no-op
  // Validate the EFFECTIVE row (patch merged over existing), so a note/status-only
  // patch isn't rejected for omitting dates — and a ref change is fully re-validated.
  assertAllocation(get().data, existing.accountId,
    patch.resourceId ?? existing.resourceId, patch.activityId ?? existing.activityId,
    patch.hoursPerDay ?? existing.hoursPerDay)
  assertDateRange(patch.startDate ?? existing.startDate, patch.endDate ?? existing.endDate)
  const safe = patch.hoursPerDay !== undefined
    ? { ...patch, hoursPerDay: clampHoursPerDay(patch.hoursPerDay) } : patch
  mutate((d) => ({ ...d, allocations: updateById(d.allocations, id, safe) }))
},

And transient UI uses plain set, with a comment that says why:

// Plain set (NOT mutate): transient UI, must never land on the undo/redo stack.
setDraggingAllocation: (id) => set({ draggingAllocationId: id }),
Action conventions

Name actions verb + noun (addClient, updateResource, setZoom, goToToday) — never a handle prefix. Create actions return the new entity. Update/delete actions resolve through findOwned first (missing → no-op; cross-account → throw). All immutability is by spread — no immer.

Why the viewer guard no-ops instead of throwing

blockedByViewer() is inert unless activeRole === 'viewer' exactly — null (OFF/local/not-fetched), owner, admin and editor all permit, so the default deploy is byte-identical. When it does block, it surfaces a read-only notice and returns; it never throws, because a throw here would read as data corruption. The server's 403 is the authoritative boundary — this is UX plus defence-in-depth (page 12).

The scoping seam

For account-scoped reads, components don't touch store.data directly — they read the scoped view through one memoised hook (and normal views through its lifecycle-filtering twin):

src/store/useScopedData.ts
export function useScopedData(): AppData {
  const data = useStore((s) => s.data)
  const activeAccountId = useStore((s) => s.activeAccountId)
  return useMemo(
    () => (activeAccountId ? scopeData(data, activeAccountId) : emptyAppData()),
    [data, activeAccountId],
  )
}

// The view seam: same slice, minus archived/soft-deleted rows (shared activeOnly).
// The 15 normal view components use THIS; export/import keeps the raw seam so a
// backup never silently loses retained rows.
export function useActiveScopedData(): AppData

The exceptions are the reads that sit outside the scope, and they go on purpose to other sources: the tenant list (the AccountPicker reads accountSummaries, which in server mode comes from GET /api/accounts — in server mode data holds only the active account's slice, so data.accounts would be wrong); a per-company export that scopes by an explicit id rather than the active one (DeleteCompanyDialog calls scopeData(data, account.id)); and the scheduler's drag-time lookups (AllocationBar reading s.data.resources / schedulingModeFor, including imperative getState() reads off the render path). Everything else goes through the hooks.

Why the useMemo is mandatory, not optional

Zustand returns a fresh scoped object each store update. Without memoising on [data, activeAccountId], every component subscribed to scoped data would re-render on every unrelated mutation. scopeData itself loops SCOPED_KEYS applying byAccount(accountId) — using the same tenancy predicate as the write guard and the cascade.

Selectors — pure functions, memoised at the call site

Selectors are plain exported functions (not hooks), taking AppData plus params and returning derived values. They are not self-memoising — the call site wraps them in useMemo when needed. Naming is {entity}For{Parent} / {entity}ById, never a get prefix.

export const allocationsForResource = (data: AppData, resourceId: ID) =>
  data.allocations.filter((a) => a.resourceId === resourceId)
export const activityById = (data: AppData, id: ID) => data.activities.find((t) => t.id === id)

// Canonical ordering gets its OWN comparator so the scheduler grouping and the
// Disciplines list always agree (sortOrder, then name as a stable tiebreak).
export const byDisciplineOrder = (a: Discipline, b: Discipline): number =>
  a.sortOrder - b.sortOrder || (a.name < b.name ? -1 : a.name > b.name ? 1 : 0)

Notices: one store field, a Sonner bridge, three tones

Transient messages go through one store action — setNotice(message, tone?) with tones 'info' | 'warning' | 'error' — and the shell bridges the notice field to a Sonner toast (the hand-rolled Toast component is retired). Tone drives duration, which is an accessibility decision, not a style:

infoneutral4 s auto-dismissConfirmations ("Imported 42 records").

Sonner's richColors is deliberately OFF (its light error palette fails AA); the danger accent is a local .toast-error style on the house tokens.

The adapters

The seam is the three-method interface from page 03. Two implementations; one ternary selects between them at module load:

src/data/storageAdapter.ts
export const STORAGE_KEY = `${STORAGE_KEY_PREFIX}v3`   // 'capacitylens/v3' — the prefix comes from shared/brand.ts
export const storageAdapter = new LocalStorageAdapter(STORAGE_KEY)

export const persistenceAdapter: PersistenceAdapter = isServerConfigured()
  ? new ServerSyncAdapter(API_BASE)
  : storageAdapter

LocalStorageAdapter stores { schemaVersion, data } as one JSON blob under the versioned key, runs migrate() on load, and raises a typed LoadError('corrupt' | 'unavailable', …, { cause }) that routes to two different recovery screens. storageAdapter is exported separately because the recovery UI needs its extra readRaw()/clear() methods, which are deliberately not on the shared interface.

API_BASE and the mode predicates read the build-time env once — and this is the load-bearing bit that inverted when server-backed became the default:

src/data/apiConfig.ts
export const API_BASE = (import.meta.env.VITE_CAPACITYLENS_API ?? '').replace(/\/+$/, '')

// The ONLY route to the in-browser localStorage build. (The persistence demo — distinct
// from the cosmetic fake-sign-in "auth demo".)
export function isDemoMode(): boolean {
  return import.meta.env.VITE_CAPACITYLENS_DEMO === '1'
}

// Server mode is the DEFAULT — true unless the demo flag is set. An empty API_BASE
// means SAME-ORIGIN (/api), not "no server".
export function isServerConfigured(): boolean {
  return !isDemoMode()
}

Server sync by diffing snapshots

The clever part of the server adapter: the store already produces a fresh AppData on every action (including undo/redo and import). So the adapter never needs a command log — it diffs the previous synced snapshot against the new one and turns the difference into ordered ops.

src/data/syncOps.ts
const UPSERT_ORDER = ['accounts','clients','disciplines','projects','phases',
  'resources','activities','allocations','timeOff'] as const

// Exhaustiveness: a new AppData key not added here won't compile.
type _MissingFromOrder = Exclude<keyof AppData, TableKey>
const _orderComplete: _MissingFromOrder extends never ? true : never = true

export function diffOps(prev: AppData, next: AppData): Op[] {
  const upserts: Op[] = [], deletes: Op[] = []
  for (const table of UPSERT_ORDER) {
    const prevById = new Map(prev[table].map((e) => [e.id, e]))
    const nextById = new Map(next[table].map((e) => [e.id, e]))
    for (const row of next[table]) {
      const before = prevById.get(row.id)
      if (!before || before.updatedAt !== row.updatedAt) upserts.push({ method: 'PUT', table, id: row.id, row })
    }
    for (const row of prev[table]) if (!nextById.has(row.id)) deletes.push({ method: 'DELETE', table, id: row.id, accountId: row.accountId })
  }
  deletes.reverse()                 // upserts parent-first, deletes child-first
  return [...upserts, ...deletes]
}
Why all-upserts-then-all-deletes, in one transaction

A reparent coalesced with the old parent's delete must land the re-binding before the cascade fires. Diffing detects changed rows by updatedAt (last-writer-wins, since ids/timestamps are client-generated). The ops are sent to POST /api/batch as one transaction; lastSynced only advances on success, so a failure replays the full delta next time. Writes coalesce to the latest snapshot while one is in flight, and the drain loop deliberately has no try/catch — a throw must propagate so lastSynced never advances past unlanded writes.

Every request carries credentials: 'include' so an auth-enabled server sees the session cookie. A 401 on /api/batch surfaces as a normal write failure → persistError → the AuthProvider re-checks the session → the login wall. An expired session therefore walls the app off loudly instead of failing writes silently.

Per-account hydration & the switch orchestrator (server mode)

In server mode the store holds only the active account's slice, hydrated via GET /api/state?accountId=…. That makes account switching a persistence operation with one non-negotiable invariant:

The #1 invariant of the sync layer

The diff snapshot (lastSynced) and data must always be the same account before any save diffs. A desync (snapshot = account A, data = account B) would emit DELETEs for all of A plus PUTs for all of B — cross-account data loss. This is guarded by a dedicated regression test, and it is why there is exactly one code path allowed to re-hydrate.

That path is the switch orchestrator in persist.ts. Picking a tenant (or returning to the tab — see below) runs: await any in-flight save → flush pending edits → adapter.loadAll(accountId), which atomically re-seeds lastSynced to exactly the returned slice (so the next diff is self-vs-self, zero ops) → a guarded replaceAll into the store (guarded so the load isn't pushed back out as a save). A switchToken discards a stale load that finishes after the user has already switched again.

  • Refresh-on-focus: returning to the tab/window (focus + visibilitychange→visible) re-runs the same orchestrator body (refreshActive), throttled to 30 s, so cross-tab and cross-device edits appear. Unsaved local edits flush first — last-writer-wins, the user wins. Never build a parallel re-hydrate path (a React hook can't re-seed the private snapshot).
  • The picker's list comes from GET /api/accounts into accountSummaries (with the caller's role per account); in auth-on mode the no-arg whole-dataset read is closed, and the adapter treats its 400 as "hydrate empty, show the picker".
  • The demo build is inert here: all accounts live in the one local blob, so switching is just a state change and the orchestrator never runs (serverMode: false).

Bootstrap & persistence wiring

main.tsx fires bootstrap before the first render; the shell gates content on hydrated, so there's no flash of empty data.

src/main.tsx (order)
import './data/runStorageMigration'      // 0. side-effect FIRST: legacy 'floaty/' keys → 'capacitylens/'
applyThemeToDom(useStore.getState().theme)        // 1. re-affirm theme from the store
watchSystemTheme(() => useStore.getState().theme)  // 2. follow OS changes when theme==='system'
void bootstrap(useStore, persistenceAdapter, {
  seedIfEmpty: isDemoMode() ? seed() : undefined,  // 3. auto-seed is DEMO-BUILD-ONLY —
  serverMode: isServerConfigured(),                //    a real server deploy first-runs EMPTY
  onError, onSuccess,
})
createRoot(rootEl).render(                         // 4. render starts alongside bootstrap,
  <StrictMode><ErrorBoundary>                      //    wrapped in the auth boundary
    <AuthProvider><RouterProvider router={router} /></AuthProvider>
  </ErrorBoundary></StrictMode>,
)

bootstrap loads via the adapter, decides whether to seed, marks hydrated, and attaches the persistence subscription. Three behaviours are load-bearing:

  • Seed-once gated on a marker, not emptiness. It uses hasExisting() to tell a genuine first run from a deliberate clear. Clearing all data does not re-seed.
  • Load failure renders empty and attaches NO persistence — so corrupt/unreachable data is never overwritten. A LoadError kind routes to the corrupt-reset screen or the connection-retry screen.
  • A failing seed write surfaces the banner but still attaches persistence — otherwise the session would silently never save and never warn.

Writes are debounced (300 ms) and flushed on teardown so a tab close inside the debounce window can't drop the last edit:

src/data/persist.ts (flush wiring)
// pagehide is the reliable close/navigate signal (fires for bfcache where beforeunload
// doesn't); visibilitychange→hidden covers tab switches and mobile. Becoming visible
// again (or window focus) also triggers the throttled refresh-on-focus re-hydration.
window.addEventListener('pagehide', onPageHide)
document.addEventListener('visibilitychange', onVisibility)   // hidden → flush; visible → retry + maybeRefreshOnFocus
window.addEventListener('focus', onFocus)                     // refresh-on-focus (server mode, 30s throttle)
window.addEventListener('online', onOnline)                   // retry a stranded write

The flush passes { unload: true } so an async adapter dispatches its writes up-front (a sequential await-loop would only get the first request out before the event loop dies; the server adapter uses a keepalive fetch). Failed writes retry with exponential backoff (cap 5 attempts, max 30 s); a fresh user edit or an online event resets the budget.

The void-promise discipline

Because ESLint enforces no-floating-promises (page 02), every intentional fire-and-forget is marked with an explicit void, and the two-argument .then(onOk, onErr) form is used over .then().catch() so a throw inside the success handler isn't misreported as a save error:

void adapter.saveAll(data, { unload: true }).catch(() => {})
void adapter.saveAll(data).then(onSuccess, onError)
Reactive vs. imperative store access

useStore(selector) in render (subscribes, re-renders on change); useStore.getState() inside an event handler (one-shot read, no subscription). Mixing them up — subscribing inside a callback — causes needless re-subscription churn. The convention is explicit and consistent across the codebase.
